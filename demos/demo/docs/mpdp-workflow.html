MPDP Workflow

1. 文档定位

当前 demo 中，MPDP 是一个新的可比较 search strategy， 命令行别名是 mpdp，并且在 --compare 输出中紧跟 DPSub。它的目标是尽量贴近论文中的 block-local join-pair 枚举， 同时把并行化边界写清楚：并行发生在同一个 DP level 内，不跨 level。

算法来源

MPDP 来自 Mancini et al., SIGMOD 2022 中面向大规模 join optimization 的思路。

Demo 定位

默认 mpdp 走同一套代码的单线程路径；mpdp-parallel 用多个 CPU worker 处理同一 level 的不同 connected set。

对比对象

DPSub 是全局 subset 枚举 baseline；二者共享基础 cost helper，但 memo 维度和 property 传播不同。

不要把本文读成性能承诺： SIGMOD 论文讨论的是 massively parallel join optimization。这个 demo 只实现了同一思想的候选枚举形状， 运行时仍是普通 C++ CPU 教学代码。这里讨论的是线程模型和依赖边界，不声称复现 GPU runtime 的吞吐。

2. 原作者实现里的并行骨架

参考实现来自 Manciukic/postgres-gpuqo。 这里最值得读的不是 PostgreSQL 集成层，而是 src/backend/optimizer/gpuqo 下几组文件：

文件本文关注点
gpuqo_cpu_dpsub_parallel.cu CPU 并行 DPSub：长生命周期 pthread worker、按 level 广播开始、用共享 next_sid 分 chunk、层末 barrier。
gpuqo_cpu_dpsub_bicc.cu CPU BiCC 枚举：对单个 join set 找 biconnected components，在 block 内枚举局部分割，再 two-sided grow。
gpuqo_cpu_level_hashtable.cuh memo 按 relation-set size 分 bucket，降低不同 DP level 之间的依赖耦合。
gpuqo_dpsub_bicc.cuh GPU BiCC 版本：把一个 join set 内的 block-subset 候选继续拆给 warp lanes，并在 shared memory 中做局部最优。

抽象出来的 CPU 并行模式是：

main thread: initialize memo with level-1 base relations for level = 2..N: publish current level reset shared work cursor wake workers workers: wait until level advances while work cursor still has join sets: claim a chunk of set ids enumerate/cost each set using only previous memo levels keep results in local output merge local output into memo under synchronization signal "this worker finished the level" main thread: wait until every worker finished this level move to next level

这个模型的关键不是“用了线程”本身，而是 level barrier： 大小为 k 的 join set 只依赖更小的 set，所以同一层里的多个 S 可以并行；但 k + 1 层必须等 k 层全部写入 memo 后才能开始。 原作者 CPU parallel 版本用 pthread condition variable 做这个 barrier； 当前 demo 用更直接的方式：每个 level 创建一组 std::thread，join 全部 worker 后再进入下一层。

和原作者实现的差异： postgres-gpuqo 的 CPU parallel 版本按 binomial rank 枚举所有大小为 level 的 set，并用 next_sid.fetch_add(skip) 分 chunk。 当前 demo 先调用 CollectConnectedSubsets 得到每层 connected sets， 再用共享 next_index 分发 Si[index]。这少了 unrank/rank 逻辑， 但保留了同一个并行边界：同层任务并行，层间严格同步。

3. 数据模型

多线程版本要先把共享状态和线程私有状态分清楚。当前 demo 的 MPDP 复用基础结构， 但 memo contract 与 DPSub 不同：

结构共享/私有MPDP 中的角色
RelSet 值类型 表示 DP 状态 S、block B、局部 pair lb/rb 和全局 left/right。
JoinGraph 只读共享 提供 IsConnected、HasPredicateAcross、CrossingPredicates 等图查询；worker 可并发读。
StatsCatalog 只读共享 只在 MakeScan / MakeJoin 中参与 cost 计算，不决定候选枚举；worker 可并发读。
MemoTable level 内只读，level 末写入 RelSet → PlanPtr。worker 枚举当前 level 时只读取已完成的更小 level；本层结果等 worker 完成后统一合并。
StateCell 每个 S 独占 保存单个 DP 状态的 best 和 trace。这就是避免多个线程同时 TryUpdate 同一个 memo entry 的核心。
cells 按 index 分片 std::vector<StateCell>，每个 worker 只写自己 claim 到的 cells[index]。
next_index + mutex 共享调度状态 当前 demo 的 work cursor。它对应原作者 CPU parallel 里的 next_sid.fetch_add(skip)，只是粒度从 set-id chunk 变成 connected-set index。

因此并行安全性来自三个约束： memo 在当前 level 计算期间只读， 每个 StateCell 只由一个 worker 写， memo 写入和全局 trace 汇总只发生在 level barrier 后。 这比给 unordered_map 的每次读写都加锁更简单，也更接近 DP 的依赖结构。

4. 当前 demo 的多线程模型

4.1 串行路径和并行路径共享同一套 DP

MPDP::Search(graph, stats, required_property):
  memo = {}

  if graph is empty:
    return no-plan

  for each relation r:
    memo[{r}] = MakeScan(r)

  connected_by_size = CollectConnectedSubsets(graph)

  for size = 2 to n:
    Si = connected_by_size[size]
    cells = vector<StateCell>(Si.size())

    fill_cell(index):
      S = Si[index]
      for each (left, right) in EnumerateJoinPairs(graph, S):
        cells[index].trace.partitions_explored++
        BuildPlansForPairIntoCell(memo, cells[index], left, right)

    if thread_count == 1:
      for index in [0, Si.size()):
        fill_cell(index)
    else:
      run worker threads over indexes in [0, Si.size())

    // level barrier: all workers are done here
    for index in [0, Si.size()):
      MergeCell(memo, Si[index], cells[index])
      MergeTrace(global_trace, cells[index].trace)

  best = memo[full_set]
  if required_property is Sorted and best exists:
    best = SortEnforcer(best, required_property.sort_key)

  return best

串行和并行路径只差在 fill_cell(index) 的调用方式。 这点很重要：并行版不是另一套算法，只是把同一 level 内互不依赖的 cell 填充拆给多个 worker。

4.2 Worker 调度

worker():
  loop:
    lock(mutex)
    if next_index == Si.size():
      unlock(mutex)
      return
    index = next_index
    next_index++
    unlock(mutex)

    fill_cell(index)

当前 demo 用 std::mutex 保护 next_index，实现最直接的动态调度。 原作者 CPU parallel 版本使用 atomic<uint_t> next_sid 和 skip 一次 claim 一段 set ids，减少调度开销。两者是同一个设计：worker 不预先固定拿哪一段工作， 而是干完一块再去共享 cursor 领下一块，避免不同 connected set 枚举代价差异导致负载不均。

为什么不用并发写 memo

一个 S 的候选计划可能很多，如果每个候选都直接写 memo[S]，就需要保护 unordered_map 和同一个 best entry。 当前 demo 先写线程私有的 cells[index]，层末再合并，锁粒度变成调度 cursor。

为什么不跨 level 并行

|S| = k + 1 的计划依赖多个更小 subset 的 best plan。 只要 k 层还没完全写入 memo，下一层就可能读到缺失结果。 所以 level barrier 是正确性边界，不只是实现方便。

4.3 Join-Pair 枚举

EnumerateJoinPairs(graph, S):
  result = {}
  emitted = {}

  for each block B in FindBlocks(G[S]):
    for each non-empty proper subset lb of B:
      rb = B - lb

      if lb or rb is disconnected:
        continue
      if no predicate crosses lb/rb:
        continue

      left = Grow(lb, allowed = S - rb)
      right = Grow(rb, allowed = S - lb)

      if left/right is disjoint, exactly covers S, connected,
         has a crossing predicate, and ordered pair is not emitted:
        emit (left, right)

这个流程有三个层次：

  1. Block-local：lb 和 rb 都来自同一个 block。
  2. Global growth：lb 和 rb 都不是直接作为最终两侧，而是分别在 S \ rb 与 S \ lb 中向外生长。
  3. Cover check：只有当 grow 后的 left 与 right 不相交、且 left | right == S 时，这个局部分割才覆盖了当前 DP 状态。
  4. Dedup：不同 block 或不同局部 pair 可能映射到同一个 global join-pair，需要用 ordered pair (left, right) 去重；不能用 left | right，因为合法候选的并集总是 S。
源码边界： 这里对齐的是 block 枚举 + two-sided grow 的工作流。当前 demo 为了让教学 trace 更直接， 把局部连通性和 crossing-predicate 检查显式前置；GPU-QO 的 BICC 实现会更宽地枚举 block subset， 再依赖 two-sided grow、覆盖检查和 join/memo 检查淘汰候选。因此不要把这段伪代码读成源码逐行移植。

5. 单个任务内部：Blocks 与 Grow

5.1 Block 是什么

本文中的 block 指 biconnected component：一个在删除任意单个顶点后仍保持连通的最大子图。 Bridge edge 也会形成一个二点 block。Block-cut tree 把这些 block 和 articulation point 组织成一棵树， 是 MPDP 能够从局部分割恢复全局 join-pair 的结构基础。 在并行模型里，找 blocks 和 grow 都发生在单个 fill_cell(index) 内部； 它们不需要和其他 worker 通信。

Figure 5-style shape: r1 -- r2 -- r3 -- r4 | | +-----------------+ \ r5 -- r9 / \ r8 -- r6 \ / r7 Block-cut tree intuition: cyclic block -- bridge block -- bridge block -- cyclic block

这个形状很适合解释论文里的链式 block-cut tree：左边一个 cycle block， 中间两个 bridge block，右边一个 cycle block。当前 demo 的 mpdp_fig5 就是这种 Figure 5-style case。

5.2 Block-cut tree 不一定是链

只看 Figure 5-style case 容易形成误解，以为 MPDP 只适用于链状 block-cut tree。 因此 demo 还加入了 mpdp_branching_blocks： 一个中心 cut vertex c 被三个 cycle block 共享，block-cut tree 是分叉形状。

Branching block-cut tree shape: a1 / \ c ---- a2 / \ / \ b1 d1 | | \ / | | b2 d2 --+ Conceptually: cycle A | cut vertex c / \ cycle B cycle D

这个用例的目的不是引入新的 cost model，而是防止把某一种图形当成算法前提。 MPDP 的枚举逻辑要能处理 block-cut tree 的分叉，而不只是线性桥接。

5.3 Grow 的作用

Grow(graph, seeds, allowed) 是一个 BFS。 MPDP 会对局部 pair 的两侧各做一次 grow： 从 lb 出发，只在 allowed = S \ rb 内沿 join graph 扩展得到 left； 从 rb 出发，只在 allowed = S \ lb 内扩展得到 right。 

Grow(seeds, allowed):
  visited = seeds
  queue = seeds

  while queue not empty:
    current = pop(queue)
    for each predicate edge current -- next:
      if next is in allowed and next not visited:
        visited += next
        push(next)

  return visited

最终不是简单令 right = S \ left，而是检查 grow 出来的 left 与 right 是否不相交、并集是否等于 S。 如果二者不能覆盖 S，或任一侧不连通，或二者之间没有 predicate， 这个候选会被丢弃。由于丢弃发生在发出结果前，trace 中不会把它记为 rejected partition。

和 GPU 版本的关系： 原作者的 gpuqo_dpsub_bicc.cuh 会把一个 relid 内的 block-subset 候选继续拆给 warp lanes，并用 shared memory 存 blocks 和局部候选。 当前 demo 不模拟 warp 内并行；它只把更粗的 S 级任务分给 CPU worker。 这更适合作为普通 C++ 多线程实现的阅读起点。

6. 物理计划与 DP 表

一旦 MPDP 发出 (left, right)，当前实现只在 relation-set memo 中 cost unordered join plan：

步骤行为Trace 影响
HashJoin 读取 memo[left] 和 memo[right]，尝试更新本地 cell.best。 plans_costed++
NestedLoopJoin 同样读取 Any 子计划，按 nested-loop cost 公式生成候选。 plans_costed++
Root SortEnforcer 只有调用方要求根输出 Sorted(...) 时，在最终 best plan 外层追加一次排序。 plans_costed++
关键对照： MPDP 不在内部生成 Sorted interesting orders，也不 cost MergeJoin。 因此它的 best cost 可以高于 DPSub；这不是 join-pair 枚举错误。 正确性测试关注是否找到合法计划、是否不低于 DPSub 的更大物理计划空间、以及串行/并行结果是否一致。

7. 两个 MPDP 用例

7.1 mpdp_fig5

mpdp_fig5 有 9 个关系、10 条 predicate。 图形是两个 cyclic blocks，中间通过 bridge blocks 连接。这对应论文 Figure 5 风格的教学形状： MPDP 可以从 block-local CCP pair 直接生成全局 join-pair，避免 DPSub 的大量无效 subset。

7.2 mpdp_branching_blocks

mpdp_branching_blocks 有 7 个关系、9 条 predicate。 一个中心关系 c 同时属于三个 cycle block。这个 case 强调： block-cut tree 可以分叉，MPDP 不应只针对链形 bridge 结构工作。

Case图形目的重点观察
mpdp_fig5 两个 cycle blocks 通过 bridge blocks 相连 展示 Figure 5-style 的链式 block-cut tree。
mpdp_branching_blocks 一个 cut vertex 被三个 cycles 共享 展示 branching block-cut tree，避免把链形结构误当成算法限制。

8. Trace 计数器

MPDP 使用和其他策略相同的 SearchTrace，但并行版不能让多个 worker 直接递增同一个普通整数。因此当前实现让每个 StateCell 带自己的 trace， 层末由主线程调用 MergeTrace 汇总。最值得关注的字段如下：

字段MPDP 中的含义
partitions_explored 每个 cell 内最终发出的 global join-pair 数；层末汇总。它通常少于 DPSub 的 explored partitions。
partitions_rejected 当前 MPDP 发出前就过滤候选，因此专门用例中为 0。
plans_costed 每个 cell 内的物理计划代价计算次数。MPDP 每个 join-pair cost HashJoin 和 NestedLoopJoin；根 Sorted 请求会额外 cost 一次 SortEnforcer。
cache_hits bottom-up DP 不按需递归查 cache，保持为 0。
dp_cells_filled 层末 merge 后的 memo 条目数，不含 property 维度。
best_cost MPDP 自身物理计划空间内的最优 cost。它可能高于 property-aware 策略。

9. 运行命令

从 demos/demo 目录运行：

$ ./build/volcano_join_demo --strategy mpdp --test mpdp_fig5
$ ./build/volcano_join_demo --strategy mpdp-parallel --threads 4 --test mpdp_fig5
$ ./build/volcano_join_demo --strategy mpdp-parallel --threads 8 --test mpdp_branching_blocks
$ ./build/volcano_join_demo --compare --test mpdp_fig5
$ ./build/volcano_join_demo --compare --test mpdp_branching_blocks

--compare 表默认展示 canonical 五种策略，不包含 MPDP(Parallel)。 想看并行路径，需要单独使用 --strategy mpdp-parallel --threads N。 正确性上，串行 mpdp 和并行 mpdp-parallel 应该有相同 BestCost、Partitions 和 PlansCosted； 线程数只影响调度方式，不应改变搜索结果。

下面的表只保留 Partitions、Rejected 和 BestCost， 方便聚焦 MPDP 的候选枚举差异。

9.1 Figure 5-style case

$ ./build/volcano_join_demo --compare --test mpdp_fig5

Strategy            Partitions    Rejected      BestCost
DPSub               5166          4422          1641477.39
MPDP                744           0             75903325.79
Transformational    0             0             1641477.39
TopDown(Naive)      7089          5933          1641477.39
TopDown(Mincut)     1156          0             1641477.39

MPDP explored partitions 从 5166 降到 744，rejected partitions 从 4422 降到 0。 这里 best cost 高于 DPSub，是因为 MPDP 内部不传播 Sorted property，也不选择 MergeJoin。

9.2 Branching block-cut tree case

$ ./build/volcano_join_demo --compare --test mpdp_branching_blocks

Strategy            Partitions    Rejected      BestCost
DPSub               1336          850           11811.99
MPDP                486           0             11811.99
TopDown(Mincut)     822           0             11811.99

这个 case 说明 MPDP 的 block-local 枚举同样适用于分叉 block-cut tree。 MPDP 比 DPSub 少枚举 850 个会被拒绝的候选；在这个数据分布下，即便没有内部 Sorted/MergeJoin， best cost 仍与 DPSub 相同。

10. 与其他策略对比

维度DPSubMPDPTopDown(Mincut)
搜索方向 Bottom-up Bottom-up Top-down
候选生成 全局 subset 枚举 + 过滤 Block-local CCP pair + Grow 递归调用中的 Mincut partitioner
无效候选 会计入 partitions_rejected 发出前过滤，专门用例中为 0 直接生成 CP-free partitions，rejected 为 0
物理计划生成 自底向上填 DP 表 relation-set memo；HashJoin / NestedLoopJoin；根 Sorted 适配 递归求解并 memoize
教学重点 完整、直接、容易理解的 baseline MPDP 论文中的 block-local join-pair 枚举 自顶向下划分和按需 memoization

一句话总结：MPDP 是独立的 block-local join-pair 枚举实现。 串行和 CPU 并行入口共享同一套 relation-set memo、block-local join-pair 枚举与 cell 合并逻辑； 读 trace 时，最应该对比的是 Partitions、Rejected、 以及因物理计划空间不同而变化的 PlansCosted / Cached。
