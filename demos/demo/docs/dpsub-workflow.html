1. 文档定位

当前项目里有五种 join order search strategy： DPSub（命令行别名 dpsub）、 MPDP、 Transformational、 TopDown(Naive)、 TopDown(Mincut)。 本文只聚焦 DPSub。

算法来源

实现自 Mancini et al., Efficient Massively Parallel Join Optimization for Large Queries, SIGMOD 2022 中的 Algorithm 1 ("Generic DPSUB")。

实现策略

Bottom-up DP：从单表开始，按 relation 数量递增枚举所有连通子集，对每个子集遍历其所有合法 CCP partition，构建物理计划并保留最优。

核心特点

无对称性 shortcut（与 DPccp 的 csgCmpPairs 不同），内层循环天然可并行化。

因此，本文会把当前实现称为 DPSub（与 Mancini 论文中的 "Generic DPSUB" 对应）， 把 Moerkotte & Neumann VLDB 2006 中的经典算法称为 DPccp。

2. 数据模型与问题定义

2.1 Join Order 问题的图表示

与项目其他策略一样，DPSub 接收 JoinGraph 作为输入。 JoinGraph 将每个 relation 视为图的顶点，每个 join predicate 视为图的边。 优化目标是找到一张 bushy join tree，使得其不包含 cross product 且总 cost 最小。

2.2 RelSet — bitmask 子集编码

搜索空间中的 relation 集合用 RelSet（uint64_t bitmask）表示。 对 chain_3（三表链式 join）：

Relationidbitmask
a01 << 01
b11 << 12
c21 << 24
{a,b,c} (full)-1 | 2 | 47

2.3 DP 表结构

DPSub 使用 CacheTable = unordered_map<string, PlanPtr> 作为 DP 表。 Key 的格式是 "RelSet|Property"，例如： 

"3|Any"            → {a,b} 的最优 Any 计划
"3|Sorted(b.y)"    → {a,b} 的最优 Sorted(b.y) 计划
关于 DP 表与 property 维度： 当前实现中 DP 表的 key 是 "RelSet|Property"，同一个 RelSet 可能对应多个不同 RequiredProperty 的最优计划（Any、Sorted(a.x)、Sorted(b.y) 等）。 这是项目 SearchStrategy 接口统一要求的 interesting order 支持， 并非 DPSub 对 DPccp 的增强——经典 DPccp 同样可以加入这一层。 实际上 DPccp（Moerkotte & Neumann, VLDB 2006）才是更高效的算法：它用 csgCmpPairs 枚举避免了全子集遍历和 (L,R)/(R,L) 对称重复， 而 DPSub（Mancini 的 "Generic DPSUB"）刻意保留了暴力全子集枚举， 牺牲单线程效率来换取内层循环的并行友好性（见 §11）。

2.4 JoinGraph 设计：顶点与边的分离

一个重要的设计选择：Relation 结构体不包含任何边/邻接信息—— 它只有 id、alias、table_name、rows、scan_cost。 所有的图结构信息集中在 JoinGraph 类内部： 

class JoinGraph {
private:
  std::vector<Relation> relations_;                       // 顶点：纯数据描述符
  std::vector<JoinPredicate> predicates_;                 // 边：ColumnRef 对 + 选择性
  std::unordered_map<std::string, std::size_t> alias_to_index_;  // alias → 顶点索引
};

边（JoinPredicate）存在图级别而非顶点级别，这是一个刻意为之的设计：

原因说明
边属于查询，不属于 relation 同一个 Relation("b", rows=100) 在 chain_3 中连接 a 和 c，在 star query 中可能连接五个维表。边结构随查询变化，把边挂在 relation 上会让 Relation 失去复用性。
bitmask 枚举不需要邻接表 DPSub 的图查询全部是 set-level 的——IsConnected(RelSet)、HasPredicateAcross(RelSet, RelSet)。 这些判断只需要知道 "给定位掩码集合内部/之间有没有边"，用扁平的 predicates_ 向量做 bitmask 交集匹配即可，无需从某个顶点出发遍历邻接表。
ColumnRef 存 alias 而非指针 JoinPredicate 中用 ColumnRef{alias, column}（字符串）引用 relation，而非 Relation* 或 size_t id。 这让 predicate 定义独立于 relation 的创建顺序——可以先建顶点再建边，JoinGraph 内部用 alias_to_index_ 做名字解析。

以 star query a.x=b.x, a.y=c.y, a.z=d.z 为例，JoinGraph 的内部形态：

vertices (relations_): [0] {id=0, alias="a", rows=..., scan_cost=...} [1] {id=1, alias="b", rows=..., scan_cost=...} [2] {id=2, alias="c", rows=..., scan_cost=...} [3] {id=3, alias="d", rows=..., scan_cost=...} edges (predicates_): [0] {left=("a","x"), right=("b","x"), selectivity=...} [1] {left=("a","y"), right=("c","y"), selectivity=...} [2] {left=("a","z"), right=("d","z"), selectivity=...} bitmask encoding: a → 0001, b → 0010, c → 0100, d → 1000 FullSet() = 1111 = 15 Topology (a is the center): b (0010) / a.x=b.x / a (0001) | \ a.y=c.y a.z=d.z | \ c (0100) d (1000) Key: a knows nothing about its 3 edges — all graph queries go through JoinGraph: graph.HasPredicateAcross(0b0001, 0b1110) // true: 3 edges cross graph.IsConnected(0b1110) // false: b,c,d with no edges among themselves graph.CrossingPredicates(0b0001, 0b0110) // [{a-b}, {a-c}], excludes a-d (d not in 0b0110)
对比邻接表模型： 如果 Relation 存邻接表（vector<pair<size_t, JoinPredicate>> neighbors）， 则 bitmask 判断需要从任意顶点做 BFS 遍历邻接表，同一条边在两个顶点中重复存储， star query 的中心表会有大量邻接条目。当前设计为 bitmask-based set enumeration 优化—— 顶点轻、边集中、查询靠 mask 运算。

3. 算法总览

DPSub::Search 的顶层控制流： 

DPSub::Search(graph, stats, required_property):
  trace = {}
  cache.clear()

  full = graph.FullSet()
  if full is empty:
    return empty result

  // Step 1: 插入所有单表计划（size-1 base case）
  for relation in graph.Relations():
    FillRelation(cache, graph, singleton, relation, trace)

  // Step 2: 预计算按大小分桶的连通子集
  connected_by_size = CollectConnectedSubsets(graph)
  // connected_by_size[i] = { 所有大小为 i 的连通 RelSet }

  // Step 3: 按大小递增做 DP
  for i = 2 .. n:
    for S in connected_by_size[i]:
      // 枚举 S 的所有非空真子集 S_left
      for S_left ⊂ S, S_left ≠ ∅:
        S_right = S \ S_left

        // CCP Block 检查
        if not IsConnected(S_left):   continue
        if not IsConnected(S_right):  continue
        if not HasPredicateAcross(S_left, S_right): continue

        // 创建物理计划
        BuildPhysicalPlans(cache, S, S_left, S_right, trace)

      // Sort enforcer：传播 interesting order
      PropagateSortEnforcers(cache, S, graph, trace)

      trace.dp_cells_filled = cache.size()

  // 返回 full set 在 required property 下的最优计划
  return cache[MakeKey(full, required_property)]

4. Step 1: 单表种子 (FillRelation)

FillRelation 为每个单表 relation 创建基础计划： 

FillRelation(cache, graph, singleton, relation, trace):
  scan = MakeScan(relation)             // SeqScan
  trace.plans_costed++
  cache["{r}|Any"] = scan

  // 为该表出现在 predicate 中的列生成 Sorted 计划
  for pred in graph.Predicates():
    if pred.left.alias == relation.alias:
      sorted = MakeSort(scan, pred.left)
      trace.plans_costed++
      cache["{r}|Sorted(pred.left)"] = min(cache[...], sorted)

    if pred.right.alias == relation.alias:
      sorted = MakeSort(scan, pred.right)
      trace.plans_costed++
      cache["{r}|Sorted(pred.right)"] = min(cache[...], sorted)

  trace.dp_cells_filled = cache.size()
为什么单表就要生成 Sorted 计划？ 这是 interesting order 的 "base case"。 当上层 merge join 要求子输入有序时，如果子输入是单表，唯一的选择就是 SeqScan + SortEnforcer。 提前在 DP 表中准备好这些 Sorted 计划，可以避免上层反复计算。

5. Step 2: 收集连通子集

CollectConnectedSubsets 做一次全枚举，按 size 分桶： 

CollectConnectedSubsets(graph):
  n = graph.RelationCount()
  connected_by_size = vector<vector<RelSet>>(n + 1)

  for s = 1 .. FullSet:
    if not graph.IsConnected(s):
      continue
    size = PopCount(s)
    connected_by_size[size].push_back(s)

  return connected_by_size

这一步其实就是 Mancini Algorithm 1 Line 5 中的 S_i：

Si = { S | S ⊆ R and |S| = i and S is connected }

预计算的好处是 DP 主循环可以直接按 size 遍历，不需要在循环体内再检查连通性。 对于 n ≤ 10 的 demo 规模，2n 枚举完全可行。

6. Step 3: 按大小递增 DP

6.1 子集枚举：Gosper's hack 的变体

DP 主循环对每个大小为 i 的连通集合 S，枚举它所有的非空真子集 S_left： 

for S_left = (S - 1) & S; S_left != 0; S_left = (S_left - 1) & S:

这是经典的 "iterate over all subsets" 技巧（与 Gosper's hack 同源）。 对于大小为 k 的集合 S，它产生 2k - 1 个非空子集 （不含空集，不含 S 自身，因为 S_left = S 不在循环范围内）。

没有对称性 shortcut！ DPccp 使用 csgCmpPairs 枚举来避免对称重复（(A,B) 和 (B,A) 只评估一次）， 但 DPSub 会评估 所有 proper subset。 这意味着对于每个 (S_left, S_right)，DPSub 在枚举 S_left 和枚举 S_right 时各评估一次。 这符合 Mancini Algorithm 1 的精确描述，代价是 partitions_explored 数量约为 DPccp 的两倍。

6.2 DP 填充循环的完整结构

for i = 2 .. n:
  Si = connected_by_size[i]

  for S in Si:
    // 内层：枚举所有 proper subset
    for S_left ⊂ S:
      S_right = S \ S_left
      trace.partitions_explored++

      // CCP Block 检查（详见 §7）
      if not IsConnected(S_left):  trace.partitions_rejected++; continue
      if not IsConnected(S_right): trace.partitions_rejected++; continue
      if not HasPredicateAcross(S_left, S_right): trace.partitions_rejected++; continue

      // 物理计划构建（详见 §8）
      BuildAnyPlans(cache, S, S_left, S_right, trace)
      BuildSortedPlans(cache, S, S_left, S_right, trace)

    // Sort enforcer 传播（详见 §9）
    PropagateSortEnforcers(cache, S, graph, trace)

    trace.dp_cells_filled = cache.size()

7. CCP Block 检查

CCP（Connected Complement Pair / csg-cmp-pair）条件确保 split 是合法的 CP-free join：

条件检查方式说明
① S_left ≠ ∅ 子集枚举保证 S_left 从非空 proper subset 枚举中产生
② S_right ≠ ∅ S_right = S \ S_left 因为 S_left ⊂ S，S_right 一定非空
③ S_left ∩ S_right = ∅ bitmask XOR 保证 disjoint by construction
④ IsConnected(S_left) graph.IsConnected() 左侧子集必须连通（否则不是合法的 join subtree）
⑤ IsConnected(S_right) graph.IsConnected() 右侧子集必须连通
⑥ Edge between (S_left, S_right) graph.HasPredicateAcross() 两侧之间至少有一条 join predicate（禁止 cross product） 
// 条件 ①-③ 由枚举保证，实际检查 ④-⑥：
if (!graph.IsConnected(S_left))          { rejected++; continue; }
if (!graph.IsConnected(S_right))         { rejected++; continue; }
if (!graph.HasPredicateAcross(S_left, S_right)) { rejected++; continue; }

// 通过 CCP 检查 → 这是一个合法的 join split
Connected 检查是 O(n) 吗？ 当前 IsConnected 的实现对每个 RelSet 做 BFS/检查。 但由于 CollectConnectedSubsets 已经预计算了所有连通子集， S_left 和 S_right 的连通性理论上可以预存。 不过 S_left 和 S_right 不一定在 connected_by_size 中出现 （因为它们是 S 的子集，而 connected_by_size 已经包含所有连通子集）， 所以更好的优化是预计算一个 bool is_connected[1 << n] 查找表。 当前实现为了简洁直接调用 IsConnected。

8. 物理计划生成：三阶段结构

8.1 总览：每个 CCP split 的三层处理

对每个通过 CCP 检查的 split (S_left, S_right)，DPSub 执行两个阶段的物理计划构建。 在所有 split 枚举完毕后，再执行第三个阶段——Sort enforcer 传播。 三层各有不同的目标和填充的 DP 表 key：

阶段何时执行填充的 key生产者消费者
(1) Optimize for Any 每个 CCP split 内 cache[S, Any] HashJoin, NestedLoopJoin, MergeJoin 上层 (1) 的 Any 输入；根节点最终答案
(2) Optimize for Sorted via predicates 每个 CCP split 内 cache[S, Sorted(join_col)] 本次 join 的 MergeJoin（输出天然有序） 上层 MergeJoin 的 Sorted 输入
(3) Sort enforcer propagation 所有 split 枚举完后 cache[S, Sorted(pred_col)] SortEnforcer 节点（显式排序） 上层 MergeJoin 的 Sorted 输入（当 join 的是其他列时）
DP 内层循环的三阶段位置： for each connected S (by increasing size): for each S_left ⊂ S: if not CCP: continue ╔══════════════════════════════════════════╗ ║ (1) Optimize for Any ║ → cache[S, Any] ╚══════════════════════════════════════════╝ ╔══════════════════════════════════════════╗ ║ (2) Optimize for Sorted via predicates ║ → cache[S, Sorted(join_col)] ╚══════════════════════════════════════════╝ ╔══════════════════════════════════════════════╗ ║ (3) Sort enforcer propagation ║ → cache[S, Sorted(other_col)] ╚══════════════════════════════════════════════╝

8.2 阶段 (1)：Optimize for Any property

为 S 产生输出顺序不限（Any）的最优计划，尝试三种物理 join： 

// 从 cache 取左右子集的 Any 计划
left_any  = cache["S_left|Any"]
right_any = cache["S_right|Any"]

// (1a) Hash Join — 输入无需有序
hash_join = MakeJoin(HashJoin, left_any, right_any, pred, Any)
cache["S|Any"] = min(cache["S|Any"], hash_join)

// (1b) Nested-Loop Join — 输入无需有序
nested_loop = MakeJoin(NestedLoopJoin, left_any, right_any, pred, Any)
cache["S|Any"] = min(cache["S|Any"], nested_loop)

// (1c) Merge Join — 输入需要 Sorted，查找左右子集是否已有按 join key 排好序的计划
// 例：pred = a.x=b.x，S_left={a}, S_right={b}
sorted_left  = cache["{a}|Sorted(a.x)"]
sorted_right = cache["{b}|Sorted(b.x)"]
if both exist:
  merge_join = MakeJoin(MergeJoin, sorted_left, sorted_right, pred, Any)
  cache["S|Any"] = min(cache["S|Any"], merge_join)
  // 注意：MergeJoin 自身输出标记为 Any（上层不要求有序），但它消费了 Sorted 孩子
Physical op子属性要求cost 公式 (cost_model.cpp)
HashJoin left: Any, right: Any left_cost + right_cost + left_rows + right_rows + output_rows
NestedLoopJoin left: Any, right: Any left_cost + right_cost + left_rows × right_rows
MergeJoin (for Any) left: Sorted(key_left), right: Sorted(key_right) left_cost + right_cost + left_rows + right_rows
MergeJoin 为何出现在 Any 阶段？ 虽然 MergeJoin 要求 Sorted 输入，但它的输出可以标注为 Any（上层不要求有序）。 如果子集的 Sorted 计划已经由 FillRelation 或之前的传播生成，且 MergeJoin 的 cost 低于 HashJoin/NLJoin， 它就会成为 cache[S, Any] 的最优候选。

8.3 阶段 (2)：Optimize for Sorted properties via matching predicates

对每条跨越 S_left 和 S_right 的 predicate，尝试用 MergeJoin 产生输出按 join 列有序的计划。 代码中遍历 predicate 的两个 "方向"（side 0 和 side 1），对应输出按 pred.left 或 pred.right 排序： 

// 对每个跨越 split 的 predicate
for pred in crossing_predicates:
  for side in {0, 1}:
    sort_key = (side == 0) ? pred.left : pred.right
    // 判断 sort_key.alias 在左子集还是右子集
    key_in_left = (S_left & MaskForAlias(sort_key.alias)) != 0

    // 确定左右各需要按什么键排序
    left_sort_key  = key_in_left ? sort_key : (side == 0 ? pred.right : pred.left)
    right_sort_key = key_in_left ? (side == 0 ? pred.right : pred.left) : sort_key

    // 查 cache：左右子集有没有对应的 Sorted 计划？
    sorted_left  = cache["S_left|Sorted(left_sort_key)"]
    sorted_right = cache["S_right|Sorted(right_sort_key)"]
    if both exist:
      merge_join = MakeJoin(MergeJoin, sorted_left, sorted_right, pred, Sorted(sort_key))
      cache["S|Sorted(sort_key)"] = min(cache["S|Sorted(sort_key)"], merge_join)

以 star query 中 S={a,b,c}、S_left={a,b}, S_right={c}、predicate a.y=c.y 为例：

迭代sort_keykey_in_left左需要右需要输出存入
side=0 (a, y) true Sorted(a.y) Sorted(c.y) cache[{a,b,c}, Sorted(a.y)]
side=1 (c, y) false Sorted(a.y) (← 注意!) Sorted(c.y) cache[{a,b,c}, Sorted(c.y)]

两次迭代需要的输入 children 完全相同（{a,b} Sorted(a.y) + {c} Sorted(c.y)）， 同一个 MergeJoin a.y=c.y 创建了两次，唯一区别是输出 property： 一次标注为 Sorted(a.y)，另一次标注为 Sorted(c.y)。 因为 MergeJoin 完成后结果在 a.y 和 c.y 上同时有序（它们是等值的）， 代码没有 "列等价" 推理，所以为两个别名各存一份（详见 §9）。

8.4 阶段 (3)：Sort enforcer propagation

阶段 (2) 只能产生 "join key 正好是本次 join 的 predicate 列" 的 Sorted 计划。 但上层可能需要 S 按一个与当前 join 无关的列排序。 例如 {a,b} 通过 a.x=b.x join 后，阶段 (2) 只产生了 Sorted(a.x) 和 Sorted(b.x)。 当上层要 join c on a.y=c.y 时，需要 cache[{a,b}, Sorted(a.y)]—— 这个条目阶段 (2) 没有生成，阶段 (3) 负责补齐： 

// 在 S 的所有 split 枚举完后执行
any_plan = cache["S|Any"]
if any_plan exists:
  // 遍历 S 中的每个 relation
  for relation in graph.Relations():
    if relation ∉ S: continue

    // 遍历所有 predicate（不只是 crossing 的），找到涉及该 relation 的列
    for pred in graph.Predicates():
      if pred.left.alias == relation.alias:
        sorted = MakeSort(any_plan, pred.left)         // SortEnforcer: cost + N log N
        cache["S|Sorted(pred.left)"] = min(..., sorted)

      if pred.right.alias == relation.alias:
        sorted = MakeSort(any_plan, pred.right)
        cache["S|Sorted(pred.right)"] = min(..., sorted)

MakeSort 创建一个 SortEnforcer 节点，cost 公式为： 

plan->cost = child->cost + child->rows × log₂(child->rows + 1.0);
//             ↑ 基础代价     ↑ O(N log N) 排序代价
阶段 (2) 和 (3) 的互补关系： (2) 是 "免费" 的有序性——MergeJoin 天然产出有序结果，无额外排序代价； (3) 是 "付费" 的有序性——在 Any 结果上加显式 SortEnforcer。 两者合在一起构成了 interesting order 机制：为上层 MergeJoin 提供所有可能被需要的 Sorted 计划。

8.5 完整数据流示例：star query 中 {a,b} → {a,b,c}

设定：star query a.x=b.x, a.y=c.y, a.z=d.z。追踪 S={a,b} 的生成及其被 S={a,b,c} 消费：

Phase 1: FillRelation cache[{a}, Any] ← SeqScan(a) cache[{a}, Sorted(a.x)] ← Sort(SeqScan(a), a.x) cache[{a}, Sorted(a.y)] ← Sort(SeqScan(a), a.y) cache[{a}, Sorted(a.z)] ← Sort(SeqScan(a), a.z) cache[{b}, Any] ← SeqScan(b) cache[{b}, Sorted(b.x)] ← Sort(SeqScan(b), b.x) cache[{c}, Any] ← SeqScan(c) cache[{c}, Sorted(c.y)] ← Sort(SeqScan(c), c.y) Phase 2, S={a,b} (join via a.x=b.x): ┌─ (1) Optimize for Any: │ HJ({a}Any, {b}Any) → cache[{a,b}, Any] ← 候选 │ NL({a}Any, {b}Any) → cache[{a,b}, Any] ← 候选 │ MJ({a}S(a.x), {b}S(b.x)) → cache[{a,b}, Any] ← 候选 │ ├─ (2) Optimize for Sorted via a.x=b.x: │ side=0: MJ({a}S(a.x), {b}S(b.x)) → cache[{a,b}, Sorted(a.x)] │ side=1: MJ({a}S(a.x), {b}S(b.x)) → cache[{a,b}, Sorted(b.x)] │ └─ (3) Sort enforcer: Sort({a,b}Any, a.y) → cache[{a,b}, Sorted(a.y)] ← 阶段(2)没有，这里补齐! Sort({a,b}Any, a.z) → cache[{a,b}, Sorted(a.z)] ← 同样补齐 Sort({a,b}Any, b.x) → 可能被 (2) 的 Sorted(b.x) 覆盖（若更优） Phase 2, S={a,b,c}, S_left={a,b}, S_right={c}: Crossing preds = [a.y=c.y] (只有一条跨越) ┌─ (1) Optimize for Any: │ MJ({a,b}S(a.y), {c}S(c.y)) → cache[{a,b,c}, Any] │ ↑ 可用，因为 (3) 补齐了 Sorted(a.y)! ├─ (2) Optimize for Sorted via a.y=c.y: │ side=0: MJ({a,b}S(a.y), {c}S(c.y)) → cache[{a,b,c}, Sorted(a.y)] │ side=1: MJ({a,b}S(a.y), {c}S(c.y)) → cache[{a,b,c}, Sorted(c.y)] └─ (3) Sort enforcer: 继续为 {a,b,c} 传播 Sorted 计划供上层使用…

关键洞察：如果没有阶段 (3)，cache[{a,b}, Sorted(a.y)] 不存在 → {a,b} join {c} 的 MergeJoin 无法生成 → 丢失一个可能的物理计划。 Sort enforcer 的存在保证了 interesting order 的传递闭包： 不只是 join key 本身，而是所有可能在未来被用到的排序列都能在 DP 表中找到。

9. Interesting Order 深入

9.1 为什么分开记录 Sorted(a.x) 和 Sorted(b.x)

阶段 (2) 对同一个 predicate a.x=b.x 生成了两份计划： 

side=0:  MJ({a}S(a.x), {b}S(b.x))  →  cache[{a,b}, Sorted(a.x)]
side=1:  MJ({a}S(a.x), {b}S(b.x))  →  cache[{a,b}, Sorted(b.x)]
//         ^^^^^^^^^^^^^^^^^^^^^^         ^^^^^^^^^^^^
//         同一个 join 计划被创建两次      property 标签不同

MergeJoin on a.x=b.x 的结果在 a.x 和 b.x 上同时有序 （因为它们是等值连接）。代码没有 "列等价"（equivalence class）推理，所以机械地为两个 alias 各存一份：

  • Sorted(a.x) 和 Sorted(b.x) 是不同的 property key——上层通过 MakeKey 查询时用 alias 区分。
  • 每次 MakeJoin 创建新 PlanPtr，cost 相同，只是 plan->property 字段不同。
  • 在生产级优化器中，会用 equivalence class 机制避免这种重复，但 demo 为简洁选择了直接存两份。

9.2 哪些列是 "interesting" 的

Sort enforcer 不会凭空发明排序列。它只处理出现在 join predicate 中的列—— 这从代码的过滤逻辑可以直接看出： 

for pred in graph.Predicates():         // ← 遍历所有 join 谓词
  if pred.left.alias == relation.alias  // ← 只有谓词中出现的列才被考虑
    → 生成 Sorted(pred.left)
  if pred.right.alias == relation.alias
    → 生成 Sorted(pred.right)
// 不在任何 predicate 中的列（如 a.g）→ 永远不生成 Sorted(a.g)

以 star query a.x=b.x, a.y=c.y, a.z=d.z 为例，对 S={a,b}：

列出现在 predicate 中?阶段 (3) 生成 Sorted?原因
a.x✓ (p0: a.x=b.x)✓未来 join on a.x 可能需要
a.y✓ (p1: a.y=c.y)✓未来 join c on a.y 时需要
a.z✓ (p2: a.z=d.z)✓未来 join d on a.z 时需要
b.x✓ (p0: a.x=b.x)✓反向排序方向
a.g✗（不在任何谓词中）✗没有任何 MergeJoin 会要求按 a.g 排序

这就是 interesting order 中 "interesting" 的精髓——只有可能被未来某个操作需要的物理属性才值得维护。 当前实现中，"可能被需要" 的判据是 "列出现在 graph.Predicates() 中"。 这同样适用于 Phase 1 的 FillRelation——singleton 阶段也只为 predicate 中涉及的列创建 Sorted 计划。

为什么不过滤？ 读者可能会问：阶段 (3) 为 a.z 生成了 Sorted(a.z)，但 d 还没 join 进来， 万一 d 不在最终查询中呢？当前实现不做跨 predicate 的 demand-driven 过滤—— 对所有 predicate 列都生成。在生产级优化器中，通常会自顶向下传递 "required property" 来做按需生成。 Demo 为了简洁选择了全量传播——对于 n≤10 的规模，额外开销可忽略。

9.3 MakeSort 代价模型

阶段 (3) 生成的 SortEnforcer 不是免费的——它带有 O(N log N) 的排序代价。 这意味着 cache[S, Sorted(col)] 的 cost 往往高于 cache[S, Any]。 但当上层 MergeJoin 消费它时，MergeJoin 的 cost 公式中不再包含排序项 （left_rows + right_rows，而非 NLJoin 的 left_rows × right_rows）， 可能在整体上更优。这正是 DP 表同时维护多种 property 的意义—— 让 cost 模型全局决定排序发生在哪里最划算。

10. Trace 计数器

DPSub 主要使用以下 trace 字段：

字段DPSub 中的含义
partitions_explored 内层循环中尝试的所有 (S_left, S_right) split 总数，包括最终被 CCP 拒绝的。
partitions_rejected 被 CCP 条件拒绝的 split 数量（连通性失败、无 crossing predicate 等）。
plans_costed 实际调用 MakeJoin 或 MakeSort 创建物理计划的次数。
cache_hits DPSub 没有显式 cache hit 计数器（总是直接 cache[key] 或 cache.find），此字段始终为 0。
plans_cached 搜索结束后 cache.size()：(RelSet, Property) 唯一条目总数。
duplicates_generated DPSub 不产生重复（无变换规则），始终为 0。
rule_applications DPSub 不使用变换规则，始终为 0。
dp_cells_filled 每处理完一个大小 i 的集合 S 后 cache.size() 的快照。注意：因为多个 S 可能更新同一批 key，这个值逐步增长最终收敛到 plans_cached。
best_cost 最终 full set 在 required property 下的最优计划 cost。
为什么 cache_hits=0 但 duplicates=0？ DPSub 是 bottom-up 枚举：每个子问题恰好被求解一次（从小集合到大集合）， 不需要 memoization cache hit 机制。与 Transformational 的变换规则不同， DPSub 不会从不同路径生成同一个子问题表达式，所以也没有 duplicate。

11. 与 DPccp 的关系

DPSub 和经典 DPccp（Moerkotte & Neumann, VLDB 2006）本质上是同一家族的 bottom-up DP 算法。 下表总结了关键异同：

维度DPccp (VLDB 2006)DPSub (当前实现)
枚举框架 csgCmpPairs：按连通子图 + 连通补图对枚举 遍历所有连通集合，对其所有 proper subset 做 CCP 检查
对称性 通过枚举顺序避免 (L,R) 和 (R,L) 重复 不避免 — 两个方向都被枚举和评估
子集枚举 按 neighborhood 扩增，避免枚举全部子集 (S-1)&S 风格的全子集枚举
Property 支持 原始论文主要讨论逻辑枚举；物理属性通常由上层优化器处理 内置 Any 和 Sorted 两种 RequiredProperty
复杂度 O(3n) 枚举，但常数更优（通过 neighborhood 剪枝） O(3n) 枚举，常数因子更大（两方向 + 全子集）
并行化 csgCmpPairs 的顺序依赖使并行化困难 内层循环天然并行（每个 S_left 独立处理）

Mancini 论文选择 "Generic DPSUB" 而非 DPccp 正是因为内层循环的并行友好性： 在 GPU/多核场景下，额外的常数因子可以被并行度摊平。

12.2 Step 1: FillRelation

三个单表各创建一个 SeqScan： 

cache["1|Any"] = SeqScan(a)          cost=1000, rows=1000
cache["2|Any"] = SeqScan(b)          cost=100,  rows=100
cache["4|Any"] = SeqScan(c)          cost=500,  rows=500

同时为 predicate 列生成 Sorted 计划： 

a 出现在 pred a.x=b.x → cache["1|Sorted(a.x)"] = Sort(SeqScan(a), a.x)
b 出现在 pred a.x=b.x → cache["2|Sorted(b.x)"] = Sort(SeqScan(b), b.x)
b 出现在 pred b.y=c.y → cache["2|Sorted(b.y)"] = Sort(SeqScan(b), b.y)
c 出现在 pred b.y=c.y → cache["4|Sorted(c.y)"] = Sort(SeqScan(c), c.y)

plans_costed = 3 + 4 = 7，cache size = 7。

12.3 Step 2: 连通子集分桶

connected_by_size[1] = { {a}, {b}, {c} }
connected_by_size[2] = { {a,b}, {b,c} }
connected_by_size[3] = { {a,b,c} }
// {a,c} 不连通，不在列表中

12.4 Step 3: size=2 DP

S = {a,b} (mask=3)

枚举 proper subset S_left：

S_leftS_rightexploredrejected原因
{a} (1){b} (2)10CCP ✓ (connected + edge a.x=b.x)
{b} (2){a} (1)20CCP ✓ (对称方向)

对 split ({a}, {b})，crossing preds = [a.x=b.x]： 

left_any  = SeqScan(a), cost=1000
right_any = SeqScan(b), cost=100
pred = a.x=b.x, sel=0.01

┌─ 阶段 (1) Optimize for Any ────────────────────────────
│ HashJoin(a,b):     cost=1000+100+1000+100+1000*100*0.01=3200
│ NestedLoop(a,b):   cost=1000+100+1000*100=101100
│ MergeJoin(a,b):    cost=cost(Sort(a,x))+cost(Sort(b,x))+1000+100=...
│ → cache["3|Any"] = min(3200, 101100, MJ_cost) = HashJoin(a,b) 或 MergeJoin
│
├─ 阶段 (2) Optimize for Sorted via a.x=b.x ─────────────
│ side=0: MJ({a}S(a.x), {b}S(b.x)) → cache["3|Sorted(a.x)"]
│ side=1: MJ({a}S(a.x), {b}S(b.x)) → cache["3|Sorted(b.x)"]
│   (同一个 MergeJoin，property 标签不同)
│
└─ 阶段 (3) Sort enforcer propagation (两个 split 枚举完后) ─
    Sort(Any, a.x) → cache["3|Sorted(a.x)"]  (与 (2) 竞争)
    Sort(Any, b.x) → cache["3|Sorted(b.x)"]  (与 (2) 竞争)

对 split ({b}, {a})：对称方向，产生 HashJoin(b,a) 等。 由于 cost 模型对称，HashJoin(b,a).cost = 100+1000+100+1000+10 = 2210。 最终 cache["3|Any"] = HashJoin(b,a) (cost=2210)。 (1) 和 (2) 对两个方向都执行，(3) 只在所有 split 枚举完后执行一次。

S = {b,c} (mask=6)

与 {a,b} 结构相同。split ({b},{c}) 和 ({c},{b})， pred = b.y=c.y, sel=0.05。 

┌─ (1) Any: HJ(b,c)=3700 → cache["6|Any"]
├─ (2) Sorted: MJ({b}S(b.y),{c}S(c.y)) → cache["6|Sorted(b.y)"], cache["6|Sorted(c.y)"]
└─ (3) Sort enforcer: 补齐 Sorted(b.y), Sorted(c.y) 等

12.5 Step 3: size=3 DP

S = {a,b,c} (mask=7)

枚举所有 proper subset：

S_leftS_rightexploredrejected原因
{a} (1){b,c} (6)30CCP ✓ (edge a.x=b.x across)
{b} (2){a,c} (5)41{a,c} 不连通
{c} (4){a,b} (3)50CCP ✓ (edge b.y=c.y across)
{a,b} (3){c} (4)60CCP ✓
{a,c} (5){b} (2)71{a,c} 不连通
{b,c} (6){a} (1)80CCP ✓

6 个 proper subset，2 个被拒绝，4 个合法。

以 split ({a,b}, {c}) 为例，crossing preds = [b.y=c.y] （注意：a.x=b.x 不跨越，因为它两端都在 {a,b} 中）： 

left_any  = cache["3|Any"] = HashJoin(b,a), cost=2210, rows=10
right_any = cache["4|Any"] = SeqScan(c), cost=500, rows=500
pred = b.y=c.y, sel=0.05

┌─ 阶段 (1) Optimize for Any ────────────────────────────
│ HashJoin((a,b),c):   cost=2210+500+10+500+10*500*0.05=3470
│ NestedLoop((a,b),c): cost=2210+500+10*500=7710
│ MergeJoin: 需要 left sorted by b.y, right sorted by c.y
│   left_sorted  = cache["3|Sorted(b.y)"]  ← (3) 补齐的!
│   right_sorted = cache["4|Sorted(c.y)"]  ← FillRelation 阶段生成的
│   MergeJoin 可行 → cache["7|Any"] ← min(HJ, NL, MJ)
│
├─ 阶段 (2) Optimize for Sorted via b.y=c.y ─────────────
│ side=0 (sort_key=b.y, key_in_left=true):
│   MJ({a,b}S(b.y), {c}S(c.y)) → cache["7|Sorted(b.y)"]
│ side=1 (sort_key=c.y, key_in_left=false):
│   MJ({a,b}S(b.y), {c}S(c.y)) → cache["7|Sorted(c.y)"]
│
└─ 阶段 (3) Sort enforcer propagation ───────────────────
    所有 4 个合法 split 枚举完后，在 {a,b,c} 的 Any 计划上加 SortEnforcer
    补齐可能被上层需要的 Sorted 列（此处为满集，上层无消费者，但仍生成）

类似地处理 ({a}, {b,c})、({c}, {a,b})、({b,c}, {a})。 最终 cache["7|Any"] 被选为 cost 最小的物理计划。

13. 五种策略对比

$ demos/demo/build/volcano_join_demo --compare --test chain_3

Strategy            Partitions    Rejected      PlansCosted   CacheHits     Cached        Dups          Rules         DPCells       Pruned        BestCost
DPSub               10            2             57            0             20            0             0             20            0             19651.56
MPDP                8             0             19            0             6             0             0             6             0             30200.00
Transformational    0             0             34            27            12            16            24            0             0             19651.56
TopDown(Naive)      7             1             25            7             12            0             0             0             0             19651.56
TopDown(Mincut)     6             0             25            7             12            0             0             0             0             19651.56

这个对比的几个观察：

  • property-aware 策略 BestCost 完全相同 (19651.56)；MPDP 使用 relation-set-only memo，未在内部传播 Sorted/MergeJoin，因此 BestCost 为 30200.00。
  • DPSub 的 Partitions=10 多于 MPDP=8，因为 MPDP 在发出 join-pair 前已经避开了一部分不连通候选。
  • TopDown(Naive)=7 少于 DPSub，是因为 TopDown 只枚举无序划分，不枚举左右两个对称方向。
  • DPSub 的 PlansCosted=57 最多，因为它为每个合法 split 都尝试 HashJoin、NestedLoopJoin、MergeJoin 三种物理实现，再加上 Sort Enforcer 传播。
  • DPSub 与 MPDP 都使用 dp_cells_filled，但 MPDP 只缓存 RelSet，因此 chain_3 的 DPCells 是 6 而不是 DPSub 的 20。
  • Transformational 的 CacheHits=27 与其他策略的 0/7/7 形成鲜明对比 — 这是它 "先构造完整逻辑 memo，再做物理优化" 的二阶段结构导致的。
  • TopDown(Mincut) 的 Partitions=6 且 0 拒绝 — Mincut 分区策略生成的都是合法 split。

14. 并行化潜力

Mancini 论文选择 "Generic DPSUB" 作为基础算法的一个关键原因是它的内层循环天然可并行化： 

for S_left ⊂ S:                              // 每个 S_left 独立
  S_right = S \ S_left
  if CCP(S_left, S_right):
    BuildPhysicalPlans(cache, S, S_left, S_right)  // 只读 cache[S_left|*] 和 cache[S_right|*]
                                                    // 只写 cache[S|*]

并行化的约束：

  • 可并行：同一个 S 内部的 S_left 循环 — 每个 S_left 只读取已经完成的更小集合的 cache entry，写入的是 S 的 cache entry。
  • 需同步：对 cache["S|Any"] 和 cache["S|Sorted(...)"] 的更新需要 atomic compare-and-swap 或最终归约。
  • 不可跨 S：size=i 的集合必须等 size<i 的集合全部完成后才能开始 — 这是 bottom-up DP 的固有顺序依赖。

当前实现是单线程的，但代码注释中已标注了并行化切入点。
