1. 核心直觉：从全集向下划分

传统的 bottom-up join 枚举（System R, DPccp）从单表开始，逐层往上拼出更大的 join tree。 Top-down partitioning 则反过来：

核心思想：给定目标关系集 \(S\) 和所需物理属性（如"有序"）， 把 \(S\) 划分成两个连通子集 \(L\) 和 \(R\)，递归求解 \(L\) 和 \(R\)， 然后用一个物理 join 把它们拼起来。从全集开始，层层向下直到单表。

这个想法来自 DeHaan & Tompa (SIGMOD 2007)。 论文的 key insight 是把 "CP-free bushy join 枚举" 规约为 minimal cut 问题： 在 join graph 中，删除一组 minimal cut 边后图恰好分成两个连通分量——这两个分量就是一个合法的 CP-free join 划分。

Top-down 方向带来了三个 bottom-up DP 没有的优势：

优势含义
按需计算 只递归到 join path 实际需要的子问题。不连通的关系集（如 chain_3 中的 {a,c}）永远不会被计算。
Branch-and-Bound 剪枝 一旦发现某个分支的下界已经 ≥ 已知最优计划，立即停止该分支的递归。
自然 property 传播 父节点的物理属性需求（如"有序"）自然推导出子节点需求，不需要预计算所有 property 变体。

2. 论文算法 → 代码映射

下面是 DeHaan & Tompa 论文的核心算法，用伪代码表示。理解这个伪代码就是理解整个策略。

2.1 论文伪代码

BestPlan(G, required_property):
    // Step 1: Memoization
    if memo[G, required_property] exists:
        return memo[G, required_property]

    // Step 2: Base case — 单表
    if |V(G)| == 1:
        return best scan satisfying required_property

    // Step 3: 枚举划分 & 递归
    best = ∞
    for each valid CP-free partition (G_left, G_right) of G:
        for each physical join operator:
            (left_prop, right_prop) = derive_child_properties(operator, required_property)

            // Predicted-cost branch-and-bound
            if lower_bound(G_left) + lower_bound(G_right) + min_join_cost ≥ best.cost:
                continue  // 剪枝

            left  = BestPlan(G_left,  left_prop)
            right = BestPlan(G_right, right_prop)
            plan  = MakeJoin(left, right, operator)
            best  = min(best, plan)

    // Step 4: SortEnforcer fallback
    if required_property is Sorted:
        any_plan = BestPlan(G, Any)
        best = min(best, Sort(any_plan, sort_key))

    // Step 5: 缓存 & 返回
    memo[G, required_property] = best
    return best

这个算法的本质是带 memoization 的递归划分。它不是"先生成所有逻辑等价表达式再优化"， 而是"每次划分直接尝试物理 join，递归求解子问题"。

2.2 C++ 代码映射

伪代码的每一行在 C++ 中都有直接对应：

伪代码C++ (top_down_partitioning.cpp)
memo[G, required_property] cache_: unordered_map<string, PlanPtr>，key = "RelSet|Property"
if memo[...] exists cache_.find(key) != cache_.end() → cache_hits++ → return
|V(G)| == 1 PopCount(relset) == 1 → MakeScan(relation)，若需排序则 MakeSort(scan, key)
for each valid CP-free partition EnumeratePartitions(graph, relset, trace) — Naive: 枚举子集 + IsValidJoinSplit() 过滤
for each physical join operator TryJoinMethods() — 试 HashJoin → NLJoin → MergeJoin
lower_bound(L)+lower_bound(R)+1 ≥ best LowerBound() = sum(scan_costs)，在 TryJoinMethods 入口检查 → branches_pruned++
Sort(fallback) BestPlan(S, Any) + MakeSort(result, sort_key)
memo[G, p] = best cache_[key] = best，在函数末尾写入

2.3 架构全景

下图将上述映射关系画成数据流图。注意它不包含 LogicalExpr、MemoStore、规则引擎— TopDown 只有一种中间表示：物理计划。

与 Transformational 的关键区别：没有 Phase 1 BuildMemo，没有 LogicalExpr，没有规则引擎。 整个搜索就是一个递归函数 + 一张缓存表。 这是因为在 join reorder 的语境下，(L, R) 划分天然就是 plan 生成器—不需要 commutativity/associativity 规则来发现等价关系。

3. chain_3 完整推演

这是理解 TopDown 如何工作的核心章节。我们用 chain_3 (a-b-c) 逐步推演 从 Search() 到最终计划的完整递归过程。

3.1 输入数据

Tables:   a (1000 rows, scan_cost=1000)    alias="a", bit=1
          b ( 100 rows, scan_cost=100 )     alias="b", bit=2
          c ( 500 rows, scan_cost=500 )     alias="c", bit=4

Predicates: a.x = b.x  (sel=0.01)
            b.y = c.y  (sel=0.05)

FullSet = 1|2|4 = 7

3.2 入口：Search()

Search(graph, stats, Any):
  cache_.clear()
  return BestPlan(graph, /*relset=*/7, /*property=*/Any, trace)

根调用是 BestPlan(7, Any)——从全关系集 {a,b,c} 开始，需要输出任意物理属性即可。

3.3 根调用：BestPlan(7, Any)

Step 1 — Memoization: key = "7|Any"。cache_ 为空，miss。

Step 2 — Base case: PopCount(7) = 3 ≠ 1。不是单表，跳过。

Step 3 — 枚举划分:

EnumeratePartitions(graph, /*S=*/7):
  遍历所有 S 的非空真子集作为 left：

  left=1 ({a})  → right=6 ({b,c})  left < right?  1<6 YES
    IsValidJoinSplit({a}, {b,c})?
      {a} 连通? YES (单节点)   {b,c} 连通? YES (有 predicate b.y=c.y)
      跨边存在? YES (a.x=b.x 连接 left 和 right)
    → YIELD (1, 6)  ✅

  left=2 ({b})  → right=5 ({a,c})  left < right?  2<5 YES
    IsValidJoinSplit({b}, {a,c})?
      {b} 连通? YES   {a,c} 连通? NO (a 和 c 之间没有 predicate)
    → REJECTED  ❌  partitions_rejected++

  left=3 ({a,b}) → right=4 ({c})   left < right?  3<4 YES
    IsValidJoinSplit({a,b}, {c})?
      {a,b} 连通? YES   {c} 连通? YES
      跨边存在? YES (b.y=c.y)
    → YIELD (3, 4)  ✅

  left=4 ({c})   → right=3 ({a,b})  left < right?  4<3 NO → skip (避免对称重复)

  结果: 2 个 valid partitions, 1 个 rejected

现在对两个 valid partition 分别调用 TryJoinMethods：

Partition 1: (L={a}, R={b,c})

TryJoinMethods({a}, {b,c}, Any, best=null):
  predicate = a.x=b.x (跨越 {a} 和 {b,c})
  B&B: best==null → skip prune

  HashJoin:  BestPlan({a}, Any) + BestPlan({b,c}, Any)  → cost 30200
  NLJoin:    BestPlan({a}, Any) + BestPlan({b,c}, Any)  → cost > Hash
  MergeJoin: BestPlan({a}, Sorted(a.x)) + BestPlan({b,c}, Sorted(b.x))
             → cost ≈ 14000+
  → best = HashJoin(a, b⋈c), cost=30200


Partition 2: (L={a,b}, R={c})

TryJoinMethods({a,b}, {c}, Any, best=HashJoin(30200)):
  predicate = b.y=c.y (跨越 {a,b} 和 {c})
  B&B: LowerBound({a,b}) + LowerBound({c}) + 1 = (1000+100) + 500 + 1 = 1601 < 30200
       → 不剪枝

  HashJoin:  BestPlan({a,b}, Any) + BestPlan({c}, Any)    → cost 30200
  NLJoin:    BestPlan({a,b}, Any) + BestPlan({c}, Any)    → cost > Hash
  MergeJoin: BestPlan({a,b}, Sorted(b.y)) + BestPlan({c}, Sorted(c.y))
             → cost 19651.6  ← WINNER!
  → best = MergeJoin(HJ(a,b)▸Sort(b.y), Sort(Scan(c),c.y)), cost=19651.6

Step 4 — SortEnforcer: property=Any，跳过。

Step 5 — 缓存: cache_["7|Any"] = MergeJoin(…)，返回。

3.4 子树展开

根调用触发了多个子问题。下面展开关键的几个。

BestPlan({b,c}=6, Any)

Partition: ({b}=2, {c}=4) ← 唯一有效划分

HashJoin(b,c):
  BestPlan(2, Any) → Scan(b)      c=100
  BestPlan(4, Any) → Scan(c)      c=500
  cost = 100+500+100+500+2500 = 3700

MergeJoin 尝试:
  BestPlan(2, Sorted(b.y)) → Sort(Scan(b),b.y)  c=764
  BestPlan(4, Sorted(c.y)) → Sort(Scan(c),c.y)  c=4984
  cost = 764+4984+100+500 = 6348 (>3700)

→ cache_["6|Any"] = HashJoin(b,c), cost=3700

BestPlan({a,b}=3, Sorted(b.y))

Partition: ({a}=1, {b}=2)

TryJoinMethods(1, 2, Sorted(b.y)):
  predicate = a.x=b.x
  sort_key = b.y, 但 predicate 列是 a.x 和 b.x
  → 没有 predicate 匹配 sort key → MergeJoin 不可能
  → best = null

SortEnforcer fallback:
  any = BestPlan(3, Any)
    → HashJoin(a,b): cost=3200
  sorted = Sort(any, b.y)
    cost = 3200 + 1000×log₂(1001) = 13167

→ cache_["3|Sorted(b.y)"] = Sort(HJ(a,b), b.y)

4. 关键设计决策

4.1 为什么 Memo Key 是 (RelSet, Property) 而不是 (RelSet, Property, Budget)？

DeHaan & Tompa 论文的一个重要发现是：accumulated-cost bounding 与 memoization 有根本冲突。

核心矛盾：如果把"当前已使用的预算"放进 memo key，同一个逻辑子图 (S, property) 会因为不同预算而被多次重新优化。 memoization 的复用能力被破坏——这正是论文实验中 TBNmcA（accumulated-cost）比穷举 TBNmc 慢 10–100 倍的原因。

TopDown 采用 predicted-cost bounding——基于关系集本身的统计属性（scan_cost 之和）做下界估计，不与调用路径的预算耦合。因此 memo key 只需要 (RelSet, Property)，不同调用路径可以安全共享缓存。

4.2 为什么用 Naive Partition（而不是论文的 MincutLazy）？

并非不想实现 MincutLazy，而是分两期演进有教学价值：

Phase策略方法教学重点
当前 Naive Gosper's hack 枚举所有子集 → IsValidJoinSplit() 过滤 理解 generate-and-test 的浪费，为 Mincut 做铺垫
第二期 Mincut Biconnection tree → 桥边 → minimal cut 直接生成 CP-free 划分 理解论文的 minimal cut 规约和 enumeration optimality

Naive 策略虽然在最坏情况下浪费（O(2n) per set），但它清晰地展示了"为什么需要 minimal cut"： 在 chain_3 中 3 个 raw partition 只有 2 个有效；在 cycle_4 中这个比例更差。 trace 中的 partitions_rejected 直观量化了这种浪费。

4.3 LowerBound 为什么这么简单（也太弱）？

当前 LowerBound = sum(scan_costs) 是一个故意简化的设计： 

double LowerBound(graph, relset, property) {
  double lb = 0.0;
  for (auto &r : graph.Relations())
    if (relset & (1<<r.id)) lb += r.scan_cost;
  return lb;
}

优点：安全 (admissible)

任何计划至少要扫描所有涉及的表，所以 LowerBound(S) ≤ true_optimal_cost(S) 恒成立。剪枝永远不会漏掉更好的计划。

缺点：太保守

在 toy 数据上，下界与实际 cost 差距可达 16×。导致 branches_pruned 在所有测试中都为 0——剪枝从未触发。

教学意图：这恰好展示了 branch-and-bound 的核心权衡——下界越紧，剪枝越强。 改进方向（加入 selectivity-weighted row estimates、predicate-dependent join cost lower bounds） 是理解 advanced top-down optimization 的自然下一步。

5. 与论文的差异

当前实现与 DeHaan & Tompa (SIGMOD 2007) 的对应关系：

论文当前实现匹配度
递归 BestPlan(G, prop) + memoization BestPlan(RelSet, Property) + cache_ 完全一致
CP-free partition = minimal cut Naive: 枚举子集 + IsValidJoinSplit() 过滤 简化版
MincutLazy: biconnection tree + lazy rebuild 尚未实现 待实现
Predicted-cost bounding (TBNmcP) LowerBound = sum(scan_costs) + B&B 架构一致，下界偏弱
Accumulated-cost bounding (TBNmcA) 未实现 刻意避免
Memo key = (G, property) MakeKey(RelSet, Property) 完全一致
Interesting orders via enforcer SortEnforcer fallback in BestPlan 一致
总的来说：当前实现是论文框架的忠实简化版。算法骨架（递归划分 + memoization + predicted-cost B&B + SortEnforcer）与论文完全对齐， 但在 partition 生成（Naive vs MincutLazy）和 lower bound 质量上做了教学性的简化。 这也是设计的初衷：先跑通框架，再做 Mincut 优化。

6. 与 Transformational 的本质对比

TopDown 和 Transformational 代表了 join enumeration 问题的两种根本范式。 它们的差异不是"一个快一个慢"，而是对"如何生成 join tree"这个问题的两种完全不同的回答。

6.2 逐维度对比

维度TransformationalTopDown
搜索空间生成 用 transformation rules (commutativity, associativity) 在已有表达式上派生出新等价表达式 用 关系集划分 (L, R) 直接生成 join 候选—划分即 plan
为什么有/无重复？ 同一个 LogicalExpr 可通过多条规则序列到达 → expression_keys_ 去重 → duplicate 计数器 每个 (RelSet, Property) 只有一个最优计划 → cache key 天然唯一 → 零重复
搜索阶段 两阶段: BuildMemo (逻辑枚举) → OptimizeGroup (物理优化) 单阶段: BestPlan 同时做枚举 + 代价计算 + 缓存
逻辑层 LogicalExpr (Get/Join) + MemoGroup + MemoStore 无—划分直接映射到物理 join
缓存结构 两层: expression_keys_ (逻辑去重) + best_cache_ (物理缓存) 单层: cache_ (RelSet, Property) → PlanPtr
Interesting Order OptimizeSorted 遍历 expr + SortEnforcer fallback 递归中自然产生 (MergeJoin 需要 Sorted child) + SortEnforcer fallback
Branch-and-Bound 无 Predicted-cost B&B (LowerBound + 1 ≥ best → prune)
扩展方式 加新 transformation rule 换 partition 策略 (Naive → Mincut)
复杂度 O(4ⁿ) 含重复 (Pellenkoft et al. 1997) O(3ⁿ) 枚举 (同 DP 下界)
一句话总结：Transformational 是"用规则变换表达式来探索 join order"； TopDown 是"用集合划分来直接生成 join order"。前者因为规则路径的重复而浪费 O(4ⁿ)；后者因为每个划分唯一地确定一个 join tree 结构而自然无重复。 这不是"TopDown 更快"，而是在 join reorder 这个问题上，集合划分比规则变换更直达本质。

7.4 数据说明

三个数据点清晰地展示了：

  1. TopDown vs Transformational 的 plans_costed 差距随 n 增长而扩大 — 前者不预计算逻辑层，工作量随"实际需要的子问题"增长；后者先构造完整逻辑空间。

  2. A. Pellenkoft, C. A. Galindo-Legaria, M. L. Kersten — "The Complexity of Transformation-Based Join Enumeration." VLDB 1997.
    证明 transformation-based join enumeration 产生 O(4ⁿ) duplicates。TopDown 对比 Transformational 的理论基础。
  3. G. Graefe, W. J. McKenna — "The Volcano Optimizer Generator." ICDE 1993.
    Volcano/Cascades 变换式搜索架构。Transformational 策略的灵感来源。
  4. G. Moerkotte, T. Neumann — "Analysis of Two Existing and One New Dynamic Programming Algorithm…" VLDB 2006.
    DPccp/DPsub — 目前最快的 bottom-up CP-free bushy join 枚举算法。
