0. 为什么这篇论文重要

SIGMOD 2022 Massively Parallel DP GPU Join Optimization

一句话概括：MPDP 试图同时拿到 DPSUB 的并行性和 DPCCP 的少枚举优势：不要像 DPSUB 那样遍历大量无效 Join-Pair，也不要像 DPCCP 那样把枚举顺序绑得过紧、难以大规模并行。

这篇论文关心的场景很现实：现代 BI / reporting / 自动生成 SQL 的系统里，一条查询可能有几十张甚至上百张表。执行引擎、存储布局、JIT、scale-out 都在进步，但优化器里 join order 这一步仍会因为指数搜索空间而成为瓶颈。

论文给出的目标有两个：

  1. 精确场景：在仍然求最优 bushy join order 的前提下，把优化时间降下来。
  2. 启发式场景：当查询大到精确 DP 不现实，用 MPDP 作为子过程，让启发式能系统地看更多候选空间，从而得到更便宜的计划。

论文的关键结果是：在 PostgreSQL 里的真实 benchmark 查询上，MPDP 相比当时的 state-of-the-art 精确算法至少快一个数量级；在相同时间预算下，PostgreSQL 的 heuristic fallback 门槛可以从 12 个关系提高到 25 个关系。对更大查询，MPDP 结合 IDP2 / UnionDP 后能找到成本显著更低的计划。

1. 大查询优化的两个瓶颈

论文把 DP join order 优化的效率拆成两个维度：

维度含义为什么影响大查询
Evaluated Join-Pairs 算法实际枚举并检查了多少候选左右子计划组合。 如果大量组合最后发现不是合法 join，时间会浪费在 generate-and-test 上。
Parallelizability 同一阶段的候选 Join-Pair 能否互不依赖地并行评估。 GPU / 多核 CPU 只有在任务粒度足够多、依赖少时才能发挥作用。

这两个维度之间有张力：

  • DPSUB / DPSIZE 枚举很规整，容易并行，但会检查大量无效 Join-Pair。
  • DPCCP 只枚举有效 connected-subgraph complement pair，因此工作量少，但枚举过程依赖 join graph 的结构，较难大规模并行。

2. 问题框架：Join-Pair 与 CCP-Pair

论文把查询看作 join graph \(G=(R,E)\)：顶点 \(R\) 是关系，边 \(E\) 是 inner equi-join predicate。一个 DP 枚举器会尝试把两个关系集合 \(S_{left}\)、\(S_{right}\) join 成更大的集合。

一个 Join-Pair 要成为合法候选，必须满足四个条件：

  1. \(S_{left}\) 和 \(S_{right}\) 都非空。
  2. 二者在 join graph 上各自诱导出的子图都是连通的。
  3. 二者互不相交。
  4. 二者之间至少有一条 join predicate 边相连。
满足这些条件的 Join-Pair 就是论文里的 CCP-Pair（Connected-subgraph Complement Pair）。这和 DPCCP / DPhyp 文献里的 csg-cmp-pair 是同一个核心概念。

论文还强调依赖关系：如果某个 Join-Pair 的左侧或右侧本身是另一个 Join-Pair 计算出来的结果，那么二者不能同时计算。DPSIZE / DPSUB 按结果集合大小推进时，同一层的 Join-Pair 往往没有依赖，因此很好并行；DPCCP 的枚举更精细，但也更难批量并行。

2.1 一个小例子

join graph:
  A -- B -- C

valid Join-Pairs:
  ({A}, {B})
  ({B}, {C})
  ({A,B}, {C})
  ({A}, {B,C})

invalid Join-Pair:
  ({A}, {C})       // 二者之间没有 join edge，会变成 Cartesian product

DPSUB 式枚举会容易走到 ({A}, {C}) 这种需要检查后丢弃的候选；DPCCP 避免了它；MPDP 的目标是大部分时候也避免它，同时让候选生成仍有规整的并行结构。

3. 从 DPSUB / DPCCP 看 MPDP 的位置

算法枚举风格优点短板
DPSIZE 按 relation set size 从小到大枚举。 概念简单，很多系统沿用类似思想。 会评估大量无效 Join-Pair。
DPSUB 按 subset precedence order 枚举所有子集。 同层候选较独立，容易 CPU/GPU 并行。 仍然需要大量有效性检查。
DPCCP 按 join graph 直接枚举 CCP-Pair。 只枚举有效候选，避免 Cartesian product。 枚举依赖强，难以发挥 massive parallelism。
MPDP 对每个 connected set \(S\)，用 graph 结构直接构造更少候选。 接近 DPCCP 的少枚举，同时保留 DPSUB 的按层并行。 实现更复杂，需要图算法、bitmap/memo/GPU 工程配合。

这篇论文的漂亮点在于它没有只说“GPU 更快”。真正的算法贡献是：先减少 GPU 不该做的无效工作，再把剩下的工作排成 GPU 擅长处理的批量任务。

4. MPDP 的核心算法

MPDP 仍是动态规划：为每个 relation set \(S\) 保存 BestPlan(S)，逐层从小集合构造大集合。变化在于：对每个 \(S\)，它如何枚举候选 Join-Pair。

4.1 Tree join graph：删边即可得到候选

如果 join graph 是树，那么任意 connected set \(S\) 的诱导子图也是树。树有一个非常直接的性质：删掉任意一条边，都会把树分成两个连通分量。这两个分量正好构成一个合法 CCP-Pair。

MPDP:Tree
for size i = 2..n:
  for each connected set S with |S| = i:
    for each edge e in induced_tree(S):
      (S_left, S_right) = components_after_removing(e)
      curr = CreatePlan(S_left, S_right)
      BestPlan(S) = min(BestPlan(S), curr)

树场景下，大小为 \(i\) 的 connected set 只需要看 \(i-1\) 条边，因此 EvaluatedCounter 可以等于 CCP-Counter。星型和雪花 schema 都常落在这个直觉附近，所以这是数据仓库查询里的关键优化。

4.2 Cyclic join graph：先找 block，再 grow

一般图里会有环，删一条边不一定把图切开。论文使用 graph theory 里的 block（双连通分量）来处理：对 \(S\) 找 blocks；在每个 block 内枚举候选 \(lb\) 和补集 \(rb\)，检查它们是否在 block 内构成 CCP；然后用 grow 把局部 CCP 扩展回整个 \(S\)。

MPDP for general graphs
for size i = 2..n:
  S_i = { S | S connected and |S| = i }
  for each S in S_i:
    blocks = FindBlocks(S)
    for each block in blocks:
      for each non-empty lb subset of block:
        rb = block - lb
        if (lb, rb) is not CCP inside block:
          continue
        S_left  = grow(lb, S - rb)
        S_right = S - S_left
        curr = CreatePlan(S_left, S_right)
        BestPlan(S) = min(BestPlan(S), curr)

grow(lb, S - rb) 的作用是：从 \(lb\) 出发，只在不经过 \(rb\) 的限制集合中继续扩张，得到整个 \(S\) 内应该放在同侧的最大连通区域。这样，block 内的局部切分被提升成 \(S\) 上的合法 Join-Pair。

这个地方容易误读：MPDP 不是简单“遍历每条边再切图”。树上可以这么理解；有环图上需要 block 和 grow，否则会漏掉候选或重复不必要候选。

4.3 正确性直觉

论文的正确性思路可以压成一句话：每个合法 CCP-Pair 都能在某个 block 的某个局部切分中被发现，再通过 grow 还原到 \(S\) 上；同时生成出来的候选经过连通性、非空、互斥、相邻性检查，仍是合法 Join-Pair。

因此 MPDP 仍是精确算法，不是剪掉 plan space 的启发式。它减少的是无效候选的评估，不是删除可能成为最优计划的合法候选。

5. 为什么 MPDP 更适合并行和 GPU

MPDP 的并行化来自两个层次：

  1. DP 层级并行：按 set size \(i\) 逐层推进；同一层的 set \(S\) 之间可以并行处理。
  2. 候选内部并行：对每个 \(S\)，block / subset / Join-Pair 检查可以批量展开，适合 GPU warp-level parallelism。

论文的 GPU 实现大致是下面这条流水线：

for size i = 2..n:
  unrank:   生成大小为 i 的所有 relation sets
  filter:   过滤掉不连通的 sets
  evaluate: 对每个 connected S 执行 MPDP Join-Pair 枚举和 cost
  prune:    每个 S 只保留 best join
  scatter:  把 (S, BestPlan(S)) 写回 memo table
工程点论文做法为什么重要
Relation set 表示 固定宽度 bitmap set。 集合 union/intersection/difference 都变成位运算，适合 GPU。
Memo table Murmur3 hashing + open addressing。 GPU 上需要简单、连续、可预测的数据结构。
Unrank 用 combinatorial unranking 生成固定大小集合。 避免递归枚举，把任务转成可并行索引空间。
Filter 用 stream compaction 过滤不连通 set。 减少后续 evaluate 阶段的无效输入。
Branch divergence 用 Collaborative Context Collection 缓解 warp 内分支发散。 简单 if 过滤会让部分线程等待，浪费 SIMD 并行度。

我的理解是：MPDP 的 GPU 化不是“把 DP 搬到显卡上”这么粗暴。它先把 DP 的图枚举改写成更规则的数据并行任务，再做 GPU 友好的内存和控制流设计。

6. 超大查询上的启发式：IDP2-MPDP 与 UnionDP

MPDP 精确算法更快，但 join order optimization 本身仍是 NP-hard。对几百张、上千张表的查询，论文还是需要启发式。

6.1 IDP2-MPDP

IDP2（Iterative Dynamic Programming version 2）的思路是：先生成一个 tentative join tree，然后反复挑出其中较贵、大小不超过 \(k\) 的子树，用精确 DP 重新优化局部。论文把这个局部精确优化器替换为 MPDP。

IDP2-MPDP:
  initial_plan = heuristic_join_order(query)
  while there is an expensive subtree with size <= k:
    optimized_subtree = MPDP(subtree)
    replace subtree in plan
  return plan

MPDP 使得 \(k\) 可以取更大值；论文提到在 snowflake schema 上，GPU 加速 MPDP 可把 \(k\) 用到 25，并在 100 表场景保持可接受优化时间。

6.2 UnionDP

UnionDP 是论文提出的新启发式。它先根据图结构和 cost model 把关系合并成不超过 \(k\) 大小的 partition；每个 partition 内用 MPDP 求局部最优；然后把每个局部计划压成 composite node，在更小的图上递归重复。

UnionDP(G, k):
  if |G| <= k:
    return MPDP(G)

  initialize one disjoint-set per relation
  sort edges by size(leftRelSet + rightRelSet), then edge weight
  for edge in sorted_edges:
    if union size <= k:
      union(leftRelSet, rightRelSet)

  for each partition:
    subplan = MPDP(induced_subgraph(partition))
    create composite node

  G' = graph(composite_nodes, cut_edges)
  return UnionDP(G', k)
UnionDP 的精神和 Query Simplification 有一点像：先对大图做结构化降维，再把剩余问题交给精确 DP。但 UnionDP 是构造 partition / composite node；Query Simplification 是给 join graph 增加顺序约束，保留一个受限后的 DP 问题。

7. 实验结论怎么读

7.1 实验设置

论文把所有算法都实现在 PostgreSQL 12 的优化器中。硬件环境包含双路 Intel Xeon E5-2650L v3（共 24 cores / 48 threads）、755GB RAM，以及 Nvidia GTX 1080 GPU。论文没有使用完整 PostgreSQL 原生 cost model，而是实现了一个接近 PostgreSQL 的 cost model；对实验查询，最坏情况下和 PostgreSQL cost 相差在 5% 内。

这个设置很重要：论文比较的是 optimization time 和估计 cost 下的 plan quality，不是端到端执行时间。作者也明确说，cost/cardinality error 的处理不在本文范围内。

7.2 精确算法：MPDP 把 fallback 门槛推高

在 star / chain / cycle / clique / JOB / MusicBrainz 等场景里，论文比较了 PostgreSQL 原优化器、DPCCP、DPE、GPU DPSub/DPSize、CPU/GPU MPDP 等。总体结论：

  • MPDP 在大查询上明显快于已有精确 DP 方案。
  • GPU MPDP 在 star / snowflake 这类常见分析型结构上尤其有优势。
  • 在相同时间预算下，可以把 PostgreSQL 从 12 表左右就 fallback 到启发式的门槛提高到约 25 表。

7.3 启发式：更大局部精确空间带来更好计划

论文比较 GEQO、GOO、LinearizedDP、IKKBZ、IDP2-MPDP、UnionDP-MPDP 等启发式，在 snowflake、star、clique 等大图上看相对 plan cost。

场景实验观察阅读时的注意点
Snowflake schema UnionDP-MPDP 和 IDP2-MPDP 计划质量强，论文报告可比 state-of-the-art 技术便宜最多约 7 倍。 这类图结构让 partition / 局部精确优化很有发挥空间。
Star schema IDP2-MPDP、UnionDP-MPDP 在许多规模上达到接近最优的 cost。 star 的结构简单，但 relation 数大时精确枚举仍会很贵。
Clique / dense graph 所有方法都更早遇到超时或退化；UnionDP 在 dense graph 上不如在 snowflake 上稳定。 边太多会让 partition balance 和 cut edge 选择变难。
JOB / MusicBrainz 用于验证真实 benchmark 图形下的优化时间和枚举行为。 这些实验更接近真实优化器输入，但仍依赖论文实现的 cost model。
不要把“plan cost 更便宜”直接读成“真实执行时间必然等比例更快”。这篇论文主要研究优化器枚举效率和估计 cost 下的计划质量；真实执行仍受 cardinality estimation、operator implementation、缓存、并行执行等影响。

8. 与 join order 学习路线和 demo 的关系

如果把这篇论文放到本仓库的 join order 学习路线里，可以这样定位：

  1. System R / DPsize：理解为什么 join order 可以用 DP 按集合大小求解。
  2. DPsub：理解 subset 枚举为何规整、并行友好，但会产生无效 Join-Pair。
  3. DPCCP / DPhyp：理解 graph-aware 枚举如何只生成 connected complement pair。
  4. MPDP：理解如何在“少枚举”和“高并行”之间重新设计枚举器。
  5. Query Simplification / Heuristic Search：理解当精确 DP 仍超预算时，如何做有控制的搜索空间退化。

8.1 对 demo 的启发

本仓库的 demo 若要吸收 MPDP 思路，不必一上来实现 GPU。更适合的教学路径是：

  • 先统计 Join-Pair。在现有 DPSub / DPCCP trace counters 里区分 evaluated Join-Pair、valid CCP-Pair、invalid Join-Pair。
  • 实现 tree-specialized MPDP。对 tree / star / snowflake join graph，用“删边得到候选”的规则替代 subset 枚举。
  • 再做 general graph block + grow。这一步可以作为高阶实验，展示 block decomposition 如何减少无效枚举。
  • 最后谈并行。即使不写 CUDA，也可以用同层 set 独立性解释为什么它适合 task parallel 或 GPU batch。

这样做能把论文最重要的思想讲清楚：并行优化器不是只靠硬件堆线程，而是要先把枚举结构整理成硬件能高效吃下去的形状。

9. 边界与实现提醒

边界说明对实现者的提醒
Join 类型 论文主要处理 inner equi-join graph。 outer/semi/anti join 或复杂 predicate 需要 hypergraph 语义约束，不能直接照搬。
Cost model 实验使用接近 PostgreSQL 的自实现 cost model。 真实系统的完整 cost model 更复杂，GPU 上复刻成本很高。
GPU 工程 性能依赖 bitmap、hash memo、stream compaction、warp-level scheduling。 算法正确只是第一步，数据布局和 branch divergence 会决定实际收益。
Dense graph clique 这类图对所有方法都难。 MPDP 不是指数复杂度的解药，只是在很多结构上减少常数和无效工作。
Plan quality IDP2-MPDP / UnionDP 是启发式。 它们靠更大局部精确搜索提高质量，但不保证全局最优。

我的理解是：MPDP 最适合教学我们如何评价 join order 算法的“工程可用性”。一个枚举器不仅要理论上少枚举，还要让每一批工作能被现代硬件并行消化。

10. 读完应记住的结论

结论一句话解释
MPDP 是精确 DP，不是启发式本身。 它减少无效 Join-Pair 评估，但不删除合法 plan space。
核心指标是 evaluated Join-Pair / valid CCP-Pair。 大查询优化时间很大程度上浪费在枚举后又丢弃的候选上。
Tree join graph 上，删边就是候选生成器。 大小为 \(i\) 的 connected tree set 只需看 \(i-1\) 条边。
一般图需要 block + grow。 先在双连通分量内找局部切分，再扩展回整个 relation set。
GPU 加速依赖算法形状。 bitmap set、unrank/filter/evaluate/scatter 流水线、branch divergence 控制同样关键。
MPDP 还能作为启发式子过程。 IDP2-MPDP 和 UnionDP-MPDP 用更大的局部精确搜索改善超大查询的 plan quality。
它连接了 DPsub 和 DPCCP 两条路线。 MPDP 的目标就是结合 subset DP 的并行性与 graph-aware DP 的少枚举。
最后的心智模型：DPCCP 问“如何只枚举正确候选”，DPSUB 问“如何规整地并行枚举”，MPDP 问“能不能把这两件事同时做得足够好”。这就是它在 join order 学习路线里的位置。
