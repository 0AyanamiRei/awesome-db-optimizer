0. 读这篇论文前要带着的问题

SIGMOD/PACMMOD 2023 A* Join Ordering

传统 join order 枚举通常有两条路线：一条是 System R 以来的 bottom-up DP，另一条是 DeHaan/Tompa、Fender/Moerkotte 一类 top-down partitioning。它们的共同点是：若要保证最优，通常会系统性枚举 connected complement pairs（ccp），只是枚举顺序和去重方式不同。

这篇论文问的是一个略微不同的问题：

如果我们把 join order optimization problem（JOOP）看成 shortest path problem，那么能不能用 AI planning 里成熟的 best-first / heuristic search，只探索“看起来最有希望”的一小部分搜索空间，同时仍然保留最优性条件？

论文的答案是：可以。关键不在于“把 A* 套上去”这么简单，而在于三件事：

  1. 给 JOOP 一个严格的 shortest path 规约，使每条从起点到终点的路径对应一个可行 join plan。
  2. 把指数级搜索空间按需展开，而不是预先构造完整图。
  3. 解决 open list、duplicate、edge weight、heuristic function 与 DBMS cost model 之间的几个细节坑。

我的理解是：这篇论文不是要替代 DPCCP 的“枚举理论”，而是把枚举问题换了一个控制流。DPCCP 按结构把该枚举的 ccp 都枚举完；heuristic search 则按当前 \(g+h\) 的优先级推进，可能在看到全部 ccp 之前已经证明某条完整路径足够好。

1. 从 JOOP 到 shortest path

1.1 查询图、子问题与 join 操作

论文把一个查询块表示成 query graph：

\[ G_Q=(R,J) \]

其中 \(R\) 是 relations 集合，\(J\) 是 join predicates / join edges。一个 subproblem 是关系集合 \(S \subseteq R\)。若 join \(j=\{r_1,r_2\}\) 的两个端点分别落在两个互不相交的子问题 \(S_1,S_2\) 中，就可以把它们 join 成 \(S_1 \cup S_2\)。

论文为了把 plan 写成“join 序列”，定义了 hoisted join \(j^*\)：它不是直接接收 \(S_1,S_2\)，而是在当前 vertex 的一组 disjoint subproblems 中，自动找到能由 \(j\) 连接的那两个 subproblem，并把它们替换为并集。

当前 vertex: {{A}, {B,D}, {C}}
应用 j_AB:
  选择 {A} 与 {B,D}
  生成 {{A,B,D}, {C}}

于是，一个 plan 就可以看成 join 的序列 \(p=j_1j_2...j_n\)。只要从所有单表集合出发，最后得到 \(\{R\}\)，这个 join 序列就是可行 plan。

1.2 Search vertex 是什么

在 shortest path 规约中，一个 vertex 不是 join tree 节点，而是一组“当前还没合并完”的 disjoint subproblems：

对象Bottom-up 视角例子
start vertex \(n_0\) 所有关系都是单表子问题 \(\{\{A\},\{B\},\{C\},\{D\}\}\)
intermediate vertex 已有一些关系被 join 成较大的子问题 \(\{\{A,B\},\{C,D\}\}\)
goal vertex \(n^*\) 所有关系已经 join 成完整结果 \(\{\{A,B,C,D\}\}\)
edge 执行一次 join，把两个 subproblem 合并 \(\{\{A\},\{B\},\{C,D\}\}\rightarrow\{\{A,B\},\{C,D\}\}\)

边权重来自 DBMS cost model \(C(S_1,S_2,j)\)：执行这一步 join 的代价。于是，路径权重就是所有 join 步骤的代价之和。

核心规约：SPJOOP 中从 \(n_0\) 到 \(n^*\) 的每一条路径，对应一个可行 join order；这条路径的权重，对应这个 join plan 的 cost。求 shortest path 就是在求最优 join order。

1.3 搜索空间为什么仍然是指数级

把 NP-hard 的 JOOP 规约到 shortest path，并没有把问题变成多项式。原因是 SPJOOP 本身的图可能指数大。论文用 Ono & Lohman 的 ccp 计数给出构造性论证：clique 查询有 \(\Theta(3^{|R|})\) 个 connected complement pairs；每个 ccp 至少对应 SPJOOP 中某个 vertex，因此 vertex / edge 数量在最坏情况下仍是指数级。

重点：shortest path 规约的价值不是“复杂度魔法消失”，而是允许使用 informed search。A* 不需要提前构造完整搜索图，它可以按需展开，并用 heuristic function 把搜索集中到更可能通向低代价 plan 的区域。

2. Bottom-up 与 top-down 的对偶

论文很漂亮的一点是把 bottom-up 和 top-down join ordering 放进同一个 search graph 里看。

方向起点一次扩展目标
Bottom-up 所有单表子问题 选两个可 join 的 subproblems，合并成一个更大的 subproblem 完整关系集合 \(\{R\}\)
Top-down 完整关系集合 \(\{R\}\) 把一个 subproblem 分割成两个由 join predicate 连接的子问题 所有单表子问题

在 Figure 1b 的搜索图里，top-down 就是沿着 bottom-up 的边反方向走。它不是另一个问题，而是同一个 shortest path 问题的 dual problem（对偶问题）。这也解释了为什么论文能比较 \(A^*_\uparrow\) 与 \(A^*_\downarrow\)：它们使用同一个规约，只是 start / goal 和 expand 方向不同。

论文中的实验结论也体现了这种对偶不是纯形式化游戏：chain / cycle 更适合 top-down 加 \(h_{sum}\)，star / clique 更适合 bottom-up 的 \(h_{zero}\) 版本。也就是说，搜索方向本身是一种优化器策略选择。

3. A* 为什么能用于 join order

3.1 Heuristic function 的性质

在 shortest path 里，heuristic function \(h(v)\) 估计的是：从当前 vertex \(v\) 到 goal 的最短剩余代价。论文沿用经典 A* 概念：

术语含义对 join ordering 的影响
goal-aware 若 \(v\) 是 goal，则 \(h(v)=0\) SPJOOP 中 goal 很容易检查，论文默认 heuristic 都满足这一点。
admissible 从不高估到 goal 的真实最短剩余代价 A* 使用 admissible heuristic 时能保证找到 optimal plan。
consistent 对每条边 \((u,v,w)\)，有 \(h(u)\le h(v)+w\) consistent 且 goal-aware 会推出 admissible；经典结果给出 A* 的 optimally efficient 性质。
optimal heuristic \(h^*\) 精确返回从 \(v\) 到 goal 的 shortest path 权重 最理想但不可指望高效计算。

3.2 为什么最优 heuristic 不可能便宜

论文的 Theorem 1 说明：除非 P = NP，否则 SPJOOP 的 optimal heuristic \(h^*\) 不可能是 PTIME 可计算的。证明思路很直接：如果 \(h^*\) 能多项式时间算出来，搜索时每一步只要选使 \(w+h^*(v)\) 最小的后继，就能沿着一条最短路径走，等价于多项式时间解决 NP-hard 的 JOOP。

这给 heuristic 设计定了一个现实边界：不要幻想有一个又准又便宜的完美估计器。工程上要找的是“足够便宜、足够有信息量”的 \(h\)，并清楚它是否 admissible。

3.3 按需搜索而不是提前建图

搜索空间指数大，所以论文的 BFS / A* 框架不会预先构造完整 SPJOOP，而是在每次从 open list 弹出当前最优 vertex 后才调用 expand 生成后继。

open_list = [start]
while open_list 非空:
    u, g_u = extract_best(open_list)
    if u is goal:
        return path

    for each edge (u, v, w) in expand(u):
        add(open_list, v, g_u + w)

对 A* 来说，extract_best 选择的是 \(g(v)+h(v)\) 最小的 vertex，其中 \(g(v)\) 是从 start 走到 \(v\) 已经付出的代价，\(h(v)\) 是估计剩余代价。

论文特别提醒：A* 是在 goal 被 extract 出 open list 时终止，不是在 goal 第一次被 generated 时终止。因为同一个 goal 之后可能通过更便宜的路径再次生成。

4. 搜索空间表示与 vertex expansion

4.1 Vertex representation：bit vector 的集合

一个 vertex 是 set of sets，直接存很笨。论文使用 bit vector 表示 subproblem：给每个 relation 一个编号，subproblem 中包含某 relation 就把对应 bit 置 1。然后一个 vertex 就是按字典序排序的 bit vector 序列。

A -> 1, B -> 2, C -> 3, D -> 4
{C,D}        -> 0011
{{A},{B},{C,D}} -> [1000, 0100, 0011]  // 按论文示例的 bit 方向

字典序排序有两个作用：

  • 方便 hash 与 equality testing，用于 open list / duplicate detection。
  • 为后面的 PreDup 提供一个稳定的 canonical order。

4.2 Bottom-up expansion

Bottom-up expand 很直接：遍历当前 vertex 中的 subproblem pair \((S_1,S_2)\)，检查是否有 join \(j\) 能连接它们。若能，则生成后继 vertex：移除 \(S_1,S_2\)，插入 \(S_1\cup S_2\)，并保持字典序。

expandBottomUp(Vu):
    for each pair (bi, bk) in Vu:
        for each join j connecting bi and bk:
            Vv = Vu - {bi, bk} + {(bi | bk)}
            sort Vv lexicographically
            yield Vv by j(bi, bk)

论文给出的粗略时间复杂度上界是 \(O(|R|^3\cdot |J|)\)。这个上界看着不轻，但实验剖析显示，expand 本身不是主要瓶颈。

4.3 Top-down expansion

Top-down expand 是反向操作：遍历当前 vertex 中每个 subproblem \(S\)，枚举把 \(S\) 切成两个 connected subgraphs 的 ccp partition。论文选择已有的 PARTITIONMinCutAGaT / MinCutAGaT 思路来做分割枚举。

expandTopDown(Vu):
    for each subproblem S in Vu:
        for each connected partition (b1, b2) of S:
            Vv = Vu - {S} + {b1, b2}
            sort Vv lexicographically
            yield Vv by j(b1, b2)

论文采用 MinCutAGaT 是一个工程取舍：MinCutBranch 在某些拓扑上有更好渐近性质，但实现复杂；而该论文的 profile 结果显示 vertex expansion 只占总体时间的一小部分，所以选择较容易落地的分割算法。

5. DeDup 与 PreDup：两个关键工程优化

5.1 DeDup：open list 里的 duplicate handling

A* 的 open list 需要高效支持：

  • 找到 \(g+h\) 最小的 vertex；
  • 删除最小元素；
  • 插入新生成的后继；
  • 如果同一 vertex 以更小的 \(g\) 再次到达，则 decrease-key。

论文用 heap + hash table：heap 管 find-min/delete-min/insert/decrease-key，hash table 管“这个 vertex 是否见过、当前最小 \(g\) 是多少、它是否还在 heap 里”。

add(L, v, g_new):
    if v not seen:
        insert heap with key g_new + h(v)
        remember g_new and heap handle
    else if g_new < remembered_g(v):
        if v is still in heap:
            decrease_key(v, g_new + h(v))
        else:
            insert v again with cheaper g
        remember g_new
    else:
        discard duplicate
论文 Table 2 的信息很强：如果 open list 允许 duplicate 而不做检测，同一个 duplicate 弹出后会继续 expand，它生成的后继又大多是 duplicate，于是 duplicate 会指数式膨胀。DeDup 不改变探索到的 unique vertices，但能显著减少无意义扩展。

5.2 Desired duplicate vs undesired duplicate

不是所有 duplicate 都该在生成前被禁止。论文区分了两类：

类型例子是否应该保留机会
Desired duplicate \((A\Join B)\Join C\) 与 \(A\Join(B\Join C)\) 到达同一个 relation set，但对应不同 partial plan。 应该生成，让 DeDup 比较 \(g\)，因为不同 partial plan 代价可能不同。
Undesired duplicate 先做 \(A\Join B\) 再做 \(C\Join D\)，或反过来先做 \(C\Join D\) 再做 \(A\Join B\)。两个 join 独立，partial plan 语义相同，总代价也相同。 可以在 expand 阶段直接禁止其中一条路径。

论文把第二类称为 redundant path：若两个 join 作用在四个互不相交的 subproblems \(q,r,s,t\) 上，且 \(q\) 可与 \(r\) join，\(s\) 可与 \(t\) join，那么这两个 join 的顺序可以交换，得到同一个 vertex，且路径权重相同。

5.3 PreDup：用字典序禁止 redundant path

PreDup 的技巧很小但很有效：在 vertex representation 中标记“最近一次 join 产生的 subproblem”，记作 \(\nabla(V)\)。Bottom-up expand 时，如果当前候选 join 的结果 \((b_i|b_k)\) 在字典序上小于 \(\nabla(V_u)\)，就跳过。

expandBottomUp with PreDup:
    for each pair (bi, bk):
        joined = bi | bk
        if joined <lex marked_recent_join_result(Vu):
            continue   // redundant path, suppress it
        yield successor and mark joined as recent

直觉上，它给彼此独立的 join 排了一个 canonical order：最近生成的 subproblem 只能按字典序“变大”，从而避免同一批独立 join 被任意排列后反复到达同一 vertex。

论文给出的正确性直觉是：PreDup 删掉的是“存在同权替代路径”的 redundant path。可达性仍在，soundness 不变；如果被删路径是最优路径，则保留下来的同权路径也是最优路径。

PreDup 对 star topology 没有效果，因为 star 查询没有 bushy plans，也就没有这类 independent bushy joins 的排列冗余。对 chain、cycle、clique，论文 Table 3 显示 duplicate 和部分 new vertices 都会减少。

6. 代价模型与启发函数

6.1 Cout 与 edge weight 的坑

论文实验采用常见的 \(C_{out}\)：计划代价等于所有 intermediate results 的 cardinality 之和。递归写法是：

\[ C_{out}(T)= \begin{cases} 0, & T\in R \\ |T| + C_{out}(T_1)+C_{out}(T_2), & T=T_1 \Join T_2 \end{cases} \]

最自然的 edge weight 定义是：

\[ C(S_1,S_2,j)=|S_1\Join S_2| \]

但是这会给 bottom-up search 带来一个很微妙的效率问题：最终结果 cardinality \(|R|\) 对所有完整 plan 都一样，比较完整 plan 时会抵消；可是在 open list 中，goal vertex 的 \(g\) 已经包含了这个最终结果基数，而 non-goal vertex 的 \(g\) 还没有包含。若 heuristic 经常低估，goal 会被不自然地推迟弹出。

论文的修正（Def. 4）：若一次 join 的结果已经是完整 \(R\)，则该 edge weight 记为 0；否则仍记为 \(|S_1\Join S_2|\)。这样不改变 \(C_{out}\) 下的最优 plan，因为所有完整 plan 只差一个共同常数，却能让 bottom-up search 更快收敛到 goal。 
C(S1, S2, j):
    if j(S1, S2) == R:
        return 0
    return cardinality(S1 ⋈ S2)

论文说明：这个 pitfall 对 top-down search 不存在，因为 top-down 从完整 \(R\) 出发，结果基数会在初始扩展时被立即纳入所有候选。

6.2 四个简单 heuristic

Heuristic方向定义/直觉最优性
\(h_{zero}\) Bottom-up / Top-down 恒为 0。A* 退化为 Dijkstra。 admissible，能求 optimal plan。
\(h_{sum}\) Top-down 把当前 vertex 中所有非单表 subproblems 的 cardinality 相加，作为未来 partition 还必须支付的一部分下界。 admissible，能求 optimal plan。
\(h_{GOO\uparrow}\) Bottom-up 从当前 vertex 开始跑 Greedy Operator Ordering：每次 join 产生最小结果的两个 subproblems，用这条实际路径估计剩余代价。 可能 overestimate，不保证 optimal。
\(h_{GOO\downarrow}\) Top-down top-down 版本的 GOO 式贪心 partition，论文描述为选择使子问题 cardinality 和较小的 partition。 可能 overestimate，不保证 optimal。

\(h_{sum}\) 的公式是：

\[ h_{sum}(v)=\sum_{S\in v\setminus R}|S| \]

这里 \(v\setminus R\) 表示当前 vertex 中那些不是 base relation 的 subproblem。因为这些 subproblem 在 top-down 搜索中还要继续被 partition，它们的 cardinality 至少会作为 \(C_{out}\) 的一部分被计入，所以这是一个下界。

6.3 信息量：effective branching factor

论文用 effective branching factor \(b^*\) 衡量 heuristic 的信息量。给定 goal depth \(d=|R|-1\) 和 generated vertices 数 \(N\)，通过下式反推 \(b^*\)：

\[ N=b^*+(b^*)^2+\cdots+(b^*)^d \]

\(b^*\) 越接近 1，说明搜索越像沿着一条线直奔 goal；\(b^*\) 越大，说明 heuristic 越不确定，搜索分叉越多。

论文 Figure 5 的核心观察：不同 topology 适合不同方向和 heuristic。\(h_{GOO}\) 往往最能把搜索拉向 goal，但牺牲最优性；\(h_{sum}\) 对 top-down 很有帮助；bottom-up 的 \(h_{zero}\) 在 star / clique 上反而很强。

7. 实验结论怎么读

7.1 与 state of the art 比较

论文在 mutable 这个 main-memory DBMS 中实现 heuristic search，并与 DPCCP、TDMinCutAGaT、LinearizedDP、GOO 等算法比较。实验使用 \(C_{out}\)，并通过 cardinality file 给优化器精确 cardinality，以便控制选择率和 join selectivity。

实验对象主要结论
Optimal algorithms chain / cycle 上，最快的 heuristic search 是 \(A^*_\downarrow+h_{sum}\)，但 DPCCP 与 TDMinCutAGaT 仍更强；star / clique 上，\(A^*_\uparrow+h_{zero}\) 最快，约比 DPCCP 或 TDMinCutAGaT 快 10 倍。
Suboptimal algorithms GOO 最快，heuristic search 最慢，LinearizedDP 居中；但 \(A^*_\uparrow+h_{GOO\uparrow}\) 通常产生质量最好的 suboptimal plans，几乎总是优于 GOO。
Plan quality LinearizedDP 在 star / clique 上可能产生成本很高的 plan。论文认为原因与其基于 IK/KBZ 的 greedy linearization 有关，而实验数据生成允许 correlated selectivities，不满足 IK/KBZ 假设。

7.2 QGraEL benchmark

论文还提出 QGraEL（Query Graph Exploration Landscape）基准，用三维探索 query graph：

  • number of relations：关系数量；
  • density：实际 join edges 占最大可能 edges 的比例；
  • edge skew：degree distribution 的偏斜程度，用 \(\chi^2\) test 的 \(p\)-value 表示；\(p\)-value 越低表示越偏离均匀分布，也就是 skew 越高。cycle / clique 的 degree 更均匀，star 的 skew 更高。

论文指出，TPC-H 和 JOB 的查询图大多位于低 density、高 edge skew 的区域，留下大片传统 benchmark 不覆盖的空间。QGraEL 在这些区域比较 DPCCP 与 \(A^*_\uparrow+h_{zero}\)，发现不少未充分探索的 graph 区域由 heuristic search 明显占优，最高有超过 1000 倍的加速。

7.3 Profile：时间花在哪里

论文用 Linux perf profile 两个代表性场景：chain-40 上的 \(A^*_\downarrow+h_{sum}\)，以及 star-22 上的 \(A^*_\uparrow+h_{zero}\)。结论是：

  • expand 函数出现在调用栈中很显眼，但真正花在 expand 内部的时间只是一小部分。
  • chain-40 上，较多时间花在 heuristic evaluation 和 heap extract。
  • star-22 上，超过一半相对时间花在 cardinality estimation；论文解释这不是因为 star 上估计更慢，而是 open list 管理开销更少。
  • duplicate 会导致重复计算 heuristic，从而拖慢搜索。

7.4 Pareto frontier

最终，论文把结论放到 optimization time vs. plan cost 的 Pareto frontier 上：optimal 的 \(A^*_\uparrow+h_{zero}\) 在 star / clique 上能明显快于 state of the art；suboptimal 的 \(A^*_\uparrow+h_{GOO\uparrow}\) 位于 GOO 与 optimal search 之间，提供“比 GOO 贵一点但计划质量更稳”的中间选择。

所以这篇论文的实践价值不是一句“A* 比 DP 快”，而是：它扩展了优化时间与 plan cost 之间的可选点。优化器可以把 heuristic search 当作一类新的 graceful degradation 策略。

8. 与 DP / top-down / IKKBZ 的关系

8.1 与 DPCCP

DPCCP 的核心目标是 duplicate-free 地枚举所有 ccp，以保证 bushy join tree 的最优性。heuristic search 则不是按“必须枚举完所有 ccp”的节奏走，而是按 open list 的 \(g+h\) 顺序走。

维度DPCCPHeuristic search
控制流 系统枚举所有 relevant ccp 按当前最有希望的 vertex 展开
最优性来源 完整 DP 子问题覆盖 shortest path + admissible heuristic + 正确 open list
优势场景 chain / cycle 等传统枚举可控场景仍很强 star / clique 及 QGraEL 中部分高收益区域
风险 必须做完整枚举，难以提前收敛 heuristic 不够 informative 时会退化，duplicates 需要认真处理

8.2 与 top-down join enumeration

DeHaan & Tompa、Fender & Moerkotte 一类 top-down enumeration 把关系集合递归 partition 成左右子问题。论文直接复用了这个思想来实现 top-down vertex expansion：从当前 subproblem 中枚举 ccp partitions。

差别是：传统 top-down enumeration 通常仍然是枚举算法；这篇论文中的 top-down search 是 A* / Dijkstra 的一个方向，具体 partition 只是 expandTopDown 的后继生成器。

可以把它理解为：top-down enumeration 给了“如何从一个 vertex 找后继”的局部算子；heuristic search 给了“下一步应该展开哪个 vertex”的全局调度策略。

8.3 与 IKKBZ / ASI

IKKBZ 依赖 ASI 性质和 rank 函数，在树查询及特定代价模型下给出多项式精确算法。这篇论文显式说：heuristic search 不要求 ASI。它只要求你能给每条 edge 算权重，并给 A* 一个合适的 heuristic。

不过两者在工程哲学上有关联：

  • IKKBZ / LinearizedDP 可用来做 search space reduction，但若 selectivity 相关性不满足假设，可能锁死很多好 plan。
  • GOO heuristic 与 IKKBZ 都体现了“先做看起来能显著缩小中间结果的 join”的直觉，但 GOO 在这里是启发函数，不是完整枚举器。
  • 对本仓库 demo 来说，IKKBZ 是“结构和代价满足强性质时如何精确求解”，这篇论文是“没有强性质时如何用搜索控制指数空间”。

8.4 对 demo / 学习路线的启发

如果把这篇论文放进 join order 学习路线，可以这样串起来：

  1. System R / DPsize：理解 join ordering 为什么能用 DP。
  2. DPsub / DPCCP：理解如何减少无效枚举和 duplicate。
  3. Top-down partitioning：理解如何从完整关系集合反向切分。
  4. IKKBZ / SWSP：理解在 ASI + 树结构下，为什么 rank 能给出多项式最优算法。
  5. 本文：理解把 JOOP 变成 shortest path 后，如何用 A* 的 \(g+h\) 控制搜索顺序，并用 admissible / inadmissible heuristic 在最优性与速度之间切换。
对 demo 实现而言，最适合先做的不是完整复刻论文，而是实现一个小型 SPJOOP 可视化：vertex 用 bitset partition 表示，支持 Dijkstra / A*，展示 DeDup 和 PreDup 打开/关闭时 generated vertices 的变化。这能把论文最核心的“搜索控制流”讲清楚。

9. 读完应记住的结论

结论一句话解释
JOOP 可以严格规约成 shortest path vertex 是 disjoint subproblems 的集合，edge 是一次 join 或一次 partition，path 是 join plan，path weight 是 plan cost。
规约不消灭指数复杂度 SPJOOP 最坏仍有指数多 vertex / edge；价值在于能按需展开并使用 heuristic search。
A* 的最优性取决于 heuristic admissible heuristic 保证 optimal plan；GOO 这类可能高估的 heuristic 更快但可能 suboptimal。
DeDup 是必要工程部件 没有 duplicate handling，open list 中的重复 vertex 会被反复 expand，引发重复后继的连锁膨胀。
PreDup 利用 canonical order 消掉无意义排列 独立 join 的顺序交换不改变 partial plan 与 cost；用最近 join 结果的字典序标记可以提前剪掉 redundant paths。
Cost model 细节会影响搜索效率 在 \(C_{out}\) 下，最终结果基数是共同常数；bottom-up edge weight 要避免把 goal 不自然地推迟弹出。
不同 query topology 适合不同方向 论文实验中 chain / cycle 偏向 top-down + \(h_{sum}\)，star / clique 偏向 bottom-up + \(h_{zero}\)。
本文提供的是新的 Pareto 选择 optimal search 在某些拓扑上大幅快于 state of the art；suboptimal search 提供 GOO 与 optimal search 之间的中间点。
