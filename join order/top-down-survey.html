1. 问题背景

一个 SPJ 查询块可以抽象成 join graph \(G=(V,E)\)：顶点是表，边是 join predicate。优化器要做的是在所有等价 join tree 中找到代价最低的计划。

概念含义为什么重要
CP-free 每个 join 的左右两侧在 join graph 中必须连通，避免 Cartesian Product。 真实优化器通常默认避免 CP；难点也集中在如何只枚举有效划分。
Interesting order / property 父算子对子计划的物理属性需求，例如排序、分布、分区方式。 这是 top-down search 的天然入口：从根需求反推子需求。
Naive transformation rules 对当前表达式逐条尝试交换律、结合律等 transformation rule；只要 pattern 能匹配，就生成新的等价表达式。 这种做法很通用，但不会提前判断不同规则路径是否会到达同一个表达式，因此后文会出现大量 duplicate。

[Ono & Lohman '90][2] 给出了不同搜索空间的 join operator 数量下界；后来的 [Moerkotte & Neumann '06][7] 又把“枚举算法最优”定义为：算法实际工作量应与必须枚举的 join operator 数量同阶，中间不能靠大量 generate-and-test 浪费时间。

这篇文章后面所有“最优”都指[Moerkotte & Neumann '06]中提的最优：枚举过程没有额外指数级浪费。

2. 为什么需要 top-down search

从历史上看，top-down search 不是凭空出现的，它是 bottom-up DP 暴露出几个工程问题后自然长出来的。

2.1 System R / DP 的优点与局限

[Selinger et al. '79][1] 的核心思路是 bottom-up dynamic programming：先枚举单表计划，再枚举两表计划、三表计划，直到得到完整 join tree。这个思想非常稳定，今天依然是很多 join 枚举算法的基础。

但 bottom-up DP 有几个不舒服的点：

  • 它先生成子问题，再考虑父需求。 如果最终计划只需要某些 physical property，bottom-up 往往还是要预先维护大量可能的 interesting orders。
  • 剪枝不自然。 DP 表是按大小逐层填满的，想用一个已知上界提前停止某些分支，比 top-down 递归困难。
  • CP-free bushy 枚举很容易退化成 generate-and-test。 枚举所有子集再检查连通性，会产生大量无效候选。
  • 优化器扩展成本高。 新算子、新属性、新 rewrite 通常要求改动枚举器本身。

2.2 Top-down search 的基本想法

Top-down search 可以理解成一个提问顺序的变化。

Bottom-up DP 的问题是从最小子表达式开始，一层层把所有可能的中间结果都算出来。Top-down search 则从最终目标开始问：

  • 根表达式需要满足什么物理属性？例如是否要求有序、分布方式是什么。
  • 有哪些候选方式可以实现这个根表达式？例如先做哪一侧、使用什么 join 算子、是否需要 enforcer。
  • 每个候选方式会对子表达式提出什么要求？例如左输入必须有序，右输入可以无序。
  • 这些子表达式是否已经被优化过？如果优化过，就直接复用 memo 里的结果。

换句话说，top-down search 不是“先把所有小计划都造出来”，而是“从最终计划的需求出发，只追问当前候选计划真正需要的子问题”。这使得搜索过程天然带有需求驱动（demand-driven）的味道。

这里要先区分两个层次：Volcano/Cascades 中的 top-down search 最早不是专门为 join order search 设计的，而是整个查询优化器选择物理执行计划的组织方式。Join order 只是优化器搜索空间中的一部分；同一个框架还要处理逻辑改写、物理算子选择、required property、enforcer 插入和代价剪枝。后文的 [DeHaan & Tompa '07][8] 才是把 top-down 思路专门落到 join-order enumeration 上。
问题Top-down search 的回答
为什么从根开始？ 因为根节点携带最终 required property，子问题应该由父需求推导出来。
为什么需要 memo？ 不同候选计划可能到达同一个子表达式；memo 让这些子问题只优化一次。
为什么容易剪枝？ 一旦已有完整计划作为上界，递归分支的下界若不可能更优，就可以停止探索。
因此，top-down search 真正关心的不只是“从根往叶递归”这个控制流，而是三件事：memoization、property-driven optimization、以及 branch-and-bound。

3. 第一条路线：Volcano/Cascades

Top-Down Transformation Rules Memo

[Graefe & McKenna '93][3] 和 [Graefe '95][4] 是 top-down optimizer framework 的经典出处。这里的 top-down 指的是优化器如何从目标表达式和 required property 出发组织搜索，而不是某个 join-order 专用枚举算法。它们把优化器做成一个可扩展框架：逻辑等价变换、逻辑到物理实现、物理属性要求，都由规则系统表达。

3.1 Volcano 带来的抽象

抽象作用
Memo / Group 一个 group 存放逻辑上等价的表达式；memo 避免重复优化同一个逻辑子问题。
Transformation rule 产生逻辑等价表达式，例如 join 交换律、结合律。
Implementation rule 把逻辑表达式映射成物理算子，例如 logical join 到 hash join / nested-loop join。
Enforcer 插入 sort、exchange 等算子来满足 required physical property。

Volcano 的搜索通常被描述成 two-pass：先用 transformation rules 生成逻辑等价空间，再做物理实现和代价选择。Cascades 的改进在于把两阶段解耦，按需探索 memo，而不是先完整生成所有逻辑表达式。

3.2 变换式搜索的核心问题：duplicate

关于duplicate问题的详细讨论，可以参考[Pellenkoft et al. '97][6]

用规则探索 join order 时，一个表达式可以通过很多条规则序列到达。例如四表 join 中，交换律和结合律的不同组合可能反复生成同一个 join tree。[Pellenkoft et al. '97][6] 证明，naive transformation rules 会让枚举复杂度从 memo 中唯一表达式的 \(O(3^n)\) 膨胀到 \(O(4^n)\)。

这里的 duplicate 指的是：某次 rule application 生成了一个 operator，但这个 operator 在 memo-structure 中已经存在，因此插入 memo 时会被检测出来并丢弃。换句话说，duplicate 不是“最终 memo 里存了两份相同内容”，而是“搜索过程中不同的 transformation rule 序列推导出了同一个 memo operator”。

例如，若 memo 中已经有 \([ab]\bowtie[c]\)，后续又从 \([c]\bowtie[ab]\) 应用 commutativity 生成 \([ab]\bowtie[c]\)，这次生成就是 duplicate。它不会增加最终 memo 大小，但生成、查重和丢弃本身都会消耗搜索时间。

3.2.1 枚举复杂度

先看 memo 里真正需要保存多少唯一 join operator。对一个完全连接的 \(n\) 表查询，任意非空关系集合 \(S\) 都对应一个 equivalence class。这里的 equivalence class 指一组逻辑上等价、产生同一个中间结果的表达式；在 join 枚举中，可以粗略理解为“同一个关系集合 \(S\) 的所有生成方式”。

设总共有 \(n\) 张表。对任意一个大小为 \(k\) 的关系集合 \(S\)，也就是 \(|S|=k\)，一个 join operator 本质上就是把 \(S\) 拆成左右两个非空子集：

\[ [L] \bowtie [R], \quad L \cup R = S,\quad L \cap R = \emptyset,\quad L \ne \emptyset,\quad R \ne \emptyset \]

关键是：这里把 \(L \bowtie R\) 和 \(R \bowtie L\) 当成两个 ordered operator，所以只要选择左边 \(L\)，右边 \(R=S-L\) 就确定了。

对大小为 \(k\) 的集合 \(S\)：\(S\) 的所有子集数量是 \(2^k\)；不能选空集作为左边，所以减 1；也不能选全集 \(S\) 作为左边，因为右边会为空，所以再减 1。因此一个大小为 \(k\) 的 equivalence class 里有 \(2^k-2\) 个 ordered join operator。

接着，大小为 \(k\) 的关系集合 \(S\) 有多少个？从 \(n\) 张表里选 \(k\) 张，共有 \(C_{n}^{k}\) 个。所以所有大小为 \(k\) 的 class 贡献：

\[ C_{n}^{k}(2^k-2) \]

再对所有 \(k \ge 2\) 求和，因为单表没有 join operator：

\[ \sum_{k=2}^{n} C_{n}^{k}(2^k-2) \]

这个求和式可以继续化简：

\[ \begin{aligned} \sum_{k=2}^{n} C_{n}^{k}(2^k-2) &= \sum_{k=2}^{n} C_{n}^{k}2^k - 2\sum_{k=2}^{n} C_{n}^{k} \\ &= (3^n - 1 - 2n) - 2(2^n - 1 - n) \\ &= 3^n - 2^{n+1} + 1 \\ &= O(3^n) \end{aligned} \]

这就是 memo 压缩后的空间规模：所有 bushy join tree 很多，但它们共享子表达式后，只需要 \(O(3^n)\) 个 operator 表示。

混淆点：直觉上容易把 \([ABC]\bowtie[D]\) 计成“\([ABC]\) 内部 12 种结构 × \([D]\) 的 1 种结构”。但这个 \(12 \times 1\) 数的是完整 join tree 展开后的组合，不是 memo 中的 root join operator 数量。memo operator 只把 \([ABC]\bowtie[D]\) 看作一个根部 operator，其中 \([ABC]\) 是一个 group 引用；\([ABC]\) 内部的 alternatives 会在自己的 group 中单独枚举，cost 阶段再按 required property 记录 best plan。

问题出在 naive rule exploration 的“生成过程”上。[Pellenkoft et al. '97][6] 分析的最小 naive 规则集 RS-B1 只保留一个方向的 associativity 加 commutativity。考虑一个包含 \(k\) 个关系的 class，取其中一个 operator \([L]\bowtie[R]\)，其中 \(|L|=\ell\)。对这个 operator：

  • commutativity 会生成 1 个候选：\([R]\bowtie[L]\)；
  • associativity 需要从左输入 \([L]\) 中抽出具体 partial tree。\(L\) 的有序二分有 \(2^\ell-2\) 种，因此会生成 \(2^\ell-2\) 个候选；
  • 所以这个 operator 会尝试生成 \(2^\ell-1\) 个候选。

把同一个 \(k\)-relation class 中所有可能的左侧大小 \(\ell\) 加起来：

\[ \sum_{\ell=1}^{k-1} C_{k}^{\ell}(2^\ell-1) = (3^k - 1 - 2^k) - (2^k - 2) = 3^k - 2^{k+1} + 1 \]

这是这个 class 在探索过程中“尝试生成”的候选数。但这个 class 中真正唯一的 operator 只有 \(2^k-2\) 个，而且其中一个 initial operator 是给定的，不是通过规则生成的。因此该 class 里产生的重复数为：

\[ (3^k - 2^{k+1} + 1) - ((2^k-2)-1) = 3^k - 3 \cdot 2^k + 4 \]

最后再对所有大小为 \(k\) 的 class 求和。大小为 \(k\) 的 class 有 \(C_{n}^{k}\) 个：

\[ \sum_{k=2}^{n} C_{n}^{k}(3^k - 3 \cdot 2^k + 4) = 4^n - 3^{n+1} + 2^{n+2} - n - 2 = O(4^n) \]
直觉上，memo 只需要 \(O(3^n)\)，但 naive transformation search 在“走图”。同一个 memo operator 可能从许多规则路径到达。路径数由“对每个 class 尝试从 child class 中抽 partial tree 再变换”放大，最后主导项变成 \(4^n\)。所以问题不是存不下，而是生成和查重太浪费。

把上面的计数对象整理一下：

计数对象公式含义
一个 \(k\)-relation class 中的唯一 root operator \(2^k - 2\) 把 \(S\) 拆成 ordered 的左右非空子集 \([L]\bowtie[R]\)。
所有大小为 \(k\) 的 class 对唯一 operator 的贡献 \(C_{n}^{k}(2^k - 2)\) 从 \(n\) 张表中选择大小为 \(k\) 的集合 \(S\)，再枚举其有序二分。
整个 memo 中的唯一 operator 数 \(\sum_{k=2}^{n} C_{n}^{k}(2^k-2)=3^n-2^{n+1}+1\) memo 压缩后的空间规模，主项为 \(O(3^n)\)。
对单个 \([L]\bowtie[R]\) 应用 RS-B1 生成的候选 \(2^{\ell}-1,\quad \ell=|L|\) commutativity 生成 1 个，associativity 从 \([L]\) 内部二分生成 \(2^\ell-2\) 个。
探索一个 \(k\)-relation class 时生成的候选总数 \(\sum_{\ell=1}^{k-1} C_{k}^{\ell}(2^\ell-1)=3^k-2^{k+1}+1\) 包含新 operator 和 duplicate，是 rule application 的尝试次数。
探索一个 \(k\)-relation class 时生成的 duplicate 数 \((3^k-2^{k+1}+1)-((2^k-2)-1)=3^k-3\cdot2^k+4\) 生成总数减去“需要由 rule 生成的唯一 operator”。初始 operator 是给定的，不算 rule 生成。
构造整个 memo 时的 duplicate 总数 \(\sum_{k=2}^{n} C_{n}^{k}(3^k-3\cdot2^k+4)=4^n-3^{n+1}+2^{n+2}-n-2\) naive transformation search 的额外浪费，主项为 \(O(4^n)\)。

线性树的重复推导要单独看，因为原论文 3.2 使用的是 RS-L1：swap 加 bottom commutativity。这里的计数口径是 Section 3.2 duplicate proof 中的 ordered operator 口径，和 Figure 2 / Theorem 2 的 compact left-linear operator 口径不同。

计数对象公式解释

3.2.2 Derivation history 怎么消掉重复

他们的解决办法是 derivation history：给每个 join operator 记录少量规则应用历史，避免沿着会产生重复的路径继续派生。这样在 bushy with CP 空间中可以达到 \(O(3^n)\) 级别。

最简单的例子是 commutativity：如果 \(B\bowtie A\) 正是从 \(A\bowtie B\) 交换得到的，那就没有必要再对 \(B\bowtie A\) 应用 commutativity，否则立刻回到 \(A\bowtie B\)。[Pellenkoft et al. '96][5][Pellenkoft et al. '97][6] 的方案把这个想法推广到 associativity / exchange 等规则：每个 operator 带一小段 rule applicability bit，表示哪些规则继续应用仍可能产生新东西。论文中的 bushy 规则集需要 4 个 bit。

这等价于给每个表达式选出一条 canonical derivation path：以前同一个 operator 可能从多条路径抵达；现在只保留一条路径，其余路径在规则级别就被关掉。

3.2.3 为什么 CP-free cyclic query 会漏计划

关键点：derivation history 为了不重复，会给每个目标表达式只保留一条 canonical 派生路径。若这条路径中间经过了包含 CP 的表达式，而优化器又用 generate-and-test 提前丢弃 CP，那么最终那个 CP-free 计划就再也没有第二条路径可以到达。

[Pellenkoft et al. '96][5][Pellenkoft et al. '97][6] 的 duplicate-free 方法对“包含 CP 的完整 bushy 空间”成立；他们另有针对 acyclic query graph 的机制，可以保证 CP-free plan 的唯一派生路径本身也 CP-free。[DeHaan & Tompa '07][8] 后来指出，这个性质不能推广到任意 cyclic query graph：某些简单 cyclic query 中，至少一个 bushy CP-free plan 的唯一派生路径必须穿过一个含 CP 的中间 plan。

可以用下面这个简化构造理解“为什么会穿过 CP”（它是机制示意，不是原论文图例）。假设当前 class 的 initial split 是：

L = {a,b,c},  R = {d,e,f}
initial: [abc] ⋈ [def]

join graph 中有这些边：

a-b, b-c          -- L 内部连通
d-e, e-f          -- R 内部连通
a-d, c-d, b-e     -- 跨 L/R 的连接边

于是目标 bushy plan 的根划分可以是：

target: [acd] ⋈ [bef]

它是 CP-free 的：\([acd]\) 可以通过 \(a-d\)、\(c-d\) 连起来，\([bef]\) 可以通过 \(b-e\)、\(e-f\) 连起来，根部两侧也有边相连。

但在 [Pellenkoft et al. '96][5][Pellenkoft et al. '97][6] 的 duplicate-free derivation 中，这种“左右两边都混合了原来 \(L\) 和 \(R\) 的关系”的划分通常由 exchange 类规则生成。相对于 initial split：

[abc] ⋈ [def]
  L 被切成 {a,c} 和 {b}
  R 被切成 {d} 和 {e,f}
  exchange 后得到 {a,c,d} ⋈ {b,e,f}

问题就在 \(L\) 的内部切分：\(\{a,c\}\) 本身没有 join predicate，是一个 disconnected subset。也就是说，canonical path 中需要先出现一个包含 \([a,c]\) 的中间 operator，而 \([a,c]\) 只能通过 Cartesian Product 得到。如果优化器为了 CP-free 搜索在这里把 \([a,c]\) 丢掉，后面的 exchange 就没有材料生成 \([acd]\bowtie[bef]\)。最终目标明明 CP-free，却被漏掉了。

这不是说所有 transformation search 都必然无法处理 CP-free cyclic graph；更准确地说，是 [Pellenkoft et al. '96][5][Pellenkoft et al. '97][6] 这种“用 derivation history 选唯一派生路径，然后用 generate-and-test 丢 CP”的组合不完整。[DeHaan & Tompa '07][8] 的转向就是避开规则派生路径，直接枚举 CP-free partition。

3.3 Volcano/Cascades 在主线中的位置

Volcano/Cascades 解决的是“如何构造一个可扩展、property-driven 的 top-down optimizer”。但如果问题缩小到 join-order enumeration，尤其是 任意 join graph 上的 CP-free bushy 枚举，单靠 transformation rules 会遇到重复和完整性问题。

所以后面的关键转向是：保留 top-down + memo 的好处，但不用 transformation rules 生成 join order，而是直接把目标关系集合递归划分成两个连通子图。

4. 第二条路线：Top-Down Partitioning

DeHaan & Tompa 2007 Partitioning Minimal Cut

4.1 从“变换表达式”改成“划分集合”

[DeHaan & Tompa '07][8] 的 top-down partitioning search 可以写成一个很直接的递归：

BestPlan(G, required_property):
  if memo[G, required_property] exists:
    return memo[G, required_property]

  if |V(G)| == 1:
    return best scan satisfying required_property

  best = infinity
  for each valid partition (G_left, G_right) in Partition(G):
    for each physical join operator that can satisfy required_property:
      left_property, right_property = derive child properties
      left_plan  = BestPlan(G_left, left_property)
      right_plan = BestPlan(G_right, right_property)
      best = min(best, join(left_plan, right_plan))

  memo[G, required_property] = best
  return best

这里最关键的抽象就是 Partition(G)。搜索空间由它决定：

搜索空间Partition(G) 怎么做难点
Left-deep with CP 每次从 \(V\) 中拿出一个基表作为右子。 简单。
Bushy with CP 枚举 \(V\) 的所有非空真子集。 数量大，但没有连通性约束。
Bushy CP-free 只输出左右两边都连通的划分。 不能枚举所有子集再检查，否则会指数级浪费。

4.2 Naive partition 为什么不够

如果对 bushy CP-free 空间使用 naive generate-and-test：

for each non-empty proper subset S of V:
  if G[S] is connected and G[V-S] is connected:
    output (S, V-S)

这在 clique 上还可以接受，因为几乎每个子集都有效；但在 chain、tree、star 这类真实查询常见拓扑上，大量子集是不连通的。算法时间被无效候选主导，因而不是 enumeration optimal。

4.3 Minimal cut：把 CP-free 划分变成图论问题

核心观察是：在连通图 \(G\) 中，把顶点划分成两个连通诱导子图，等价于找到一个 minimal cut。删除这个 cut 后，图正好分成两个连通分量；这两个分量就是一个 CP-free bushy join 的左右输入。

这一步是论文的关键：它不再从所有子集里筛有效划分，而是直接生成有效划分。

4.4 MincutLazy：为什么它能最优

通用 minimal-cut 枚举算法仍可能开销过大。[DeHaan & Tompa '07][8] 进一步利用 join graph 的特点：真实查询通常不是 clique，而是低环形度、接近 tree/chain/star 的稀疏图。因此他们用 biconnection tree 辅助枚举 minimal cut，并尽可能复用父递归节点的树结构。

MincutLazy(S, T, old_bcc_tree):
  if S is not empty:
    output partition (S, V-S)

  if S cannot be expanded:
    return

  if old_bcc_tree is still usable after removing S:
    tree = old_bcc_tree
  else:
    tree = BuildBiconnectionTree(G[V-S])

  for each pivot v selected from the tree:
    MincutLazy(S expanded by v's descendant region, updated_T, tree)

它的要点不是某个实现细节，而是 lazy tree building：构建 biconnection tree 是 \(O(|E|)\) 的昂贵操作，只要父节点的树还能用，就不重建。

图形态MincutLazy 行为直觉
Tree / chain / star 每个 cut 摊销接近 \(O(1)\)。 图结构变化局部，父 biconnection tree 大量可复用。
低环形度真实查询 实验中显著快于 eager 重建。 真实 join graph 往往稀疏，不像 clique 那样处处强连通。
Clique 每个 cut 可能要 \(O(|E|)\)。 但 clique 中有效划分本来就是指数多，这个开销仍与下界相容。

把 MincutLazy 接到 top-down partitioning 框架上，就得到 TBNmc：Top-down Bushy No-CP with Minimal Cuts。它是第一个在任意 join graph 上达到 enumeration optimal 的 top-down CP-free bushy join 枚举算法。

TBNmc 的意义不是“比 DPccp 穷举更快很多”。论文实验表明，在纯穷举情况下，它和 bottom-up DPccp/BBNccp 性能接近。它真正有价值的地方在于：保留了 top-down 框架对 property、剪枝、缓存和多阶段优化的友好性。

5. Branch-and-Bound：top-down 的真正优势

Top-down search 从根需求开始递归，因此很适合携带“当前已知最好完整计划”的上界。如果某个分支的下界已经不可能打败上界，就可以停止探索。这就是 branch-and-bound。

5.1 两类 bound

策略剪枝条件问题
Accumulated-cost bounding 从根到当前节点已经累积的实际代价超过上界。 会把“预算”变成子问题上下文，破坏 memoization。
Predicted-cost bounding 基于逻辑属性预测该子表达式的最低可能代价，若已超过上界则剪枝。 依赖下界估计质量，但与 memoization 更相容。

5.2 Accumulated-cost 与 memoization 的冲突

[DeHaan & Tompa '07][8] 一个很重要的发现是：accumulated-cost bounding 在带 memo 的 divide-and-conquer top-down search 中可能变慢。

同一个逻辑子图 \(G[S]\) 如果在“小预算”下优化失败，memo 里不能简单记成“无解”。之后另一路径可能带着更大的预算再次到达 \(G[S]\)，于是整个子树要重新优化。预算成为上下文后，memo key 不再只是 \((S, property)\)，复用能力被破坏。

论文实验中，TBNmcA（只用 accumulated-cost）虽然减少了存储计划数量，但 CPU 时间反而比穷举 TBNmc 慢 10 到 100 倍；TBNmcP（predicted-cost）则能显著减少搜索时间。

实践结论：在 memoized top-down search 中，优先使用 predicted-cost bounding。Accumulated-cost bounding 看起来直观，但很容易把递归子问题变成“带预算的子问题”，从而制造重复优化。

6. 放在一起看：各算法在解决什么

阶段代表工作核心问题留下的问题
Cost-based DP 起点 [Selinger et al. '79][1] 用 bottom-up DP 搜索 left-deep join order。 搜索空间窄；property-driven 与剪枝不自然。
复杂度下界 [Ono & Lohman '90][2] 说明不同 join graph/topology 下必须枚举多少 join operator。 只给下界，不给具体最优算法。
变换式 top-down 框架 [Graefe & McKenna '93][3], [Graefe '95][4] 用 memo + rules 做 extensible、property-driven optimizer。 join-order transformation 容易重复；CP-free 完整枚举困难。
变换式复杂度修复 [Pellenkoft et al. '97][6] 用 derivation history 把 naive rule 重复从 \(O(4^n)\) 降到 \(O(3^n)\)。 对 CP-free bushy 空间不完整。
Bottom-up CP-free 最优枚举 [Moerkotte & Neumann '06][7], [Neumann & Moerkotte '09][9] 直接枚举 connected-subgraph complement pairs，避免 generate-and-test。 仍是 bottom-up；top-down 的 property 与 B&B 优势不明显。
Top-down CP-free 最优枚举 [DeHaan & Tompa '07][8] TBNmc 用 minimal cut 直接生成 CP-free partition，实现任意图最优 top-down 枚举。 工程实现比普通 DP 复杂；剪枝质量依赖 cost lower bound。

6.1 三种方法的心智模型

方法一句话理解适合强调的能力
Bottom-up DP 从单表开始，把所有小集合的最优计划拼成大集合。 穷举简单、结构稳定、容易证明。
Transformational search 从目标表达式出发，用规则探索等价表达式和物理实现。 优化器框架可扩展、property-driven、适合 rewrite + implementation 统一管理。
Top-down partitioning 从目标关系集合出发，递归划分成两个可 join 的连通子图。 保留 top-down 好处，同时在 CP-free join order 上达到最优枚举。

7. 读完应记住的结论

  1. Top-down search 首先是一种优化器控制结构。 它从根表达式和 required property 出发，按需生成子问题，并用 memo 复用结果。
  2. Volcano/Cascades 是 top-down optimizer framework 的出处。 它们用 memo、group、rules、enforcer 把逻辑优化和物理优化组织起来。
  3. Transformational search 不等于 join-order 枚举的最终答案。 它可扩展，但 naive rules 有重复问题；[Pellenkoft et al. '96][5][Pellenkoft et al. '97][6] 的修复对 CP-free bushy 空间仍有限制。
  4. [DeHaan & Tompa '07][8] 的关键贡献是把 CP-free partition 规约为 minimal cut。 这样 top-down partitioning 不需要枚举所有子集再过滤，而是直接生成有效划分。
  5. TBNmc 的价值在于“最优枚举 + top-down 优势”。 穷举性能与 DPccp 接近，但更容易接入 required property、predicted-cost branch-and-bound、多阶段优化和缓存复用。
  6. Branch-and-bound 要小心 accumulated-cost。 在带 memo 的 top-down search 中，它可能破坏子问题复用；predicted-cost bounding 更安全。
如果只想抓主线，可以记成一句话：System R 证明了 cost-based DP 可行，Volcano/Cascades 把优化器变成 top-down rule engine，DPccp 解决了 bottom-up CP-free 最优枚举，[DeHaan & Tompa '07][8] 则用 minimal cut 把这个最优枚举能力带回 top-down search。
