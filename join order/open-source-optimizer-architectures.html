开源数据库优化器架构调研：从 System R 到 Cascades 与 Hypergraph

调研日期：2026-06-16｜范围：著名开源数据库、SQL 引擎和优化器框架｜资料优先级：官方文档、源码 README、官方博客、论文

这份文档关心的不是单个 join order 算法，而是“一个可维护的开源优化器通常怎么分层”： SQL AST 如何变成逻辑关系代数，规则如何组织，cost 和统计信息如何介入，join reorder 在哪里发生， physical property 如何表达，以及最终怎么落到物理计划。调研对象横跨 PostgreSQL、SQLite、DuckDB、 Apache Calcite、Apache DataFusion、Spark Catalyst、Trino、CockroachDB、TiDB、GPORCA、Apache Doris Nereids 和 MySQL hypergraph optimizer。

一句话结论：著名开源优化器大致分成四类：PostgreSQL/SQLite 式的工程化 System R 后代， Spark/DataFusion/DuckDB/Trino 式的规则流水线加局部 CBO，Calcite/Cockroach/ORCA/Doris 式的 memo/Cascades 架构，以及 DuckDB/MySQL 新一代 join optimizer 中更直接的 hypergraph/DP 枚举。真实系统往往混合使用这些路线。

一、架构谱系总览

System RPath / Joinrel

核心是按 relation set 自底向上产生候选路径，保留不同访问方法、join 方法和排序属性。PostgreSQL 是最典型的开源代表。

PipelineTree Rewrite

用一系列 analyzer/optimizer/planner rules 改写树，必要处嵌入 cost-based 阶段。Spark Catalyst、DataFusion、DuckDB、Trino 都有这个味道。

MemoCascades

把等价表达式放进 memo group，用规则探索逻辑等价和物理实现，并用 required/derived properties 剪枝。Calcite、ORCA、CockroachDB、Doris Nereids 属于这一类。

GraphHypergraph Join Search

把 join reorder 问题显式建模为 join graph/hypergraph 上的子集枚举，专注解决复杂 join 谓词和 bushy plans。DuckDB DPccp 和 MySQL Hypergraph 是代表。

三、PostgreSQL：RelOptInfo / Path 驱动的工程化 System R 后代

PostgreSQL 的优化器不是 Cascades memo，而是围绕 RelOptInfo 和 Path 组织搜索： base relation、join relation 都有自己的 RelOptInfo，候选访问路径和 join 路径作为 Path 挂在 relation 上。源码 README 明确区分 optimizer path stage 和 plan creation stage，前者产生路径、估算 cost， 后者把最便宜 path 转成 executor 能跑的 plan tree。[1]

这种架构的关键思想是：relation set 自底向上聚合。单表先产生 seq scan、index scan 等 paths； 两表 joinrel 再由两个较小 rel 的 path 组合产生 nested loop、merge join、hash join 等；更大 joinrel 继续递推。 PathKeys 保存排序属性，因此同一个 relation set 会保留多条 cost/property 不同的 path。

PostgreSQL 的 join search 还带有工程控制：普通查询用动态规划枚举 join relations；当 join 数太大时可以启用 GEQO 遗传算法，牺牲最优性换搜索时间。它代表了一类成熟系统的现实选择：不追求“理论上最优雅的 optimizer 框架”， 而是让 access path、join path、sort order、parameterized path、parallel path、FDW path 等执行器能力可以增量接入。

四、SQLite：极小内核里的 NGQP 与嵌套循环优化

SQLite 的执行模型以嵌套循环 join 为中心，优化器主要决定表访问顺序、每层循环用什么 index、是否能利用 ORDER BY/GROUP BY 等。 官方文档把 join reorder 描述为对 FROM 子句中 nested loops 顺序的选择，并说明 SQLite 可以用 polynomial-time graph algorithm 在几十路 join 上很快产生顺序。[2]

SQLite 的 Next Generation Query Planner 文档则把 join planning 显式画成图搜索问题，节点是 loop，边权是依赖于外层 loop 的代价；优化目标变成找一条低成本路径。[3] 这和大型 Cascades memo 不同：SQLite 保持极小状态和高可预测性，非常适合嵌入式场景。

借鉴点是：并不是所有优化器都需要 memo。只要执行器形态简单、join algorithm 主要是 nested loop，优化器可以用更窄的模型获得稳定表现。 代价是物理实现空间较小，复杂 bushy plan、分布式 property 和多种 join algorithm 的表达能力有限。

五、DuckDB：规则优化流水线 + DPccp Join Order

DuckDB 的官方 optimizer 博客把 optimizer 描述为一组逐步应用的优化规则：expression rewriter、filter pull-up/pushdown、 join order optimizer、top-N optimizer 等。官方文档也列出 DuckDB 有多个 optimizer 阶段。[4] [5]

对 join reorder 来说，DuckDB 著名之处在于使用 DPccp 这类 connected-subgraph DP 枚举方法，面向 join graph 直接枚举无交叉积的 connected partitions。它不像 Calcite/ORCA 那样把所有逻辑等价表达式都放入通用 memo， 而是在优化流水线中为 join order 放一个专门的高质量算法。

这种混合架构对分析型单机数据库很自然：大多数 rewrite 用简单 rule pipeline，join reorder 作为独立“重型 pass”处理。 它给本仓库 demo 的启发是：Volcano/Cascades skeleton 可以作为理解框架，但后续要研究 CP-free bushy search 时， 应该把 join graph partitioning 算法作为一个独立模块接入，而不是只依赖 commutativity/associativity 规则撞出搜索空间。

六、Apache Calcite：把优化器做成可嵌入框架

Calcite 的定位不是单一数据库，而是查询处理框架。官方文档强调它提供 SQL parser、validator、relational algebra、 metadata provider、planner rules 和 adapter 机制。它的 optimizer 核心围绕 RelNode、RelOptRule、 RelOptPlanner 和 trait system。[6] [7]

Calcite 同时提供 heuristic planner 和 VolcanoPlanner。后者是 cost-based optimizer，会在 equivalence classes 中探索不同 relational expressions；trait 则表达 convention、collation、distribution 等物理属性。 Adapter 可以把同一套关系代数映射到不同后端，因此 trait/convention 是 Calcite 架构的中心。

如果要设计研究 demo，Calcite 值得学的是边界：SQL parse/validate、relational algebra、metadata、rule、planner、adapter 都是独立层。即使 demo 很小，也应该避免让 SQL AST 直接污染 optimizer 内部，这一点和我们当前 demo 的 DuckDBSqlFrontend 边界是一致的。

七、Apache DataFusion：Rust 里的阶段化 Optimizer Pipeline

DataFusion 官方架构文档把查询执行分成 SQL/DF API、logical plan、logical optimization、physical planning、 physical optimization 和 execution。文档明确列出 optimizer 是一组规则，并提供自定义 optimizer rules。 [8] [9]

DataFusion 的架构重点不是通用 Cascades memo，而是清晰的 stage contract： analyzer 负责名字和类型相关改写，logical optimizer 做 projection/filter/simplification/join reorder 等逻辑层优化， physical planner 产生 ExecutionPlan，physical optimizer 再处理 ordering、coalesce、repartition 等物理层问题。

这类设计的好处是插件性和可读性强。对于研究项目，它提醒我们把“逻辑等价探索”和“物理属性修复”拆开记录： 一个 trace 只写最终 cost 不够，还要能看到每个阶段改了什么、为什么改。

八、Spark Catalyst：统一的 Tree Transformation 框架

Spark SQL 的 Catalyst 优化器论文把 Catalyst 描述为基于 Scala pattern matching 的可扩展 optimizer， 使用 tree transformation rules 处理 analysis、logical optimization、physical planning 和 code generation。 [10] Spark 源码中 Catalyst package 也以 expressions、plans、rules、analysis、optimizer、planner 等模块组织。 [11]

Catalyst 的优势是统一抽象：逻辑计划、表达式、物理计划都是 tree，规则可以用模式匹配重写子树。 它不是传统意义上“所有等价表达式进 memo”的 Cascades，但通过 rule batches 和 fixed-point execution 达到很强的工程扩展性。后来 Spark 也加入了统计信息和 cost-based join reorder，但其骨架仍然是 rule pipeline。

对分布式系统来说，Catalyst 后半段要处理 partitioning、ordering、exchange、whole-stage codegen 等物理约束。 这说明 optimizer architecture 不能只看 join order：物理计划阶段对数据分布和执行后端的建模同样决定最终性能。

九、Trino：分布式 SQL 引擎中的规则优化与数据移动成本

Trino 文档把 optimizer 暴露为一组可配置的优化能力，例如 join reordering strategy、join distribution type、 cost-based optimizations 和 table statistics。[12] 它的计划 IR 是 PlanNode 树，优化器由大量 plan optimizers/rules 迭代改写。

Trino 的特殊性在于它是 MPP 查询引擎：join order 不能只看 CPU/IO，还要看广播 join、partitioned join、exchange、 connector statistics、动态过滤等。换句话说，distribution property 是核心 cost 变量。

对 demo 的启发是：一旦研究对象从单机 join reorder 扩展到分布式 join reorder，RequiredProperty 至少要从 Sorted 扩成 Partitioned(keys)、SingleNode、Broadcast 或 HashDistributed(keys) 这样的属性，否则无法表达 exchange/enforcer 的真实成本。

十、CockroachDB：生产级 Cascades 与 Optgen

CockroachDB 官方文档介绍其 cost-based optimizer 会枚举等价 SQL 表达式，利用统计信息估算成本，并选择最低成本计划。 官方博客和源码文档进一步说明它借鉴 Cascades，使用 memo 保存等价表达式，并通过 optgen 生成表达式和规则相关代码。 [13] [14]

Cockroach 的架构重点是让 Cascades 可工程化：规则不是散落在手写 visitor 里，而是有 DSL/生成器； memo group 中保存 logically equivalent expressions；physical properties 会影响实现选择和 enforcer 插入； metadata/statistics 独立于表达式树。

它是研究 Cascades 生产化的好样本：比 Calcite 更接近一个完整数据库内核，比 ORCA 更容易从现代 Go 代码和文档中追踪实现。

十一、TiDB：MySQL-compatible 分布式数据库的 Planner 分层

TiDB developer guide 把 planner 切成逻辑优化和物理优化：逻辑阶段做列裁剪、谓词下推、聚合下推等， 物理阶段为不同 task type 和存储引擎选择访问路径、join 算法、是否下推到 TiKV/TiFlash。 [15]

TiDB 还有 Cascades planner 方向的实现和设计文档，试图用 memo、transformation rules、implementation rules 管理更复杂的搜索空间。[16] 这体现了许多成熟系统的演进路线：旧 planner 常常是 rule pipeline + CBO modules，随着规则和物理属性变复杂， 会引入 Cascades 或 Cascades-like 结构来控制复杂度。

TiDB 还有一个重要提醒：分布式数据库的 optimizer 不只是 SQL 层，它还必须知道数据在哪、哪些算子可下推、 不同存储引擎的 cost 结构如何、MPP 和 coprocessor 的边界在哪里。

十二、GPORCA：模块化 Cascades 优化器的经典样本

GPORCA 是 Greenplum 使用的开源 optimizer。ORCA 论文把它定位为 modular query optimizer architecture for big data， 使用 memo、transformation/implementation rules、metadata cache、required/derived properties 和 cost model。 [17] Greenplum 文档也介绍 ORCA 作为可选 optimizer，用于复杂 analytical queries。[18]

ORCA 的 property system 很值得研究：除了 ordering，还要处理 distribution、rewindability、partition propagation 等。 在 MPP 系统中，一个 join 的物理实现不仅是 hash/merge/nestloop，还包括是否需要 motion/exchange、数据是否已经按 key 分布。

如果本仓库的 Volcano demo 要继续往“大系统味道”走，ORCA 是最接近教科书 Cascades + MPP 现实约束的模板。

十三、Apache Doris Nereids：OLAP 系统迁移到 Cascades 的案例

Apache Doris 的 Nereids 是新一代 optimizer，官方设计文档说明它采用 Cascades-style framework， 以 memo、rule、cost、stats 和 property 组织优化流程。[19]

Nereids 的价值在于迁移案例：很多 OLAP 系统一开始有一个可用但逐渐复杂的旧 planner，随着物化视图、复杂 join、 分布式执行、统计信息和新算子加入，旧 planner 难以维护，于是引入 Cascades-like optimizer。

对研究者来说，Doris/Nereids 可以回答一个现实问题：不是“Cascades 是否优雅”，而是“什么时候旧 pipeline 复杂到值得上 memo”。

十四、MySQL Hypergraph Optimizer：主流关系数据库的 Hypergraph 产品化

MySQL 传统 optimizer 长期以 join order、access path 和 cost-based choices 为核心。近年的 Hypergraph Optimizer 把 join 查询表示为 hypergraph，能够更系统地处理复杂 join predicates 和 bushy plans。MySQL 9.7 官方 release notes 说明 Hypergraph Optimizer 已可在 Community Edition 中使用，并可通过 optimizer_switch 开关启用。 [20]

Hypergraph 路线和 Cascades 路线的区别在于关注点：Cascades 是通用等价表达式管理框架， hypergraph optimizer 则把 join reorder 这个子问题抽成图模型，直接在图上枚举和剪枝。 对于 join-heavy workloads，后者可以更精确地表达多表谓词和 bushy search。

和 DuckDB DPccp 一样，MySQL hypergraph optimizer 提醒我们：join reorder 未必只能靠通用 rewrite rules。 当研究目标聚焦 join order，显式 join graph/hypergraph IR 可能比通用 logical expression memo 更直接。

十五、对本仓库 Volcano Demo 的启发

当前 demos/volcano_join_reorder 已经具备一个最小 Volcano/Cascades 骨架： parser boundary、JoinGraph、memo groups、transformation rules、implementation rules、required property、 SortEnforcer、costing 和 trace。对比上面的系统，可以得到几个下一步设计方向。

1. 把 optimizer 明确分成 “framework” 和 “join algorithm” 两层

Calcite/ORCA/Cockroach 证明 memo framework 很适合表达 rule、property 和 enforcer； DuckDB/MySQL 证明 join order 本身最好有专门的 graph/hypergraph 枚举模块。因此后续可以让 demo 保留 memo， 但把 join group 的候选 split 来源替换为 DPccp/min-cut/top-down partitioning，而不是只靠交换律和结合律生成。

2. 扩展 property system，而不是只扩展 join algorithm

PostgreSQL 的 PathKeys、Calcite traits、ORCA/Cockroach required properties、Trino distribution choices 都说明 property 是 optimizer 架构核心。当前 demo 只有 Any 和 Sorted(alias.column)； 后续可增加 Partitioned(keys)、Replicated、SingleNode、PreservesOrder 等， 并把 exchange/sort/materialize 都作为 enforcer。

3. 独立 metadata/statistics provider

真实系统不会让 optimizer 到处直接读 JSON 或 catalog。Calcite metadata provider、ORCA metadata cache、 Cockroach/TiDB stats 模块都提示：stats、约束、唯一键、NDV、histogram、foreign key、functional dependency 应成为独立接口。当前 demo 的 StatsCatalog 可以演进成 MetadataProvider。

4. Trace 要记录 “为什么”，不只是 “结果”

研究 demo 的价值在可观察性。现有 trace.json 已记录 rule attempts、duplicates、cache hits 和 final cost； 后续可以增加每条 rule 的 before/after、被剪枝原因、property request stack、cardinality 推导、cost breakdown。 这比单纯输出 best plan 更适合研究 Volcano/Cascades 的优缺点。

5. 不要过早绑定某个生产系统的全部复杂度

PostgreSQL、Spark、Trino、ORCA 的复杂度来自各自执行器和部署模型。研究 demo 应先保留最小核： local IR、memo、property、stats、cost、trace。等 join enumeration 研究清楚后，再逐步加入 distribution 和 execution backend。

十六、按研究用途推荐阅读顺序

  1. 想理解传统 CBO：PostgreSQL optimizer README → SQLite query planner docs。
  2. 想理解规则流水线：DuckDB optimizer blog → DataFusion optimizer docs → Spark SQL Catalyst paper。
  3. 想理解 Volcano/Cascades：Calcite planner/trait docs → Cockroach optimizer docs → ORCA paper。
  4. 想理解 join graph/hypergraph：SQLite NGQP → DuckDB join order optimizer → MySQL Hypergraph Optimizer。
  5. 想做 demo 下一步：先把 JoinGraph enumeration 模块化，再扩展 RequiredProperty 和 trace。
