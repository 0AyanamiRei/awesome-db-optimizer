1. 文档定位

当前项目里有三种 join order search strategy： BottomUpDP、Transformational、TopDownPartitioning。 本文只聚焦 Transformational。

路径说明：当前 checkout 中 demo 目录是 demos/demo。 早期提交中相同模块曾位于 demos/volcano_join_reorder，如果 IDE 标签仍显示旧路径， 可以把本文中的 demos/demo 视为重命名后的对应位置。

这个实现的名字叫 Transformational，但它不是完整论文版本的 Volcano optimizer。 它更准确的定位是：

教学目标

展示 memo group、逻辑变换规则、重复表达式、required property、enforcer、物理实现规则之间的关系。

实现策略

先批量构造逻辑 memo，再在已经构造好的 memo 上做 top-down physical optimization。

刻意简化

没有 promise 排序、没有 cost limit 剪枝、没有通用 move/rule registry，也没有边优化边生成 memo。

因此，本文会把当前实现称为 two-phase transform prototype， 把论文里更完整的框架称为 on-demand Volcano。

2. 数据模型

2.1 RelSet 与 JoinGraph

Join reorder 的搜索空间用 RelSet 表示。RelSet 是一个 uint64_t bitmask，每一位代表一个 relation。 对 chain_3 来说：

Relationidbitmask
a01 << 01
b11 << 12
c21 << 24
{a,b,c}-1 | 2 | 47

JoinGraph 负责判断一个 split 是否可以作为 CP-free join： 

IsValidJoinSplit(left, right, allow_cross_products):
  if left == 0 or right == 0 or left intersects right:
    return false

  if allow_cross_products:
    return true

  return IsConnected(left)
      and IsConnected(right)
      and HasPredicateAcross(left, right)

当前 Transformational::Search 调用时把 allow_cp 固定传为 false，所以逻辑 memo 中不会插入物理意义上的 cross product split。

2.2 LogicalExpr 与 MemoGroup

struct LogicalExpr {
  enum class Kind { Get, Join };

  Kind kind;
  RelSet relset;
  std::size_t relation_id;  // Get
  RelSet left_set;          // Join
  RelSet right_set;         // Join
};

当前实现中，逻辑表达式只有两种：

LogicalExpr含义例子
Get(relation_id) 单表逻辑访问 Get(a)
Join(left_set, right_set) 两个 relation set 的逻辑 join Join({a,b}, {c})

MemoGroup 是等价类的简化版本：同一个 RelSet 下的所有 LogicalExpr 放在同一个 group 里。 

struct MemoGroup {
  RelSet relset;
  std::vector<LogicalExpr> expressions;
};
重要简化：生产级 Volcano/Cascades 的 group 通常由 group id 标识， group expression 的 child 指向 child group。当前 demo 直接用 RelSet 作为 group key，这对 pure inner join reorder 足够直观，但表达能力比完整 memo 窄。

2.3 MemoStore 去重

MemoStore 同时维护：

  • groups_：按 RelSet 保存 group。
  • group_by_relset_：从 RelSet 找 group index。
  • expression_keys_：全局表达式去重集合。
LogicalExpr::Key():
  if Get:
    return "G:" + relation_id
  else:
    return "J:" + left_set + ":" + right_set

MemoStore::InsertExpression(expr):
  if expr.Key() already exists:
    duplicates_generated++
    return false

  expression_keys_.insert(expr.Key())
  EnsureGroup(expr.relset).expressions.push_back(expr)
  return true

注意 Join({a},{b}) 与 Join({b},{a}) 的 key 不同。 因此 commuted join 会作为不同 expression 进入同一个 memo group。 它们逻辑等价，但方向不同，可以让后续物理优化看到不同 child order。

3. 当前 Transformational 工作流

Transformational::Search 的顶层控制流可以概括为： 

Transformational::Search(graph, stats, required_property):
  trace = {}
  memo.clear()
  best_cache.clear()

  full = graph.FullSet()
  if full is empty:
    return empty result

  // Phase 1: 先构造完整逻辑 memo
  work = deque()
  SeedInitialExpressions(graph, allow_cp=false, work, trace)
  ExploreTransformations(graph, allow_cp=false, work, trace)

  // Phase 2: 再在 memo 上做 property-aware physical optimization
  plan = OptimizeGroup(graph, full, required_property, trace)

  result.best_plan = plan
  result.trace.best_cost = plan.cost
  result.trace.plans_cached = best_cache.size()
  return result
┌──────────────────────────────────────────────────────────────────────┐ │ Input: JoinGraph + RequiredProperty │ └─────────────────────────────┬────────────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────────────────────────┐ │ Phase 1: BuildMemo │ │ │ │ 1. Seed singleton Get expressions │ │ 2. Enumerate every valid ordered partition of every relation set │ │ 3. Put every inserted expression into a work queue │ │ 4. Pop expressions and apply commutativity / associativity rules │ │ 5. Deduplicate by expression key │ └─────────────────────────────┬────────────────────────────────────────┘ │ ▼ ┌──────────────────────────────────────────────────────────────────────┐ │ Phase 2: OptimizeGroup(full, property) │ │ │ │ 1. Memoize by (RelSet, RequiredProperty) │ │ 2. For every logical expression in a group, try physical methods │ │ 3. Recursively optimize children with derived required properties │ │ 4. Try SortEnforcer fallback for Sorted property │ │ 5. Return lowest-cost physical plan │ └──────────────────────────────────────────────────────────────────────┘

这就是它和完整 on-demand Volcano 最大的分界： 当前实现把 transformation 全部放在物理优化之前完成。

4. Phase 1: BuildMemo

4.1 SeedInitialExpressions

seed 阶段做两件事：插入所有单表 Get，然后枚举所有 relation set 的合法 ordered partition。 

SeedInitialExpressions(graph, allow_cp=false):
  for relation in graph.Relations():
    insert Get(relation.id)
    work.push(Get)

  full = graph.FullSet()
  for set in [1..full]:
    if set contains bits outside full or PopCount(set) < 2:
      continue

    for left in graph.ValidOrderedPartitions(set, allow_cp=false):
      right = set ^ left
      TryInsertJoin(left, right, from_rule=false)

这里的 ValidOrderedPartitions 会保留 (L,R) 和 (R,L)。 也就是说 Join({a},{b}) 与 Join({b},{a}) 都会被 seed 出来。

为什么是 ordered partition？ 对逻辑关系代数来说 join 通常是 commutative 的，但对物理实现来说 child order 可能影响 build/probe side、nested-loop outer/inner side、以及后续 required property 推导。 demo 保留两个方向，可以让物理优化阶段显式比较。

4.2 TryInsertJoin

TryInsertJoin(graph, allow_cp, left, right, from_rule):
  if from_rule:
    trace.rule_applications++

  if not graph.IsValidJoinSplit(left, right, allow_cp):
    return false

  expr = LogicalExpr::Join(left, right)
  inserted = memo.InsertExpression(expr, trace, count_duplicate=from_rule)

  if inserted:
    work.push(expr)

  return inserted

这个函数是 Phase 1 的唯一插入入口，所以所有初始枚举和规则生成都会经过同一层过滤。 当 from_rule=true 时，失败的重复插入会计入 duplicates_generated。

4.3 ExploreTransformations

变换阶段使用 work queue。每个新插入的 expression 都会被拿出来匹配规则。 

ExploreTransformations(work):
  while work is not empty:
    expr = work.pop_front()
    ApplyRules(expr)

4.4 ApplyRules

规则触发条件生成表达式
Commutativity Join(L, R) Join(R, L)
Left associativity-like L group 中存在 Join(A, B) Join(A, B|R)
Join(B, A|R)
Right associativity-like R group 中存在 Join(B, C) Join(L|B, C)
Join(L|C, B)		 
ApplyRules(expr):
  if expr is not Join:
    return

  // Join(L, R) -> Join(R, L)
  TryInsertJoin(expr.right_set, expr.left_set, from_rule=true)

  // If left child is itself a join: (A join B) join C
  for left_expr in memo.FindGroup(expr.left_set).expressions:
    if left_expr is Join:
      TryInsertJoin(left_expr.left_set,
                    left_expr.right_set | expr.right_set,
                    from_rule=true)
      TryInsertJoin(left_expr.right_set,
                    left_expr.left_set | expr.right_set,
                    from_rule=true)

  // If right child is itself a join: A join (B join C)
  for right_expr in memo.FindGroup(expr.right_set).expressions:
    if right_expr is Join:
      TryInsertJoin(expr.left_set | right_expr.left_set,
                    right_expr.right_set,
                    from_rule=true)
      TryInsertJoin(expr.left_set | right_expr.right_set,
                    right_expr.left_set,
                    from_rule=true)
这里正是 duplicate 演示的来源： seed 阶段已经枚举了许多合法 ordered partition，规则阶段又会从不同路径推导出相同 Join(L,R)。memo 去重会阻止表达式实际膨胀，但每次失败插入仍然是一次搜索工作。

5. Phase 2: 物理优化

5.1 OptimizeGroup

物理优化阶段在已经构造好的 memo 上递归搜索。 缓存 key 是 (RelSet, RequiredProperty)。 

OptimizeGroup(relset, property):
  key = relset + "|" + property.ToString()

  if key in best_cache:
    trace.cache_hits++
    return best_cache[key]

  if property == Any:
    result = OptimizeAny(relset)
  else:
    result = OptimizeSorted(relset, property.sort_key)

  best_cache[key] = result
  return result

这部分已经很接近论文里的 FindBestPlan(Group, PhyProp)： 它递归优化子 group，并且按照 required property 做 memoization。 但它少了 Limit，因此不会做 branch-and-bound 剪枝。

5.2 OptimizeAny

OptimizeAny(relset):
  group = memo.FindGroup(relset)
  best = null

  for expr in group.expressions:
    candidate = OptimizeAnyExpr(expr)
    best = min_cost(best, candidate)

  return best

对 Get，它直接生成 SeqScan。 对 Join，它递归获取左右 child 的 Any 最优计划， 然后尝试三类 join implementation：

Physical op子属性要求cost 公式
HashJoin left: Any, right: Any left_cost + right_cost + left_rows + right_rows + output_rows
NestedLoopJoin left: Any, right: Any left_cost + right_cost + left_rows * right_rows
MergeJoin left: Sorted(join_key_left), right: Sorted(join_key_right) left_cost + right_cost + left_rows + right_rows 
OptimizeAnyExpr(Join(L, R)):
  left_any  = OptimizeGroup(L, Any)
  right_any = OptimizeGroup(R, Any)

  best = min(
    HashJoin(left_any, right_any),
    NestedLoopJoin(left_any, right_any)
  )

  if there is a crossing predicate:
    left_sorted  = OptimizeGroup(L, Sorted(left_join_key))
    right_sorted = OptimizeGroup(R, Sorted(right_join_key))
    best = min(best, MergeJoin(left_sorted, right_sorted))

  return best

5.3 OptimizeSorted

当 root 要求 Sorted(alias.column) 时，当前实现先尝试找到一个天然产生该排序的 merge join；如果没有，就回退到 Any + SortEnforcer。 

OptimizeSorted(relset, sort_key):
  best = null

  for expr in memo.FindGroup(relset).expressions:
    best = min(best, OptimizeSortedExpr(expr, sort_key))

  any = OptimizeGroup(relset, Any)
  if any exists:
    best = min(best, SortEnforcer(any, sort_key))

  return best

单表 sorted property 的处理更简单：如果 sort key 属于该表，就在 SeqScan 上补一个 SortEnforcer；如果 sort key 属于别的表，则该单表 group 无法满足这个 property。 

OptimizeSortedExpr(Get(r), sort_key):
  if sort_key.alias != r.alias:
    return null

  return SortEnforcer(SeqScan(r), sort_key)

6. Trace 计数器

Transformational 主要使用以下 trace 字段：

字段当前 Transformational 中的含义
rule_applications 从规则触发的 TryInsertJoin 次数，包括最终因为 CP-free 检查失败的尝试。
duplicates_generated 规则生成了一个已存在 expression key，memo 拒绝插入。
plans_costed 实际创建并比较过的 physical plan 或 sort enforcer 数量。
cache_hits OptimizeGroup(RelSet, Property) 命中 best_cache_。
plans_cached 物理优化结束后，best_cache_ 中的唯一 (RelSet, Property) 条目数。
best_cost 最终被选中的 best physical plan cost。
为什么 partitions_explored 是 0？ 当前实现的 seed 阶段确实枚举了 partitions，但没有把这个过程计入 partitions_explored。这个字段主要由 BottomUpDP 和 TopDownPartitioning 使用。

7. 完整 on-demand Volcano 框架

微软 Extensible Query Optimizers in Practice 对 Volcano/Cascades 类优化器的描述， 核心是一个按需递归的 FindBestPlan。它不会先完整展开逻辑 memo， 而是在优化某个 (Group, PhysicalProperty) 时，把 transformation rule、 implementation rule、enforcer 都作为 move 统一调度。 

FindBestPlan(group, property, limit):
  if BestPlan(group, property) is already memoized:
    if cached_cost <= limit:
      return cached_plan
    return null

  MarkInProgress(group, property)

  moves = GetAllMoves(group, property)
  SortMovesByPromise(moves)

  best = null
  best_cost = +infinity

  for move in moves:
    if move is transformation rule:
      for neighbor_group in Transform(group, move):
        if not InProgress(neighbor_group, property):
          candidate = FindBestPlan(neighbor_group, property, limit)
          best = min(best, candidate)

    else if move is implementation rule:
      local_cost = DeriveLocalCost(move)
      child_requirements = DeriveChildProperties(move, property)

      for child in move.children:
        child_plan = FindBestPlan(child.group,
                                  child.required_property,
                                  limit - local_cost)
        local_cost += child_plan.cost

      best = min(best, CreatePhysicalPlan(move, child_plans))

    else if move is enforcer:
      local_cost = DeriveEnforcerCost(move)
      weaker_property = GetInputProperty(move, property)
      child_plan = FindBestPlan(group, weaker_property, limit - local_cost)
      best = min(best, Enforce(move, child_plan))

    limit = min(limit, best_cost)

  MemoizeBestPlan(group, property, best)
  return best

这个框架的几个关键点：

  • 按需展开：只在某个 group/property 被请求时才应用相关 rule。
  • 统一 move：逻辑变换、物理实现、enforcer 都是同一搜索循环里的候选动作。
  • promise 排序：优先尝试更可能产生好计划的 move，以便更早降低 limit。
  • limit 剪枝：已有更好上界时，子搜索可以提前失败返回。
  • in-progress 标记：防止 transformation 递归导致循环。
  • property-aware memo：best plan cache 以 (Group, PhysicalProperty) 为核心。

8. 当前实现与 on-demand Volcano 的差异

维度当前 Transformational完整 on-demand Volcano影响
搜索入口 Search 先 BuildMemo，再 OptimizeGroup。 FindBestPlan(group, property, limit) 是唯一递归入口。 当前实现更容易读；论文框架更适合剪枝与按需生成。
Memo 构建时机 物理优化之前批量构造。 优化过程中按需插入 group expression。 当前会预生成最终可能不用的逻辑表达式。
Transformation rule 只在 Phase 1 work queue 中运行。 是 FindBestPlan 循环里的一类 move。 当前不能用 property、limit、promise 影响变换顺序。
Implementation rule HashJoin、NestedLoopJoin、MergeJoin 硬编码在函数里。 通常注册为可匹配的 implementation rules。 当前扩展新物理算子需要改 C++ 控制流。
Enforcer SortEnforcer 是 OptimizeSorted 的 fallback。 enforcer 是独立 move，可参与 promise 排序和 limit 剪枝。 当前只有排序 enforcer，且调度位置固定。
Property 支持 Any 和 Sorted(alias.column)。 可扩展到 ordering、distribution、rewindability、parallelism 等。 当前足够展示 interesting order，但不是通用 property framework。
Cost limit 没有 Limit 参数。 递归调用携带剩余 cost budget。 当前不会 branch-and-bound；会完整比较 memo 中候选表达式。
Promise 没有 move priority。 moves 按 promise 排序。 当前搜索顺序基本由 vector/deque 顺序决定。
In-progress Phase 1 用去重集合终止，Phase 2 cache 已完成结果。 对 (Group, Property) 标记 in-progress，避免递归环。 当前规则很少且简单；完整 rule set 下需要显式 cycle guard。
Group 表达能力 group key 是 RelSet。 group id + group expression，child 指向 child group。 当前适合 join reorder；完整优化器可表示 projection/filter/aggregate 等。
主要教学价值 直观看到 rule-driven duplicates 和 property cache。 展示 extensible optimizer 的统一搜索框架。 两者关注点不同；当前实现是更小、更透明的实验切片。
一句话总结： 当前实现是 “eager logical memo + lazy physical optimization”； 完整 Volcano 是 “lazy logical transformation + lazy physical implementation + lazy enforcer” 的统一 top-down search。

9.4 Phase 2 physical optimization

root 调用是： 

OptimizeGroup({a,b,c}, Any)

它会遍历 group {a,b,c} 的四个 logical expression。 最终胜出的路径等价于： 

MergeJoin(
  Sort[b.y](
    HashJoin(SeqScan(a), SeqScan(b))
  ),
  Sort[c.y](
    SeqScan(c)
  )
)

代价计算如下：

Planrowscost说明
SeqScan(a) 1000 1000 base scan
SeqScan(b) 100 100 base scan
SeqScan(c) 500 500 base scan
HashJoin(a,b) 1000 * 100 * 0.01 = 1000 1000 + 100 + 1000 + 100 + 1000 = 3200 child costs + input rows + output rows
Sort[b.y](HashJoin(a,b)) 1000 3200 + 1000 * log2(1001) = 13167.23 为 merge join 左输入准备 b.y 排序
Sort[c.y](SeqScan(c)) 500 500 + 500 * log2(501) = 4984.32 为 merge join 右输入准备 c.y 排序
MergeJoin((a,b),c) 1000 * 500 * 0.05 = 25000 13167.23 + 4984.32 + 1000 + 500 = 19651.56 merge join cost 不加 output rows

对比一个直接 HashJoin((a,b),c)： 

HashJoin((a,b), c).cost
  = cost(HashJoin(a,b)) + cost(SeqScan(c))
    + rows(a,b) + rows(c) + rows((a,b),c)
  = 3200 + 500 + 1000 + 500 + 25000
  = 30200

因此即使 merge join 需要两个 sort，仍然比最终 hash join 更便宜。

Best plan: MergeJoin(Sort[b.y](HashJoin(SeqScan(a), SeqScan(b))), Sort[c.y](SeqScan(c)))
Cost: 19651.6

这个 plan 与 Any 场景相同，因为最后一层 merge join 使用 b.y = c.y，且右侧 key 是 c.y。 当前实现把该 merge join 的输出 property 标记为 Sorted(c.y)， 所以无需额外 root sort。

10. 如何演进到更完整的 Volcano

如果目标是让当前 Transformational 更接近 Extensible Query Optimizers in Practice 描述的 on-demand Volcano， 可以按下面几个步骤演进。

10.1 引入 GroupId / GroupExpr

当前 group key 是 RelSet。下一步可以显式区分： 

using GroupId = std::size_t;

struct GroupExpr {
  LogicalOp op;
  std::vector<GroupId> children;
  RelSet relset;
};

struct Group {
  GroupId id;
  RelSet relset;
  std::vector<GroupExpr> expressions;
};

这样 memo 就能从 join reorder demo 扩展到 filter、project、aggregate 等更多 logical operator。

10.2 把规则抽象成 Move

enum class MoveKind {
  Transformation,
  Implementation,
  Enforcer
};

struct Move {
  MoveKind kind;
  double promise;
  RuleId rule_id;
};

然后让 FindBestPlan 统一处理三类 move：

  • Transformation move：产生新的 group expression 或 neighbor group。
  • Implementation move：把 logical join 映射为 hash join、merge join、nested-loop join。
  • Enforcer move：添加 sort 等 property enforcer。

10.3 将 transformation 挪入 FindBestPlan

当前是： 

BuildMemo()
OptimizeGroup()

目标是： 

FindBestPlan(group, property, limit):
  for move in GetAllMoves(group, property):
    if move is transformation:
      Generate neighbor on demand
      FindBestPlan(neighbor, property, limit)
    if move is implementation:
      Recursively optimize children
    if move is enforcer:
      Recursively optimize same group under weaker property

10.4 加入 Limit 与 InProgress

Limit 是论文框架中非常重要的剪枝入口。 它可以先从一个启发式 plan 获得上界，再在递归中逐步收紧： 

FindBestPlan(group, property, limit):
  if lower_bound(group, property) >= limit:
    return null

  mark_in_progress(group, property)

  for move in ordered_moves:
    candidate = explore(move, limit)
    if candidate.cost < best.cost:
      best = candidate
      limit = best.cost

  unmark_in_progress(group, property)
  memoize(group, property, best)

InProgress 用来避免 transformation cycle，例如 A 通过 commutativity 变成 B， B 又通过 commutativity 变回 A。

10.5 保留当前实现作为对照组

即使未来加入更完整的 on-demand Volcano，也建议保留当前 Transformational。它非常适合作为对照组：

  • 它能稳定展示 transformation-based duplicate generation。
  • 它和 BottomUpDP / TopDownPartitioning 的 trace 字段可以直接横向比较。
  • 它的 two-phase 结构便于教学：先理解 memo，再理解 property-aware physical search。
