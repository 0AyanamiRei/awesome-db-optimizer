1. 文档定位

当前 demo 中，MPDP 是一个新的可比较 search strategy， 命令行别名是 mpdp，并且在 --compare 输出中紧跟 DPSub。它的目标不是替代 cost model 或物理算子，而是隔离一个问题： 如果 DPSub 的物理计划生成逻辑不变，只换候选 join-pair 的枚举方式，会发生什么？

算法来源

MPDP 来自 Mancini et al., SIGMOD 2022 中面向大规模 join optimization 的思路。

Demo 定位

当前实现是单线程 CPU 教学实现，用来展示候选枚举逻辑；它不是 GPU 运行时实现。

对比对象

最直接的对照是 DPSub：二者的 DP 表、cost model、physical property 处理保持一致。

不要把本文读成性能承诺： SIGMOD 论文讨论的是 massively parallel join optimization。这个 demo 只实现了同一思想的候选枚举形状， 运行时仍是普通 C++ 单线程教学代码。本文中的收益主要体现在 trace 计数器上：少生成无效划分， 而不是展示 GPU 加速。

2. 核心直觉

DPSub 对每个连通 DP 状态 S，会枚举 S 的许多 proper subset， 再用连通性和 CP-free 检查过滤。这个方式很直接，但会把大量最终会被拒绝的候选也数进 partitions_explored。

MPDP 的做法是先看 G[S] 的图结构。一个 connected join-pair (L, R) 一定会在某个双连通块附近被一条局部的 CCP pair "见到"。 因此当前实现对每个 S：

Connected DP state S │ ▼ Find blocks of induced graph G[S] │ ▼ For each block B, enumerate local CCP pairs (lb, rb) │ ▼ Grow lb inside S \ rb │ ▼ Emit global Join-Pair (left, right = S \ left) │ ▼ Use the same physical plan generation as DPSub

直观地说，DPSub 是"先枚举全局候选，再问它合不合法"； MPDP 是"先在 block 中找到局部合法分割，再把它扩展成全局 join-pair"。 所以在 MPDP 的两个专门测试图上，partitions_rejected 为 0。

3. 数据模型

MPDP 与 DPSub 使用同一组核心数据结构：

结构用途MPDP 中的角色
RelSet uint64_t bitmask 表示 DP 状态 S、block B、局部 pair lb/rb 和全局 left/right。
JoinGraph 关系和 predicate 拓扑 提供 IsConnected、HasPredicateAcross、CrossingPredicates 等图查询。
StatsCatalog rows、scan cost、selectivity 只在 MakeScan / MakeJoin 中参与 cost 计算，不决定候选枚举。
CacheTable "RelSet|Property" → PlanPtr 与 DPSub 相同，保存每个状态和 required property 的当前最优物理计划。

这也是为什么 MPDP 和 DPSub 的 PlansCosted、Cached、 DPCells 在专门用例中相同：它们最终填的 DP 表和尝试的物理计划集合一致； 不同的是候选 split 从哪里来。

4. 算法工作流

4.1 顶层控制流

MPDP::Search(graph, stats, required_property):
  cache = {}

  if graph is empty:
    return no-plan

  for each relation r:
    FillRelation(cache, r)

  connected_by_size = CollectConnectedSubsets(graph)

  for size = 2 to n:
    for each connected set S with |S| = size:
      join_pairs = EnumerateJoinPairs(graph, S)

      for each (left, right) in join_pairs:
        partitions_explored++
        BuildPhysicalPlansExactlyLikeDPSub(cache, S, left, right)

      AddSortEnforcersForInterestingOrders(cache, S)
      dp_cells_filled = cache.size()

  best = cache[full_set|required_property]
  if required_property is Sorted and best is absent:
    best = SortEnforcer(cache[full_set|Any])

  return best

注意这里没有 partitions_rejected++ 的路径。 MPDP 的候选生成器在发出 join-pair 前已经检查了局部和全局连通性； 如果某个候选不能成为 CP-free join，它不会进入输出列表。

4.2 Join-Pair 枚举

EnumerateJoinPairs(graph, S):
  result = {}
  emitted = {}

  for each block B in FindBlocks(G[S]):
    for each non-empty proper subset lb of B:
      rb = B - lb

      if lb or rb is disconnected:
        continue
      if no predicate crosses lb/rb:
        continue

      left = Grow(lb, allowed = S - rb)
      right = S - left

      if left/right is a connected CP-free split and not emitted:
        emit (left, right)

这个流程有三个层次：

  1. Block-local：lb 和 rb 都来自同一个 block。
  2. Global growth：lb 不是直接作为最终左侧，而是在 S \ rb 中向外生长。
  3. Dedup：不同 block 或不同局部 pair 可能映射到同一个 global join-pair，需要用 left|right 去重。

5. Blocks 与 Grow

5.1 Block 是什么

本文中的 block 指 biconnected component：一个在删除任意单个顶点后仍保持连通的最大子图。 Bridge edge 也会形成一个二点 block。Block-cut tree 把这些 block 和 articulation point 组织成一棵树， 是 MPDP 能够从局部分割恢复全局 join-pair 的结构基础。

Figure 5-style shape: r1 -- r2 -- r3 -- r4 | | +-----------------+ \ r5 -- r9 / \ r8 -- r6 \ / r7 Block-cut tree intuition: cyclic block -- bridge block -- bridge block -- cyclic block

这个形状很适合解释论文里的链式 block-cut tree：左边一个 cycle block， 中间两个 bridge block，右边一个 cycle block。当前 demo 的 mpdp_fig5 就是这种 Figure 5-style case。

5.2 Block-cut tree 不一定是链

只看 Figure 5-style case 容易形成误解，以为 MPDP 只适用于链状 block-cut tree。 因此 demo 还加入了 mpdp_branching_blocks： 一个中心 cut vertex c 被三个 cycle block 共享，block-cut tree 是分叉形状。

Branching block-cut tree shape: a1 / \ c ---- a2 / \ / \ b1 d1 | | \ / | | b2 d2 --+ Conceptually: cycle A | cut vertex c / \ cycle B cycle D

这个用例的目的不是引入新的 cost model，而是防止把某一种图形当成算法前提。 MPDP 的枚举逻辑要能处理 block-cut tree 的分叉，而不只是线性桥接。

5.3 Grow 的作用

Grow(graph, seeds, allowed) 是一个 BFS： 从局部 lb 出发，只在 allowed = S \ rb 内沿 join graph 扩展， 把所有能连到 lb 的点都吸收到最终左侧。 

Grow(lb, S - rb):
  visited = lb
  queue = lb

  while queue not empty:
    current = pop(queue)
    for each predicate edge current -- next:
      if next is in allowed and next not visited:
        visited += next
        push(next)

  return visited

最终 right = S \ left。如果 left/right 不连通，或二者之间没有 predicate， 这个候选会被丢弃。由于丢弃发生在发出结果前，trace 中不会把它记为 rejected partition。

6. 物理计划与 DP 表

一旦 MPDP 发出 (left, right)，后续步骤与 DPSub 保持一致：

步骤行为Trace 影响
HashJoin 读取 left|Any 和 right|Any，尝试更新 S|Any。 plans_costed++
NestedLoopJoin 同样读取 Any 子计划，按 nested-loop cost 公式生成候选。 plans_costed++
MergeJoin 读取 predicate 两侧的 Sorted 子计划，生成 Any 或 Sorted 输出。 plans_costed++
SortEnforcer 对 S|Any 追加 interesting-order 变体。 plans_costed++，更新 dp_cells_filled
关键对照： MPDP 的搜索结果应该和 DPSub 在 best cost 上一致。若二者 cost 不一致， 首先怀疑的是 join-pair 枚举是否漏掉了合法候选，而不是 cost model。

7. 两个 MPDP 用例

7.1 mpdp_fig5

mpdp_fig5 有 9 个关系、10 条 predicate。 图形是两个 cyclic blocks，中间通过 bridge blocks 连接。这对应论文 Figure 5 风格的教学形状： MPDP 可以从 block-local CCP pair 直接生成全局 join-pair，避免 DPSub 的大量无效 subset。

7.2 mpdp_branching_blocks

mpdp_branching_blocks 有 7 个关系、9 条 predicate。 一个中心关系 c 同时属于三个 cycle block。这个 case 强调： block-cut tree 可以分叉，MPDP 不应只针对链形 bridge 结构工作。

Case图形目的重点观察
mpdp_fig5 两个 cycle blocks 通过 bridge blocks 相连 展示 Figure 5-style 的链式 block-cut tree。
mpdp_branching_blocks 一个 cut vertex 被三个 cycles 共享 展示 branching block-cut tree，避免把链形结构误当成算法限制。

8. Trace 计数器

MPDP 使用和其他策略相同的 SearchTrace。最值得关注的字段如下：

字段MPDP 中的含义
partitions_explored 最终发出的 global join-pair 数。它通常少于 DPSub 的 explored partitions。
partitions_rejected 当前 MPDP 发出前就过滤候选，因此专门用例中为 0。
plans_costed 物理计划代价计算次数。因为物理逻辑匹配 DPSub，专门用例中与 DPSub 相同。
cache_hits bottom-up DP 不按需递归查 cache，保持为 0。
dp_cells_filled DP 表中填充的 (RelSet, Property) 条目数，与 DPSub 的缓存维度一致。
best_cost 最优计划 cost，应该与 DPSub、Transformational、TopDown 结果一致。

9. 运行命令

从 demos/demo 目录运行：

$ ./build/volcano_join_demo --strategy mpdp --test mpdp_fig5
$ ./build/volcano_join_demo --compare --test mpdp_fig5
$ ./build/volcano_join_demo --compare --test mpdp_branching_blocks

下面的表只保留 Partitions、Rejected 和 BestCost， 方便聚焦 MPDP 的候选枚举差异。

9.1 Figure 5-style case

$ ./build/volcano_join_demo --compare --test mpdp_fig5

Strategy            Partitions    Rejected      BestCost
DPSub               5166          4422          1641477.39
MPDP                744           0             1641477.39
Transformational    0             0             1641477.39
TopDown(Naive)      7089          5933          1641477.39
TopDown(Mincut)     1156          0             1641477.39

MPDP 与 DPSub 的 best cost 一致，但 explored partitions 从 5166 降到 744， rejected partitions 从 4422 降到 0。

9.2 Branching block-cut tree case

$ ./build/volcano_join_demo --compare --test mpdp_branching_blocks

Strategy            Partitions    Rejected      BestCost
DPSub               1336          850           11811.99
MPDP                486           0             11811.99
TopDown(Mincut)     822           0             11811.99

这个 case 说明 MPDP 的 block-local 枚举同样适用于分叉 block-cut tree。 MPDP 比 DPSub 少枚举 850 个会被拒绝的候选，同时保持相同 best cost。

10. 与其他策略对比

维度DPSubMPDPTopDown(Mincut)
搜索方向 Bottom-up Bottom-up Top-down
候选生成 全局 subset 枚举 + 过滤 Block-local CCP pair + Grow 递归调用中的 Mincut partitioner
无效候选 会计入 partitions_rejected 发出前过滤，专门用例中为 0 直接生成 CP-free partitions，rejected 为 0
物理计划生成 自底向上填 DP 表 与 DPSub 相同 递归求解并 memoize
教学重点 完整、直接、容易理解的 baseline 同一 DP 框架下的候选枚举优化 自顶向下划分和按需 memoization

一句话总结：MPDP 是 DPSub 的候选枚举实验组。 它保留 bottom-up DP 的表填充方式，也保留 DPSub 的 cost/property 行为； 读 trace 时，最应该对比的是 Partitions 和 Rejected。
