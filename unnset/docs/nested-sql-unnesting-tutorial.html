1. 先建立直觉：嵌套查询像函数调用

SQL 允许在很多地方写子查询：SELECT、FROM、WHERE、HAVING、ORDER BY 等。理解优化的第一步，是区分子查询是否引用了外层查询的列。

非相关子查询

子查询不引用外层列。逻辑上只要算一次，然后把结果复用给外层每一行。

SELECT name
FROM students
WHERE score = (
  SELECT MAX(score) FROM students
);

相关子查询

子查询引用外层列。逻辑上外层每来一行，内层结果都可能不同。

SELECT name, major
FROM students AS s1
WHERE score = (
  SELECT MAX(s2.score)
  FROM students AS s2
  WHERE s2.major = s1.major
);

可以把相关子查询想成一个函数：subquery(s1.major)。优化器的问题是：我们真的要对外层每一行调用一次这个函数吗？如果外层有一百万行，内层表也很大，这个写法会把声明式 SQL 悄悄变成近似嵌套循环。

2. 优化器眼里的相关子查询：Dependent Join

PPT 这一段的讲法很适合初学者：先不要急着看 2015 论文的代数公式，而是先看优化器如何把 SQL 里的 SUBQUERY 节点变成一个显式的逻辑算子。这个算子就是 dependent join。

这一节沿用的示范 SQL 是在学生表里找“每个专业中分数最高的学生”；内层子查询负责算当前外层学生所属专业的最高分。

2.1 从 SUBQUERY 节点到 LHS/RHS

还是用按专业找最高分的例子。SQL 里看起来只是 WHERE score = (SELECT ...)，但在逻辑计划里，外层 students AS s1 和内层 students AS s2 被拆成两边：

  • LHS：Left-Hand Side，左侧输入，也就是外层扫描 students s1。
  • RHS：Right-Hand Side，右侧输入，也就是内层子查询的扫描、过滤和聚合。

这里“依赖”的含义非常具体：RHS 不是一个可以提前独立算完的子计划。它里面有一个外层列 s1.major，必须等 LHS 当前行给出 major 的值，RHS 才知道自己要筛哪些 s2 行。

普通 join：L ⋈ R    右侧 R 可以独立求值 Dependent join：L ⧑ R(L)    右侧 R 要在每个 L 的绑定下求值

2.2 为什么要引入去重绑定域 D？

如果直接按 dependent join 执行，外层 students s1 每来一行，RHS 都要重新算一次 MAX(score)。但这个例子里 RHS 真正需要的外层信息只有 s1.major。如果有很多学生同属 CompSci，为每个学生重复计算 CompSci 的最高分就是浪费。

PPT 的下一步是引入 DUP_ELIM_SCAN，也就是从 LHS 抽取相关列并去重，形成绑定域 D。在这个例子里，D 更准确地说是一张只有绑定列的临时关系：D(d.major) = DISTINCT SELECT s1.major FROM students AS s1。

这一步把“每个学生执行一次 RHS”改成“每个不同专业执行一次 RHS”。绑定仍然存在，但绑定对象从外层每一行缩小为去重后的相关列组合；后续 RHS 不再直接引用外层的 s1.major，而是用 d.major 作为普通输入列，和内层扫描出来的 s2.major 做连接或过滤。

2.3 把 dependent join 往 RHS 里面推

有了 D 以后，优化器继续把 dependent join 往 RHS 子树深处推。它经过 filter，再经过 aggregate。经过 aggregate 时必须把绑定列加入分组键：否则 MAX(score) 会变成全表一个最大值，而不是每个 d.major 一个最大值。

理由是：下推之前，RHS 是“带着一个绑定值执行一次”。当 d.major = 'CompSci' 时，内层只看 CompSci 的行；当 d.major = 'Streets' 时，内层只看 Streets 的行。下推之后，优化器想把这些执行合并成一次关系计算，于是同一张中间关系里会同时出现多个绑定值的行。如果聚合仍然没有分组键，MAX(score) 就会把所有绑定值的行混在一起，只产出一个全局最大分数。

绑定值 RHS 应该看到的行 正确结果
d.major = 'CompSci' CompSci 学生的分数，例如 90, 80 MAX = 90
d.major = 'Streets' Streets 学生的分数，例如 100 MAX = 100

所以聚合必须从“算一个 MAX(score)”变成“按 d.major 分组分别算 MAX(score)”。这不是性能细节，而是保持相关子查询语义的必要条件：原来每个绑定值各自执行一次，现在合并执行时也必须保留“每个绑定值一个结果”的边界。

这正是后面 2015 论文里 D ⧑ ΓA;agg(T) → ΓA∪A(D);agg(D ⧑ T) 的直觉版：如果内层聚合依赖外层值，就要按外层绑定值分组计算。

2.4 到底后，把 dependent join 改成普通 join

dependent join 被一直推到 RHS 足够底部之后，RHS 不再需要“当前外层行”这个上下文。此时它可以先变成 cross join，再由过滤条件 d.major = s2.major 变成普通 inner join。

2.5 最后删除冗余 D，得到手写 join 形状

在这个例子里，d.major 最终和 s2.major 等值绑定，优化器还可以进一步删除 DUP_ELIM_SCAN 和上方多余过滤，让计划变成更熟悉的“先按专业聚合，再和外层学生表 join”的形状。

这里有两个术语可以回头再记：

  • 自由变量：内层表达式使用但自己不能产生的列。例如 s1.major 出现在内层 students AS s2 的过滤条件里，它来自外层。
  • 绑定域 D：外层真正需要传给内层的不同取值集合。例如按专业求最高分时，内层只需要不同的 major，不需要外层每个学生行。

优化目标

把“每个外层 tuple 触发一次内层求值”改成“为每个必要的绑定值集合求一次内层结果，然后用普通 join 接回外层”。这就是 decorrelation / unnesting 的核心。

2.6 这几步分别为什么成立？

PPT 步骤 计划改动 为什么语义不变 为什么更适合优化
展开 SUBQUERY 把内层子查询表示成 DEPENDENT_JOIN 的 RHS。 RHS 仍然能读取 LHS 当前行的 s1.major，等价于原 SQL 中内层引用外层列。 相关性从 SQL 语法现象变成显式逻辑算子，后续规则可以在代数树上工作。
引入绑定域 D 从 LHS 抽取并去重相关列，得到 D(d.major)，再用 s1.major = d.major 接回 LHS。 RHS 的结果只取决于 major，同一个 major 的多行外层 tuple 会得到同一个内层聚合结果。 把“按外层每行执行”降为“按不同绑定值执行”，避免重复计算。
下推穿过 filter 把 dependent join 推到 s2.major = d.major 这个过滤条件下面或旁边。 过滤条件仍然使用同一个绑定值 d.major 来选择 RHS 行，只是绑定值变成了普通输入列。 相关谓词开始变成普通关系条件，为后续 join 化做准备。
下推穿过 aggregate 把聚合改成 MAX(score) GROUP BY(d.major)。 原语义是每个绑定值单独执行一次 RHS；分组键保留了这个“每个绑定值一个聚合结果”的边界。 多个绑定值可以在一次扫描/聚合中同时计算，而不会混成一个全局聚合。
转成普通 join 当 RHS 不再读取 LHS 当前行时，把 dependent join 变成 cross join，再把 d.major = s2.major 变成 inner join 条件。 RHS 所需的绑定列已经由 D 提供，不再需要逐行调用上下文。 普通 join 可以使用 hash join、join reorder 等成熟优化能力。
删除冗余 D 在 d.major 可由 s2.major 等值替代时，移除 DUP_ELIM_SCAN 和多余过滤。 最终 join 仍会用 s1.major = s2.major 和 s1.score = MAX(score) 保证结果匹配。 计划回到更简洁的 join + group by 形状，少一个中间绑定域。

3. 2015：任意查询的代数级 Unnesting

2015 论文的主张很强：不要只靠 SQL 文本模式匹配；把查询先翻译成带 dependent join 的关系代数，然后在代数层面消除 dependent join。这样比“看到某种 SQL 写法就套某条规则”的启发式方法更通用。

3.1 先看论文的 Q1 案例

论文的运行例子是：找出每个学生成绩最好的考试。原 SQL 是相关子查询，因为内层 e2.sid 要和外层 s.id 比较。

-- Q1: correlated form
SELECT s.name, e.course
FROM students s, exams e
WHERE s.id = e.sid
  AND e.grade = (
    SELECT MIN(e2.grade)
    FROM exams e2
    WHERE s.id = e2.sid
  );

如果手写成去相关 SQL，大致就是先按学生聚合出最好成绩，再接回原查询。

-- Q1': decorrelated shape
SELECT s.name, e.course
FROM students s, exams e,
     (SELECT e2.sid AS id, MIN(e2.grade) AS best
      FROM exams e2
      GROUP BY e2.sid) m
WHERE s.id = e.sid
  AND m.id = s.id
  AND e.grade = m.best;

3.2 核心等价式：先造 D，再下推 D

2015 算法最关键的一步，是把原来的 dependent join 拆成两个部分：一个普通 join 加一个以去重绑定域驱动的新 dependent join。

T1 ⧑p T2 ≡ T1 ⋈(p ∧ T1=A(D)D) (D ⧑ T2)，其中 D := ΠF(T2)∩A(T1)(T1)

这里的 T1=A(D)D 表示普通 join 需要把外层原列和投影/重命名后的 D 列对应起来，不是随便做一个自然连接。比如 Q1 里论文把 s.id 投影成 d.id，接回外层时要用 s.id = d.id。

这句话可以翻译成新手版：

  1. 先从外层 T1 取出内层 T2 真正引用的外层列。
  2. 对这些列去重，得到绑定域 D。
  3. 让内层只为每个 D 值算一次。
  4. 再用普通 join 把结果接回原来的外层行。

为什么 D 必须去重？

论文后续下推规则依赖 D 是集合而不是 SQL bag。去重只发生在绑定域上，不表示整个查询丢掉了 SQL 多重集语义中的重复行。

3.3 下推规则的直觉

有了 D，算法把 D ⧑ T 一路往右侧子树下推。遇到不同算子时做相应改写：

遇到的算子 直觉 典型改写
Selection σ 先把 dependent join 推到过滤输入，再保留过滤条件。 D ⧑ σp(T) → σp(D ⧑ T)
Join 如果依赖只在一侧，就推到那侧；如果两侧都依赖，就复制到两侧并用 D 的列对齐。 必要时变成 (D ⧑ T1) ⋈naturalD (D ⧑ T2)
Group By 内层聚合必须按绑定值分别算，所以把 D 的列加入分组键。 D ⧑ ΓA;agg(T) → ΓA∪A(D);agg(D ⧑ T)
Base table 表扫描本身不需要外层列，dependent join 到这里就可以变成普通 join。 D ⧑ R → D ⋈ R

3.4 什么时候可以把 D 整个消掉？

有时 D 的列和内层已有列通过等值条件绑定。Q1 中 d.id = e2.sid，所以可以用 map/substitution 把 d.id 替换成 e2.sid，让计划看起来完全像普通手写 join。

但论文强调：替换不总是更快。消掉 D 的 join 会丢掉早期过滤能力，可能制造更大的中间结果。最终要不要替换，应交给成本模型。

3.5 更复杂的 Q2：非等值条件让 D 留在计划里

Q2 计算“学生某门考试是否比自己或高年级同专业学生的平均成绩差一档以上”。它的相关条件包含 OR、s.year > e2.date 这类非等值比较。这里无法简单用内层已有列替代外层 s.year、s.major，因此 D 作为 sideways information passing 保留下来。

2015 的贡献一句话

把“相关子查询消除”从一堆特殊 SQL 重写规则，提升为一套代数级、可覆盖任意嵌套形状的 dependent join 消除方法。

4. 2025：为什么还需要 Holistic Unnesting？

2015 方法已经能处理大量查询，但 2025 论文指出一个新问题：当 dependent join 多层嵌套时，自底向上一个个消除会让较外层的 D 穿过已经改写过的内层结构，可能生成过大的中间交叉组合。结果仍然正确，但代价可能爆炸。

4.1 多层相关子查询的示例

2025 论文开头用 TPC-H 风格查询说明：找汽车市场客户，这些客户有超过 5 个订单，每个订单的 lineitem 总价超过 300000。这里有两层相关关系：orders 引用 customer，lineitem 引用 orders。

SELECT *
FROM customer
WHERE c_mktsegment = 'AUTOMOBILE'
  AND (
    SELECT COUNT(*)
    FROM orders
    WHERE o_custkey = c_custkey
      AND (
        SELECT SUM(l_extendedprice)
        FROM lineitem
        WHERE l_orderkey = o_orderkey
      ) > 300000
  ) > 5;

4.2 自底向上的病理情况

问题的本质是：先消除内层 dependent join 时，优化器可能保留一个和内层绑定有关的 D；再消除外层 dependent join 时，外层绑定可能和内层绑定组合成更大的域。这个域包含很多最后会被过滤掉的组合，但 group by 等算子已经为它们付出了代价。

这不是语义错误，而是计划质量错误。2025 论文提到实际触发场景来自把复杂 UDF 翻译成纯 SQL 后产生的深层嵌套查询；旧策略可能耗尽内存，新策略可以在毫秒级完成。

4.3 Holistic 的主意：从上到下一起处理

Holistic unnesting 仍然继承 2015 的核心思想：识别外层引用，构造或推理绑定域，最终消除 dependent join。差别在执行顺序和状态管理：

  1. 先识别所有非平凡 dependent join，也就是右侧确实访问左侧列的 join。
  2. 按从根到叶的顺序处理 dependent join。
  3. 在向下改写时维护 outer references、等价类、替换映射等状态。
  4. 遇到嵌套 dependent join 时合并处理，避免不同 D 集合互相穿越造成交叉爆炸。

可以这样记

2015 是“看到一个 dependent join，就把它的 D 往下推到底”。2025 是“先看整棵树里所有相关性，再从上往下携带状态，别让多个 D 无意义地相乘”。

4.4 2025 的算法状态

论文中的状态结构大致包含这些信息：

  • outerRefs：右侧访问了左侧哪些列。
  • D：这些外层引用的绑定域。
  • parent：如果当前 unnesting 在另一个 unnesting 内部，记录父状态。
  • cclasses：等价类，用 union-find 记录 a=b、b=c 这种可传递关系。
  • repr：如果外层列已经被某个内层列替代，记录替代映射。

这些状态让优化器在下推时能做选择：是显式加入 D 的 join，还是用已有等价列直接替换 outer reference。

5. 复杂 SQL 构造怎么处理？

2025 论文还补齐了 2015 没有详细覆盖的一些 SQL 构造。这里不需要一开始就掌握证明细节，但要知道优化器为什么不能简单套用普通下推规则。

5.1 Full outer join 条件里的相关子查询

如果 full join 的条件里有子查询，并且子查询同时引用左右两侧，问题会变难：full join 要保留 unmatched rows，而带动态输入的相关 full outer join 语义本身并不良定义。2025 论文给出的处理方式，是构造左右两侧绑定域 DR 与 DS 的组合，为每组绑定计算谓词结果，再用带 is not distinct from 条件的 outer/full outer join 接回两侧，从而明确 NULL 和 unmatched row 的行为。

5.2 WITH RECURSIVE

递归查询不能只把外层引用加到最终结果上。递归体每一轮都必须按外层绑定值分区执行，否则不同绑定的递归状态会混在一起。论文把外层引用加入 recursive CTE 的列，并让 iteratescan 也产生这些列。

5.3 CTE / DAG 查询计划

CTE 可能让代数表达式从树变成 DAG：同一个子计划被多个地方读取。若天真地沿左右路径分别 unnest，可能重复改写同一个共享子树。论文建议把 DAG 切成若干树，让共享子树形成自己的处理单元，并指定一个共享读触发真正的 unnesting。

5.4 ORDER BY LIMIT

相关子查询里如果有 ORDER BY ... LIMIT，limit 的语义是“每个外层绑定各取前 N 个”，不是整个内层全局取前 N 个。常见改写是先变成窗口函数：

SELECT *
FROM (
  SELECT *,
         ROW_NUMBER() OVER (ORDER BY x) AS rn
  FROM S
) AS ranked
WHERE rn BETWEEN offset + 1 AND offset + limit;

unnesting 时再把外层绑定列加入 PARTITION BY，让 row number 按每个绑定值独立编号。

6. 新手最容易漏掉的语义坑

NULL 比较不是普通等号

论文中 natural D 使用 IS semantics，也就是 NULL 被当成专门值，两个 NULL 可视为相等。若实现时直接使用 SQL 普通 =，可能改变结果。

绑定域 D 去重，不等于整条查询 DISTINCT

D 必须是 duplicate-free，这服务于下推等价式；原查询中的重复行仍要按 SQL 多重集语义保留。

聚合空输入有特殊语义

历史上的 SQL unnesting 规则曾因 COUNT 和空输入出错。2025 论文在 group by static case 中特别提到，空分组聚合需要保证仍产生正确的单行结果，必要时借助 outer join 或 groupjoin。

消掉 D 是成本选择，不是语义必需

用等价列替换 D 可以让计划更干净，但会减少早期过滤。如果 D 的 join 很有选择性，保留它可能更好。

深层嵌套不能只看局部

2015 的局部下推在多数情况下很好，但多层 dependent join 堆叠时，局部最优可能制造巨大的中间域。2025 的 holistic 方法正是为这个场景设计的。

8. 一页总结

嵌套 SQL 优化的核心，不是把 SQL 文本改得更漂亮，而是让优化器摆脱“外层每行调用一次内层”的执行形状。2015 论文给出通用代数方法：把相关子查询表达为 dependent join，抽出外层绑定域 D，把 D 下推到内层，最终把 dependent join 替换成普通 join。2025 论文进一步指出，多层相关子查询不能天真地自底向上逐个处理；应该从上到下整体识别和改写相关性，避免多个绑定域制造无用的中间交叉组合。

如果你只记一个心法：相关子查询的优化，就是把“逐行执行的函数调用”重写成“按绑定域分组的一次性关系计算”。
