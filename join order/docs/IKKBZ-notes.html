1. §4.1–4.3 回顾：假设、代价模型与示例

既然你已经阅读过前面的背景内容，这里仅对 §4.4–4.5 所依赖的核心机制做简要回顾：

4.1 — 五条假设（均在 §5 中被放宽）

  1. 查询 仅包含 join 谓词（不含 R.a = constant 这类选择谓词）。
  2. 每种 join 方法的代价均形如 n₁ · g(n₂) —— 即外层基数 × 内层"每元组微分代价"的线性形式。
  3. 执行空间限定为 PPT（完全流水线）或 BLPT（二叉线性，物化临时关系）。
  4. 数据库 完全驻留内存（代价 = 比较次数，而非 I/O）。
  5. 查询的 join 图是一棵 树（无环）。

4.2 — 递归代价函数与 Rank

对一个一致的 LPT 序列 S（尊重有根 join 树的偏序关系），代价递归定义如下：

C(Λ) = 0                          // 空序列
C(Rⱼ) = 0  若 Rⱼ 为根, 否则 gⱼ(nⱼ)
C(S₁S₂) = C(S₁) + T(S₁) · C(S₂)

T(Λ) = 1
T(S) = Π sᵢ · nᵢ   （序列中各关系 selectivity × 基数的连乘积）

此代价函数满足 相邻序列交换（ASI） 性质：

引理 1. 对一致序列 A·U·V·B 与 A·V·U·B：
C(AUVB) ≤ C(AVUB)  当且仅当  rank(U) ≤ rank(V)，
其中 rank(S) = (T(S) − 1) / C(S)。

Rank 的直觉含义： 它度量的是 单位微分 join 代价所产生的中间结果增长量。Rank 越小，越应先 join。

4.3 — 示例（论文 Figure 3）

一个以 R₁ 为根的 5 关系 join 树。算法 自底向上 运作：

  1. 在每个叶子子树中，按 rank 升序排列子节点 → 形成一条链。
  2. 合并兄弟链：按 rank 交错合并（类似合并有序列表）。
  3. 归一化： 若链头的 rank > 其直接后继的 rank，将它们合并为一个复合节点（因为在任何最优序中，不可能有其他节点插入到它们之间）。重新计算复合 rank。重复直至链按 rank 有序。
  4. 持续向上，直到整棵树变成一条链 → 该链就是最优 join 序。

2. §4.4 — 有根树查询的算法

核心算法 单根 O(N log N)

问题陈述

给定一棵有根 join 树，找到一个使递归代价函数最小化的关系全序（一致的 LPT 序列）。该序列必须是有根树的拓扑排序 —— 子节点只能出现在其父节点之后。

OPT 算法

算法 OPT
输入： 有根树查询 Q，以及每个节点的 T 和 C 值。
输出： 该有根树的最优全序（最优 join 序列）。

1. 若树已是单条链 → 停止。
2. 找一个子树（根为 r），其所有子节点都已是链。
3. 合并这些链，基于 rank 排序，使得结果单链
   按 rank 非递减排列。
4. 归一化该子树的根 r（现在是一条单链）：
   a. 当 rank(根) > rank(其直接后继 c) 时：
        将根和 c 替换为一个新的复合节点，
        表示子链（r 后跟 c）。
        重新计算该复合节点的 T 和 C。
5. 回到步骤 1。

关键细节

步骤 2 — 自底向上选择

算法总是选取一个其所有子节点已是链的子树。在叶子层，这显然成立（单个节点即是链）。随着算法向上推进，子树被转换为链，使其父节点能够被接着处理。

步骤 3 — 合并链

给定 k 条兄弟链，每条内部已按 rank 排序，将它们合并为一条按 rank 排序的单链 —— 完全等同于合并 k 个有序列表。由于 ASI 性质（引理 1），每条链内部的相对顺序得以保留，链间的交错仅由 rank 比较决定。

步骤 4 — 归一化（最关键的步骤）

合并后，根 r 位于链的头部（由于偏序约束，它必须在最前面）。但 r 的 rank 可能大于下一个节点 c 的 rank，这违反了 rank 有序的要求。

为什么可以把 r 和 c 合并为复合节点？
若 rank(c) < rank(r)，则在任何最优全序中，来自兄弟子树的节点都不能放在 r 和 c 之间。证明：若某个外部节点 x 被放在 r 和 c 之间，将 x 与 c 交换可降低代价（由 ASI 性质）。因此 r 和 c 是"粘在一起的"—— 它们可以作为一个原子单元继续参与后续计算。

对于表示子链 (r, c) 的复合节点：

T(r∘c) = T(r) · T(c)
C(r∘c) = C(r) + T(r) · C(c)
rank(r∘c) = (T(r∘c) − 1) / C(r∘c)

步骤 4a 的 while 循环重复执行，直到链头的 rank ≤ 下一节点的 rank。此时该子树的归一化完成，它被父级视为一条单链。

步骤 5 — 迭代

循环持续向上推进。当整棵 join 树的根被处理完毕，得到的链（展开所有复合节点后）即为最优 join 序。

时间复杂度：O(N log N)

借助适当的数据结构（如优先队列用于合并），单棵有根树可在 O(N log N) 内完成 [Lawler 78]。但由于需要尝试所有根（下一节），单根的复杂度并非瓶颈 —— 整体 O(N²) 才是主导项。

3. §4.5 — 树查询算法（全根遍历）

核心贡献 整体 O(N²)

问题

§4.4 解决了单个根的问题。要找到树查询的全局最优 join 序，必须考虑所有 N 种可能的根的选择，并从中选出代价最小的那个。

朴素做法 —— 运行 OPT N 次 —— 代价为 O(N² log N)。本节将其改进为 O(N²)，核心观察是：相邻根的 OPT 结果共享大量计算。

核心洞见：相邻根变换

已知 join 树在根 r 下的最优 join 序，则同一棵树在相邻节点 a（a 是 r 的子节点）为根时的最优序可在 O(N) 时间内算出 —— 无需重新运行完整的 O(N log N) 算法。

变换过程（论文 Figure 4）

令 Tr 为以 r 为根的 join 树，假设我们已有其最优全序。要计算 Ta（以 r 的邻居 a 为根）的最优序：

  1. 将 Tr 的最优序重新表达为两个交错的子序列：
    • a₁, a₂, … — 来自以 a 为根的子树的节点（保持该子树内的最优序）。
    • b₁, b₂, … — 来自 r 的其他所有子树的节点（保持其最优序）。
    此重组是对已有最优序的一趟 O(N) 扫描。
  2. 重新计算 r 的 rank，此时以 a 为其父节点（而非作为根无父节点）。在 Ta 中，a 是根，r 现在是 a 的子节点。
  3. 合并两条链到新根 a 下，基于 rank 排序。a 的子树（即原来的 a₁,a₂,…）与其余部分（r,b₁,b₂,…）现在成为 a 下的兄弟链。

为什么这样做是对的

关键观察：若一个节点在两次定根中父节点不变，则其 rank 不变。当我们将根从 r 切换到 a：

  • a 的子树中所有节点的父节点不变 → 它们的 rank 和相对顺序不变。
  • a 子树之外的所有节点的父节点（r）不变 → 它们的 rank 和相对顺序不变。
  • 只有 r 有了新的父节点（现在是 a）→ 只需重新计算 r 的 rank。
  • 只有 a 成为根 → 无论其 rank 如何，它必须在最前面。

因此，只需 O(N) 的工作量：一次 rank 重算 + 一次两条有序链的合并。

整体算法

1. 任选一个起始根 r₀。运行 OPT (O(N log N)) 得到其最优序。
2. 遍历 join 树（例如 DFS）。对每条边 (u → v)：
   a. 将 u-rooted 树的最优序变换为 v-rooted 树的最优序 (O(N))。
   b. 记录代价。
3. 遍历完所有 N 个节点后，选择代价最小的根。
4. 总时间：O(N) · O(N) = O(N²)。
为什么这很重要： 此前的最优方法（System R 风格）是 O(N!) 的穷举搜索。IKKBZ 将树查询的复杂度降至 O(N²) —— 使得包含数百个 join 的查询（如在演绎/知识库系统中）的优化变得可行。

4. §5.1 — 下推选择

对关系进行预选择（在 join 前应用 R.a = constant）可以减少其基数，但会使原关系上的既有索引失效。优化器面临权衡：预选择但失去索引，或保留索引但对全关系做 join。

论文认为这一选择与 join 序是正交的：对一棵给定的有根 join 树，是否对每个关系做预选择可以独立于 join 序来决定，只需与 join 方法的选择协同考虑。join 序算法本身无需改变。

5. §5.2 — 容纳其他 Join 方法

近似 实用性

有两种 join 方法因不符合 n₁ · g(n₂) 形式而被排除在原始代价模型之外：

方法为何不符合建议的修正
Sort-Merge（排序归并） 代价包含 n₁·log(n₁) + n₂·log(n₂) —— 对两个输入均非线性 将微分代价近似为常数，以此作为 g(n)。
在临时关系上建索引 "外层"关系被选择性访问，代价不是 n₁·g(n₂) 而是 g'(n₁)·n₂ 做近似；选择一个符合 n₁·g(n₂) 形式的模型。
核心哲学观点： 作者认为，既然 selectivity 本身已是粗略估计，对代价公式做简单近似就足够了。优化器的首要目标是避免最坏情况，而非追求完美的全局最优。对 g(n) 的合理近似仍能引导 rank 函数产生合理的 join 序。

关于内存驻留的注记： 若所有关系都在内存中，sort-merge 永远没有优势 —— nested loop 或 selective access 始终更优。这在该时代的知识库应用场景中大大简化了问题。

5. §5.3 — LPT 查询优化

PPT vs BLPT 物化

到目前为止，讨论仅限于 PPT（完全流水线）或 BLPT（每次 join 都物化临时关系）。这是 LPT 频谱的两个极端。问题是：什么时候需要物化临时关系？

答案

当且仅当后续 join 操作需要时，才物化临时关系。

具体而言：

  • 若所有 join 都使用 nested loop 或 selective access → PPT（流水线）永远更优。物化只增加代价而无收益。
  • 若某个 join 使用 sort-merge → 内层关系必须被排序，因此必须先物化。此时处理树变为 LPT（而非纯 PPT）。

扩展策略

  1. 使用 §5.2 的近似方法处理 sort-merge 等 join 方法。
  2. 假设 PPT（完全流水线）计算最优全序。
  3. 后处理：对每个其 join 方法需要物化输入的关系，在处理树的对应位置插入物化步骤。

这样，join 序算法保持不变，同时能处理一般 LPT。

6. §5.4 — 磁盘驻留数据库

推广

论文声称扩展到磁盘数据库在结构上是直接的。论证如下：

  • §4.2 的递归代价方程仅依赖于 LPT 执行类，而不依赖数据在内存还是磁盘上。
  • 磁盘驻留情形仅改变了 g(n) 函数 —— 从比较次数变为页面访问次数。
  • 这已被 [Ibaraki 84] 证实：他们从基于磁盘的 I/O 模型出发，却推导出了结构完全相同的代价方程。

切换到磁盘模型的间接影响：

  • Sort-merge 变得更重要（这印证了 §5.2 近似的必要性）。
  • 代价估计从元组级转向块级。
  • 但 rank 函数、ASI 性质和 OPT 算法完全不变。

7. §5.5 — 优化循环查询

启发式 NP-hard 生成树

问题

对于含环的 join 图，优化问题是 NP-hard 的（[Ibaraki 84] 在 PPT 空间上证明了这一点；推广到一般 LPT 是直接的）。因此（假设 P ≠ NP），不存在精确的多项式算法。需要一种启发式方法。

两个观察

观察 1（超集性质）： 对 join 图的任一子图执行查询，其结果都是原查询结果的超集。丢弃一个 join 谓词只会增加元组。
观察 2（selectivity 的重要性）： selectivity 好（值小）的 join 比 selectivity 差（接近 1）的 join 对 join 序的决定作用更大。极限情况下，cross product（selectivity = 1）不提供任何有用的排序约束 —— 它应该尽可能晚做。

启发式方法：最小 Selectivity 生成树

1. 从含环的 join 图中，选择一棵生成树，
   使得其边上 selectivity 的乘积最小
   （不是求和 —— 这是最小生成树的一个变种）。

2. 在此生成树上运行 IKKZB O(N²) 算法，
   计算出最优 join 序。

3. 以此顺序执行原始（含环）查询。
   额外的 join 谓词（生成树之外的边）
   在执行过程中作为过滤器应用。

为什么这是合理的

  • selectivity 乘积最小的生成树，最小化了中间结果的基数（树查询的结果大小）。
  • 这最小化了"浪费的工作"—— 树查询结果与最终（含环）查询结果之间的差距尽可能小。
  • selectivity 差（接近 1）的边类似于 cross product —— 它们几乎不提供排序指导，因此将其排除在生成树之外是正确的选择。
  • 即使得到的顺序不是最优的，它保证避免最坏情况的执行 —— 与论文的格言一致。

生成树代价定义

与经典 MST（边权求和）不同，这里生成树的"代价"是其边上 selectivity 的乘积：

cost(生成树) = Π sₑ   对树中所有边 e

这等价于标准 MST：取 log(sₑ) 为边权并最小化和。

8. §6 — 结论与关键洞见

贡献总结

  1. 一个形式化的优化模型，捕获了多种已有方法（BLPT, PPT），并将搜索空间与启发式规则分离 —— 使每种方法所依赖的假设变得清晰。
  2. 一个 O(N²) 的精确算法用于树查询，改进了从 [Ibaraki 84] 引入的 O(N² log N) 版本，且远优于 System R 的 O(N!) 穷举搜索。
  3. 扩展以处理选择下推、sort-merge join、一般 LPT、磁盘数据库和循环查询（通过生成树启发式）。

深层洞见

1. 微分代价是关键抽象。
通过将 join 代价表达为 n₁·g(n₂) —— 对外层基数线性 —— 使每个关系的 join 方法选择独立于 join 序。这正是 rank 函数得以运作、多项式时间解法成为可能的根本原因。在作者看来，这是与传统方法最大的分歧点，也是实现高效优化的关键。
2. 对内存驻留数据库，PPT 是最优的。
无需 sort-merge 时，流水线执行（无物化）始终优于 BLPT。这已被实际系统（SQL/DS, QBE）验证，并对专家系统有启示：逐元组处理本质上并非性能问题，只要数据库能装入内存。
3. 避免最坏情况，不要执着于全局最优。
鉴于 selectivity 估计本身就不准确，优化器的首要职责是避免灾难性的执行计划。IKKBZ 算法对树查询保证了这一点，生成树启发式则将这一保证扩展到循环查询。
4. LPT 执行空间并非终点。
论文明确指出，一般处理树（PT，允许多个并发的临时关系）可能支持更多并行性 —— 这在 1986 年是一个开放的研究方向。

9. 总结：需要记住的核心要点

概念一句话
Rank rank(S) = (T(S) − 1) / C(S) — rank 越小越先 join。度量"单位代价下的中间结果膨胀量"。
ASI 性质 局部的 rank 比较即可决定全局最优序 —— 代价函数是相邻序列可交换的。
OPT（§4.4） 自底向上：子树转链 → 按 rank 合并 → 归一化（根与子节点 rank 倒挂时合并） → 重复。
O(N²)（§4.5） 从节点 u 换根到相邻节点 v 仅需 O(N)，因为只有一个节点的 rank 改变了。遍历全部 N 个根 → O(N²)。
Join 方法（§5.2） 任何方法都可用近似套入 n₁·g(n₂) 形式。精度不必完美 —— selectivity 本身已是估计值。
物化（§5.3） 仅当后续 join 需要时才物化（如 sort-merge）。否则全部流水线执行。
循环查询（§5.5） 选一棵 selectivity 乘积最小的生成树，在其上运行 IKKBZ，其余 join 谓词作为过滤器应用。NP-hard → 启发式是必要的。
