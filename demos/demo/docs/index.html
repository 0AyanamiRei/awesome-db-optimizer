1. 设计动机

Join Order 优化是查询优化中最核心的问题之一。不同的枚举策略——自底向上 DP、 基于变换规则的备忘录搜索、以及自顶向下的递归划分——在搜索效率、重复工作、 缓存友好度等方面有着本质差异。然而在完整的生产级优化器中，这些差异往往被工程技巧 （如启发式剪枝、cost-based pruning）掩盖，不利于初学者理解算法本质。

本项目将 Join Order 枚举抽象为以下问题：

输入：一个 Join Graph G = (V, E)，其中 V 是关系（表）的集合， E 是 join predicate 的集合。每个关系有行数 (rows) 和扫描代价 (scan_cost)， 每条边有选择率 (selectivity)。
输出：一棵代价最小的 Join Tree（物理执行计划），满足所有 predicate 且无笛卡尔积。

实验室的设计目标：

  • 可对比 — 统一接口 + 统一 Trace，跨策略对比一目了然
  • 教学友好 — 简单透明的 cost model，算法伪代码与实现一一对应
  • 自包含 — 纯 C++20，无外部依赖（零 DuckDB / PostgreSQL / CMU 依赖）
  • 可扩展 — 新策略只需实现 SearchStrategy 接口
  • 可复现 — C++ 代码构造测试用例，结果确定且可重复

2.3 核心接口

// ── 统一的、跨策略可对比的搜索轨迹 ──
struct SearchTrace {
  uint64_t partitions_explored;     // 考虑了多少 (L,R) 划分
  uint64_t partitions_rejected;      // 被连通性 / CP-free 检查拒绝
  uint64_t plans_costed;            // 实际计算了代价的计划数
  uint64_t cache_hits;              // memoization 缓存命中
  uint64_t plans_cached;            // 缓存的唯一 (RelSet, Property) 条目数

  // 策略特定（不适用则为 0）
  uint64_t duplicates_generated;    // Transformational: 重复逻辑表达式
  uint64_t rule_applications;       // Transformational: 规则应用次数
  uint64_t dp_cells_filled;         // DPSub: DP 表条目
  uint64_t branches_pruned;         // TopDown: branch-and-bound 剪枝

  double   best_cost;               // 最优计划的代价
};

struct SearchResult {
  bool has_plan;                     // best_plan 是否有效
  PhysicalPlan best_plan;            // has_plan=true 时的最优物理计划
  SearchTrace trace;
};

// ── 搜索策略抽象接口 ──
class SearchStrategy {
public:
  virtual ~SearchStrategy() = default;
  virtual std::string Name() const = 0;
  virtual SearchResult Search(const JoinGraph& graph,
                               const StatsCatalog& stats,
                               const RequiredProperty& property) = 0;
};

每个策略实现 SearchStrategy 接口， 返回 SearchResult { bool has_plan; PhysicalPlan best_plan; SearchTrace trace; }。 调用方必须先检查 has_plan，再读取或导出 best_plan。 通过统一的 SearchTrace，五种输出策略可以在同一张对比表中直接比较。

设计原则 策略独立

每个策略是独立的 .hpp/.cpp，不共享搜索状态（memo / cache / work queue）， 确保 trace 数据纯粹反映该策略的行为。

设计原则 Cost Model 共享

所有策略调用同一个 MakeScan / MakeSort / MakeJoin， 保证同一 query 不同策略得到相同 best cost（正确性验证）。

3. 核心数据结构

3.1 RelSet — 关系集合的位掩码表示

using RelSet = std::uint64_t;  // 最多 63 个关系

每个关系分配一个唯一的 id (0…n-1)，关系集合用位掩码表示。 例如：RelSet{1} << 0 | RelSet{1} << 2 表示 {a, c}。 这允许 O(1) 的集合并/交/差和极快的哈希。

3.2 JoinGraph — 查询图

class JoinGraph {
  // 添加关系
  std::size_t AddRelation(alias, table_name, rows, scan_cost);
  // 添加 join predicate
  void AddPredicate({left_alias, left_col}, {right_alias, right_col}, selectivity);

  // 关键查询方法
  RelSet  FullSet() const;                        // 全集的位掩码
  bool    IsConnected(RelSet set) const;          // BFS 连通性检查
  bool    IsValidJoinSplit(L, R, allow_cp) const; // CP-free 划分检查
  bool    HasPredicateAcross(L, R) const;         // 两子集间是否有 join 边
  std::vector<JoinPredicate> CrossingPredicates(L, R) const; // 跨子集 predicate
  double  CrossingSelectivity(L, R) const;       // 跨子集选择率乘积
};
IsValidJoinSplit 的检查逻辑：
  1. L ≠ ∅ ∧ R ≠ ∅ ∧ L ∩ R = ∅
  2. IsConnected(L) — 左侧子图连通（避免笛卡尔积子计划）
  3. IsConnected(R) — 右侧子图连通
  4. HasPredicateAcross(L, R) — 两子集间至少有一条 join predicate
如果 allow_cross_products = true，跳过检查 2-4，允许笛卡尔积。

3.3 RequiredProperty — 所需物理属性

enum class PropertyKind { Any, Sorted };

struct RequiredProperty {
  PropertyKind kind;
  ColumnRef sort_key;   // 仅在 kind == Sorted 时有效

  static RequiredProperty Any();
  static RequiredProperty Sorted(ColumnRef key);
};

物理属性目前支持两种：Any（无要求，可接受任何物理计划）和 Sorted(alias.column)（要求输出按指定列排序）。 Sorted 属性驱动了 interesting orders 机制——即使上层不需要排序， 中间结果也可能需要排序以支持 MergeJoin。

3.4 PhysicalPlan — 物理执行计划

enum class PhysicalOp { SeqScan, HashJoin, NestedLoopJoin, MergeJoin, SortEnforcer };

struct PhysicalPlan {
  PhysicalOp           op;
  RelSet               relset;           // 该计划覆盖的关系集合
  double               rows;             // 估计输出行数
  double               cost;             // 累计代价
  RequiredProperty     property;         // 该计划提供的物理属性
  std::string          relation_alias;   // SeqScan 时的关系别名
  std::optional<JoinPredicate> predicate;// Join 时的 predicate
  PlanPtr              left, right;      // Join 的左右子树
  PlanPtr              child;            // SortEnforcer 的子计划
};

SortEnforcer 是一个特殊的物理算子——它不做数据操作， 只是在对子计划施加排序需求并累加排序代价。SortEnforcer 的存在使得优化器可以在需要时 "支付排序代价"来启用 MergeJoin。

4. 代价模型

所有策略共享同一套简单、确定性的教学代价公式：

SeqScan

scan.cost  = relation.scan_cost
scan.rows  = relation.rows
scan.prop  = Any             // SeqScan 总是提供 Any 属性

SortEnforcer

sort.cost  = child.cost + child.rows · log₂(child.rows + 1)
sort.rows  = child.rows
sort.prop  = Sorted(sort_key)

Join 算子

join.rows = left.rows · right.rows · crossing_selectivity(L, R)

HashJoin:         join.cost = left.cost + right.cost + left.rows + right.rows + join.rows
NestedLoopJoin:   join.cost = left.cost + right.cost + left.rows · right.rows
MergeJoin:        join.cost = left.cost + right.cost + left.rows + right.rows
                  (要求 left.prop = Sorted(key_L) ∧ right.prop = Sorted(key_R))
教学性质：此代价模型刻意简化，不反映真实数据库的 I/O、内存、并行等细节。 HashJoin 的 cost 略高于 MergeJoin（因为多了 join.rows 项）， 但 HashJoin 不需要排序子节点，在大多数 toy 数据上反而是最优方案。 MergeJoin 在某谓词列天然有序（如索引扫描）时体现优势。

5. 策略一：DPSub（自底向上动态规划）

System R 风格 无 duplicate 无剪枝

这是最经典的 Join Order 优化方法，源自 System R (Selinger et al., SIGMOD 1979)。 从单表开始，按关系集合大小递增，系统性地填充 DP 表。

5.1 算法

DPSub::Search(graph, stats, property):
  cache: map<string, PlanPtr>           // key = "relset|property"

  // Step 1: 播种 — 单表扫描
  for each relation r:
    cache["{r}|Any"] = SeqScan(r)
    // 对每个涉及 r 的 predicate 列，也生成 SortEnforcer 变体
    for each predicate where r participates:
      cache["{r}|Sorted(col)"] = SortEnforcer(SeqScan(r), col)

  // Step 2: 自底向上，按大小递增
  for size = 2 to n:
    for each set S of |S| = size:                // 遍历所有 n 选 size 个子集
      for each valid partition (L, R) of S:       // L ∪ R = S, L ∩ R = ∅, CP-free
        partitions_explored++

        // 对每个 partition，尝试三种 join 方法

        // HashJoin (子节点提供 Any)
        plan = HashJoin(cache[L|Any], cache[R|Any])
        plans_costed++; update cache[S|Any]

        // NestedLoopJoin (子节点提供 Any)
        plan = NLJ(cache[L|Any], cache[R|Any])
        plans_costed++; update cache[S|Any]

        // MergeJoin (子节点提供 Sorted)
        plan = MergeJoin(cache[L|Sorted(keyL)], cache[R|Sorted(keyR)])
        plans_costed++; update cache[S|Any]

        // 对每个 predicate，也尝试生成 Sorted 输出
        for each predicate across (L, R):
          plan = MergeJoin(cache[L|Sorted(keyL)], cache[R|Sorted(keyR)])
          plans_costed++; update cache[S|Sorted(key)]

      // Interesting Orders: 对每个 relation 的 predicate 列，SortEnforcer(cache[S|Any])
      for each predicate column col involving a relation in S:
        cached[S|Sorted(col)] = min(cached[S|Sorted(col)], SortEnforcer(cached[S|Any]))

      dp_cells_filled = cache.size()

  return cache["full_set|required_property"]

5.2 核心特征

特征行为
枚举顺序自底向上 — 小集合 → 大集合
重复表达式零 — 每个 (RelSet, Property) 只填充一次最优计划
Cache Hits零 — 所有子问题都已预计算，无需按需查找
搜索完整性穷举所有 valid partition，保证最优
空间复杂度O(3n) — 所有子集的 all valid partition
Interesting Orders对每个子集，为所有可能的排序列生成 SortEnforcer
缺点预先计算所有子问题，即使某些最终不需要；无剪枝

dp_cells_filled 计数器追踪 DP 表中填充了多少条目。 由于 Interesting Orders，实际条目数 > 2n（每个子集可能有多个 property 变体）。

6. 策略二：MPDP（Block-local bottom-up DP）

SIGMOD 2022 风格 Block-local CCP 零 rejected 候选

MPDP 保留 DPSub 的 bottom-up DP 表、物理算子和 cost/property 逻辑， 但把候选划分枚举替换成 block-local 的 Join-Pair 生成。 它的命令行别名是 mpdp，并在 --compare 中紧跟 DPSub 输出。

6.1 算法

MPDP::Search(graph, stats, property):
  cache: map<string, PlanPtr>           // 与 DPSub 相同

  for each relation r:
    FillRelation(cache, r)

  connected_by_size = CollectConnectedSubsets(graph)

  for size = 2 to n:
    for each connected set S of |S| = size:
      for each block B in induced graph G[S]:
        for each local CCP pair (lb, rb) in B:
          left = Grow(lb, allowed = S \ rb)
          right = S \ left
          if (left, right) is a new connected CP-free Join-Pair:
            partitions_explored++
            BuildPhysicalPlans(cache, S, left, right)  // 与 DPSub 相同

      AddSortEnforcers(cache, S)
      dp_cells_filled = cache.size()

6.2 核心特征

特征行为
枚举顺序自底向上 — 与 DPSub 一样按 relation set 大小递增
候选生成先找 G[S] 的 blocks，在 block 内枚举局部 (lb, rb)，再 Grow 成全局 Join-Pair
物理计划与 DPSub 共享 HashJoin / NestedLoopJoin / MergeJoin / SortEnforcer 逻辑
Rejected partitions候选发出前已经过滤，MPDP 专门用例中 partitions_rejected=0
教学定位当前实现是单线程 CPU 教学实现，不是 GPU runtime 实现
深度文档MPDP Workflow

在 mpdp_fig5 上，DPSub 探索 5166 个 partitions、拒绝 4422 个； MPDP 探索 744 个、拒绝 0 个，并保持同样的 BestCost=1641477.39。 mpdp_branching_blocks 则覆盖一个 cut vertex 被三个 cycle block 共享的分叉 block-cut tree， 避免把 Figure 5 风格的链状结构误读成 MPDP 的唯一形状。

7. 策略三：Transformational（变换式搜索 / Volcano 风格）

Volcano/Cascades 风格 O(4ⁿ) duplicate 演示 两阶段搜索

7.1 两阶段架构

┌─────────────────────────────────────────────────────────────────┐ │ Phase 1: BuildMemo (bottom-up 逻辑枚举 + 变换规则) │ │ │ │ Seed: Get(a), Get(b), Get(c) // 单表 │ │ Join({a},{b}), Join({b},{c}), Join({a,c},{b}) │ │ ↓ │ │ Work-queue 循环: │ │ ApplyRules(expr): │ │ Commutativity: J(L,R) → J(R,L) │ │ Associativity: J(J(A,B),C) → J(A,J(B,C)), J(B,J(A,C)) │ │ 每个新表达式 insert 到 Memo（有 dedup） │ │ ↓ │ │ 输出: MemoStore { Group[{a}], Group[{b}], Group[{c}], │ │ Group[{a,b}], Group[{b,c}], Group[{a,b,c}] } │ │ │ ├─────────────────────────────────────────────────────────────────┤ │ Phase 2: Top-down physical optimization on the pre-built memo │ │ │ │ OptimizeGroup(full_set, property): │ │ for each logical expression in group: │ │ 递归优化子表达式 → 物理 join → 取最优 │ │ return best physical plan │ │ │ │ 使用 (RelSet, Property) → PlanPtr 缓存避免重复物理优化 │ └─────────────────────────────────────────────────────────────────┘

7.2 MemoStore 去重机制

class MemoStore {
  std::vector<MemoGroup> groups_;                    // 按 RelSet 分组的逻辑表达式
  std::unordered_set<std::string> expression_keys_;  // 全局去重集合

  bool InsertExpression(expr, trace, count_duplicate):
    key = expr.Key()    // 例如 "J:3:4" 表示 Join({a},{b})
    if key ∈ expression_keys_:
      if count_duplicate: trace.duplicates_generated++     // ★ 计数重复
      return false
    expression_keys_.insert(key)
    groups_[relset].expressions.push_back(expr)
    return true
};

7.3 变换规则

规则模式生成
Commutativity Join(L, R) → Join(R, L)
Left Associativity Join(Join(A,B), C) → Join(A, Join(B,C))
→ Join(B, Join(A,C))
Right Associativity Join(A, Join(B,C)) → Join(Join(A,B), C)
→ Join(Join(A,C), B)
O(4n) 重复问题 (Pellenkoft et al., VLDB 1997)：
变换规则从不同的路径生成相同的逻辑表达式。例如 Join({a},{b,c}) 既可以从初始枚举产生， 也可以从 Join({a,b},{c}) 经 associativity 产生。去重机制阻止了 memo 膨胀， 但每次去重都是一次无效工作 — 这正是 duplicates_generated 计数器展示的内容。 4 表查询的 duplicates 远超 3 表，增长趋势清晰地展示了 O(4n) 行为。

7.4 核心特征

特征行为
Memo 构建自底向上枚举 + work-queue 应用变换规则
重复表达式显著 > 0 — 核心演示价值，随 n 指数增长
物理优化在预构建 memo 上自顶向下递归优化
Cache Hits物理优化阶段有缓存命中，避免重复物理代价计算
优缺点文法声明式（规则可扩展），但 memo 构建的 repeat work 是本质缺陷

8. 策略四：TopDownPartitioning（自顶向下划分搜索）

DeHaan & Tompa 2007 风格 Memoization + Branch-and-Bound Naive 或 Mincut 划分

直接从全关系集开始，递归地将其划分为两个连通子图。 每个子问题以 (RelSet, RequiredProperty) 为 key 缓存结果。 利用预测代价下界进行 branch-and-bound 剪枝。

8.1 算法

TopDownPartitioning::Search(graph, stats, property):
  cache.clear()
  return BestPlan(graph, graph.FullSet(), property, trace)


BestPlan(S, required_prop):
  // ── Memoization ──
  key = "S|required_prop"
  if key in cache:
    cache_hits++; return cache[key]

  // ── Base case: 单表 ──
  if |S| == 1:
    plan = SeqScan(the_relation)
    plans_costed++
    if required_prop.IsSorted():
      plan = SortEnforcer(plan, sort_key)   // 或返回 null（无法满足）
    cache[key] = plan; return plan

  // ── 递归: 枚举划分 ──
  best = null
  for each valid partition (L, R) of S:      // ← EnumeratePartitions
    partitions_explored++

    // Branch-and-bound: 先检查下界
    if LowerBound(L) + LowerBound(R) + 1 ≥ best.cost:
      branches_pruned++; continue

    // HashJoin
    plan = HashJoin(BestPlan(L, Any), BestPlan(R, Any))
    plans_costed++; update(best)

    // NestedLoopJoin
    plan = NLJ(BestPlan(L, Any), BestPlan(R, Any))
    plans_costed++; update(best)

    // MergeJoin (需要 sorted 子节点)
    plan = MergeJoin(BestPlan(L, Sorted(keyL)), BestPlan(R, Sorted(keyR)))
    plans_costed++; update(best)

  // SortEnforcer fallback (对 Sorted property)
  if required_prop.IsSorted():
    plan = SortEnforcer(BestPlan(S, Any), sort_key)
    plans_costed++; update(best)

  cache[key] = best; return best

8.2 划分策略

策略方法复杂度状态
PartitionStrategy::Naive 枚举所有非空真子集 → 过滤连通性 O(2|S|) per call 已实现
PartitionStrategy::Mincut 用 biconnection tree 直接生成 CP-free 划分 O(|V|·|E|) per call 第二期

8.3 Branch-and-Bound 下界

LowerBound(S, property):
  return sum of scan_cost for each relation in S   // 忽略 property（简单下界）
  // 这个下界总是 ≤ 真实最优代价，保证正确性
  // 但可能太弱无法触发剪枝（对 toy 数据尤甚）

8.4 核心特征

特征行为
搜索方向自顶向下 — 从全集递归到单表
Memoization(RelSet, Property) → 最优计划，高 cache_hits
剪枝预测代价 branch-and-bound，效果取决于下界质量
重复工作零 — 每个 (RelSet, Property) 只计算一次
按需计算只计算实际需要的子问题（vs. DPSub 全预计算）
优点自然的 branch-and-bound 结构，可扩展下界
缺点Naive 划分枚举仍为 O(2n)，需 Mincut 优化

9. 预定义测试用例

测试用例完全在 C++ 代码中构造（examples/test_cases.cpp），无需 SQL 或 JSON 输入。

two_table

两表 join，最简非平凡 case

14.2 核心洞察

1. 变换式搜索的 O(4n) 问题

从 --compare 输出可直观看到：4 表 clique，Transformational 产生了 290 duplicates + 290 rule attempts，而 DPSub 和 TopDown 均为 0。 这直接展示了 Pellenkoft et al. (VLDB 1997) 的分析 — associativity/commutativity 规则 在自底向上的逻辑枚举框架中，会产生指数级的重复推导。

2. 自顶向下 + Memoization 的优势

TopDown 在 cache_hits 上显著 > 0（如 clique_4: 74 hits for 4 relations）， 因为 (RelSet, Property) 组合在递归树的多个分支中重复出现。 这是自顶向下方法的自然优势 — 按需计算 + 自动去重。

3. 自底向上的"over-computation"

DPSub 对所有子集预计算了计划（dp_cells_filled > 0），即使某些最终不需要。 如果查询有剪枝机会（如按某些属性排序），TopDown 可以通过 branch-and-bound 避免 计算无用的子问题。

4. MPDP 隔离了候选枚举差异

MPDP 与 DPSub 使用相同的物理计划生成逻辑，因此在 mpdp_fig5 和 mpdp_branching_blocks 上 PlansCosted、Cached、 DPCells 与 DPSub 一致；变化集中在 Partitions 与 Rejected。这让它很适合作为"只替换 join-pair generator"的教学对照。

5. Branch-and-Bound 的实际效果

当前 LowerBound 实现非常保守（仅 sum of scan costs）， 在 toy 数据上可能不触发剪枝（branches_pruned = 0）。 这正是教学目的 — 展示下界质量对剪枝效果的直接影响。 改进下界（如加入最小 join cost 估计、利用 predicate selectivity） 是理解 advanced top-down optimization 的下一步。

关键观察：

  • BestCost 一致（14800）→ 五种策略都找到了真正的最优计划
  • MPDP 减少无效划分：相对 DPSub，cycle_4 上 partitions 46→36，rejected 10→0。
  • Transformational 的 Duplicates 164 — 虽然有去重，但每次去重都是一次失败的 insert 尝试
  • TopDown 的 PlansCosted 最少（86 vs 264）— 因为只有被递归到的路径才被评估
  • Mincut 减少无效划分：TopDown(Naive) 探索 43 个分区、拒绝 9 个；TopDown(Mincut) 探索 34 个、拒绝 0 个。
