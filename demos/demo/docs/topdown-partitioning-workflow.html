1. 核心直觉：从全集向下划分

传统的 bottom-up join 枚举（System R, DPccp）从单表开始，逐层往上拼出更大的 join tree。 Top-down partitioning 则反过来：

核心思想：给定目标关系集 \(S\) 和所需物理属性（如"有序"）， 把 \(S\) 划分成两个连通子集 \(L\) 和 \(R\) (EnumeratePartitions())， 递归求解 \(L\) 和 \(R\) (BestPlan())， 然后用一个物理 join 把它们拼起来 (TryJoinMethods())。 从全集开始，层层向下直到单表。 生成 CP-free 分区的具体方法由 PartitionStrategy 枚举控制—— Naive（枚举+过滤）或 Mincut（BC-tree 直接生成）。

这个想法来自 DeHaan & Tompa (SIGMOD 2007)。 论文的 key insight 是把 "CP-free bushy join 枚举" 规约为 minimal cut 问题： 在 join graph 中，删除一组 minimal cut 边后图恰好分成两个连通分量——这两个分量就是一个合法的 CP-free join 划分。

Top-down 方向带来了三个 bottom-up DP 没有的优势：

优势含义
按需计算 只递归到 join path 实际需要的子问题。不连通的关系集（如 chain_3 中的 {a,c}）永远不会被计算。
Branch-and-Bound 剪枝 一旦发现某个分支的下界已经 ≥ 已知最优计划，立即停止该分支的递归。
自然 property 传播 父节点的物理属性需求（如"有序"）自然推导出子节点需求，不需要预计算所有 property 变体。

2. 论文算法 → 代码映射

下面是 DeHaan & Tompa 论文的核心算法，用伪代码表示。理解这个伪代码就是理解整个策略。

2.1 论文伪代码

BestPlan(G, required_property):
    // Step 1: Memoization
    if memo[G, required_property] exists:
        return memo[G, required_property]

    // Step 2: Base case — 单表
    if |V(G)| == 1:
        return best scan satisfying required_property

    // Step 3: 枚举划分 & 递归
    best = ∞
    for each valid CP-free partition (G_left, G_right) of G:
        for each physical join operator:
            (left_prop, right_prop) = derive_child_properties(operator, required_property)

            // Predicted-cost branch-and-bound
            if lower_bound(G_left) + lower_bound(G_right) + min_join_cost ≥ best.cost:
                continue  // 剪枝

            left  = BestPlan(G_left,  left_prop)
            right = BestPlan(G_right, right_prop)
            plan  = MakeJoin(left, right, operator)
            best  = min(best, plan)

    // Step 4: SortEnforcer fallback
    if required_property is Sorted:
        any_plan = BestPlan(G, Any)
        best = min(best, Sort(any_plan, sort_key))

    // Step 5: 缓存 & 返回
    memo[G, required_property] = best
    return best

这个算法的本质是带 memoization 的递归划分。它不是"先生成所有逻辑等价表达式再优化"， 而是"每次划分直接尝试物理 join，递归求解子问题"。

2.2 代码执行流程

下面按实际的执行顺序，逐函数描述 TopDown 策略如何从收到搜索请求到输出最优计划。 每个自然段末尾标注了对应的源码位置，读者可以打开源码边看边读。

入口：Search()（ top_down_partitioning.cpp:54-67）。 搜索的唯一公开入口。清空 memoization 缓存，然后以全关系集和根需求属性调用 BestPlan(graph, graph.FullSet(), property, trace)。 返回的 SearchResult 包含最优计划、trace 计数器、以及缓存条目总数。 整个搜索的"主循环"就是这一个递归调用——Search 本身没有循环。

关键起点：TopDown 的入口函数 Search() 只有 13 行代码。 真正的工作全部在 BestPlan() 的递归中完成。 这与 BottomUpDP（for size=2..n 的双重循环）和 Transformational（while work_queue not empty 的工作队列）形成鲜明对比。

Phase 1 — Memoization 查找（ top_down_partitioning.cpp:74-81）。 BestPlan() 被调用后，第一步是用 MakeKey(relset, property)（ top_down_partitioning.cpp:44-48）构造一个 "RelSet|Property" 格式的字符串 key——例如 "7|Any" 或 "3|Sorted(b.y)"。 然后在 cache_（类型 unordered_map<string, PlanPtr>， top_down_partitioning.hpp:72）中查找。 命中：cache_hits++，直接返回缓存的 PlanPtr——该子问题已被求解过。 未命中：进入 Phase 2。 这是 TopDown 避免重复计算的核心机制：每个 (RelSet, Property) 组合最多求解一次。

Phase 2 — 基础情况：单表（ top_down_partitioning.cpp:84-107）。 用 PopCount(relset) == 1（ top_down_partitioning.cpp:16）检测。 遍历 graph.Relations() 找到属于该 relset 的那张表，调用 MakeScan(relation)（cost_model.cpp） 构造 SeqScan 物理计划。 如果需要的属性是 Sorted(key) 且该表能产生这个排序键（alias 匹配）， 则用 MakeSort(scan, key) 包装；否则返回 nullptr （表示该单表无法满足排序需求——这在 MergeJoin 需要特定排序列时很常见）。 结果立即写入 cache_ 并返回。

Phase 3 — 递归划分（ top_down_partitioning.cpp:110-132）。 对于多表集合，BestPlan 调用 EnumeratePartitions(graph, relset, trace) （top_down_partitioning.cpp:138-180） 生成所有合法的 CP-free (L,R) 划分。 划分策略由 PartitionStrategy 枚举决定（ top_down_partitioning.hpp:23-26）：

  • Naive（:144-158）： 用 Gosper's hack 遍历 relset 的所有非空真子集，对每个候选 left 调用 graph.IsValidJoinSplit(left, right)（types.cpp） 做连通性 + 跨边界谓词检查。对称对 (L,R)/(R,L) 通过 left >= right 去重。 不满足的候选计入 partitions_rejected。
  • Mincut（:160-175）： 懒初始化一个 MincutPartitioner 实例（mincut.cpp）， 委托 mincut_partitioner_->EnumeratePartitions(relset) （mincut.cpp:267-292）直接生成 CP-free 划分。 Mincut 路径不调用 IsValidJoinSplit——它保证输出的每一对都已满足条件。

Phase 3 续 — TryJoinMethods（ top_down_partitioning.cpp:182-261）。 对每个划分 (left_set, right_set)：

  1. 查找跨 L/R 边界的 join 谓词（graph.CrossingPredicates(left, right)， types.cpp）。
  2. Branch-and-bound 剪枝（:196-201）： 调用 LowerBound(L) + LowerBound(R) + 1 估算最低可能 cost。 若 predicted_lower >= current_best->cost，则 branches_pruned++ 并跳过该划分的物理 join 尝试。
  3. 对于 Any 属性：依次构造 HashJoin、NestedLoopJoin、MergeJoin（仅当有谓词时）。 每种 join 类型递归调用 BestPlan 获取左/右子树（可能带 Sorted 属性需求）。
  4. 对于 Sorted 属性：只尝试 MergeJoin，且谓词的排序键必须匹配所需的排序键。
  5. 所有 join 构造通过 MakeJoin()（cost_model.cpp） 计算 cost，用 Better(candidate, best)（cost_model.hpp:27） 保持全局最优。

Phase 4 — SortEnforcer 回退（ top_down_partitioning.cpp:123-132）。 当需要的属性是 Sorted 但没有一个划分的 MergeJoin 能直接产生时， 先递归求 BestPlan(S, Any)，用 MakeSort() 包装。 如果 Any 子计划已在前面的 Phase 3 中间接算出，这次递归会命中缓存（cache_hits++），几乎零开销。

Phase 5 — 缓存与返回（ top_down_partitioning.cpp:134）。 将最终最优计划（或 nullptr——表示无可行计划）写入 cache_[key]。 nullptr 也会被缓存——这避免了未来对同一不可满足子问题的重复探索。

MincutPartitioner 速览：构造函数（mincut.cpp:38-45） 首先用 BuildAdjacency()（:47-55）为每个顶点构建邻接位掩码， 然后调用 BuildBCTree()（:61-119） 运行 TarjanDFS()（:121-172） 一次性发现所有关节点和双连通分量，构建 BC-tree。 EnumerateConnectedSubsets()（:182-245） 用 BFS 从单点子集出发生成所有连通真子集——每个子集只生成一次（unordered_set 去重）。 最后 EnumeratePartitions()（:267-292） 检查补集连通性（IsConnectedFast，:247-265）， 按 c < complement 去对称，输出。整个过程零过滤——正是 MincutLazy 的核心思想。

3.5 缓存命中分析

§3.0 的搜索树全景图中，蓝色虚线边和 cache hit / memo reuse 标签表示命中缓存的路径。 这正是 TopDown 相比 BottomUpDP 的核心效率来源：子问题一旦求解就被缓存，后续任何路径需要同一 (RelSet, Property) 组合时直接命中。

在 chain_4 搜索中，关键缓存命中包括：

  • 单表扫描（a, b, c, d 的 Any 和各种 Sorted）：每个最多计算一次，后续全部命中。
  • 二表子问题（{a,b}, {b,c}, {c,d} 的 Any）：在多个划分路径中共享。
  • 三表子问题（{a,b,c}-Any, {b,c,d}-Any）：最昂贵的子问题但缓存后零开销重用。
  • SortEnforcer 回退：BestPlan(S, Any) 在 Phase 3 中已算出，SortEnforcer 的递归直接命中。

与之对比，BottomUpDP 必须预计算所有子集的所有属性组合——即使某些组合在最终最优计划中根本不会被用到。 chain_4 上 DPSub 填了 42 个 DP 格子（dp_cells_filled=42），而 TopDown 只缓存了 22 个条目——只有 DPSub 的 52%。 这些"省下"的 20 个格子就是按需计算 vs 全量预计算的差距。

3.6 枚举划分：Naive vs Mincut

在 §3.3 Step 3 中我们展示了 Naive 分支枚举 chain_4 的划分：遍历 7 个 unordered 候选 left，4 个被拒绝。 下面从算法层面对比两种策略。

Naive 方法（top_down_partitioning.cpp:144-158） 是最直接的思路：用 bit-subset iteration（Gosper's hack）枚举 relset 的所有非空真子集作为 left， right = relset ^ left，通过 left >= right 过滤对称对。 对每个候选，调用 graph.IsValidJoinSplit(left, right, allow_cross_products) （types.cpp）做三重检查：left 连通、right 连通、跨边界谓词存在。 不满足任何一项即计入 partitions_rejected。 这是经典的 generate-and-test 模式——先生成所有数学上可能的划分，再逐一过滤。 对 n 表查询，无序候选数为 2^(n-1)-1，其中大量是无效的（补集不连通）。 chain_4 上：7 个候选 → 4 个被拒绝（57% 浪费）。

Mincut 方法（top_down_partitioning.cpp:160-175） 采用完全不同的思路：直接生成连通子集，永不产生无效候选。 懒初始化的 MincutPartitioner（mincut.cpp） 在首次使用时运行 Tarjan DFS（:121-172） 一次性发现所有关节点和双连通分量，构建 BC-tree。 此后 EnumerateConnectedSubsets()（:182-245） 用 BFS 从每个顶点作为种子出发，每次扩张加入一个邻接顶点， unordered_set<RelSet> 全局去重保证每个连通子集恰好生成一次。 EnumeratePartitions()（:267-292） 遍历所有连通子集，用 IsConnectedFast()（:247-265） 检查补集连通性，通过 c < complement 去对称后输出。 整个过程零过滤——每个生成的连通子集就是一个合法的 left 候选， 只需要检查补集连通性即可。这就是 DeHaan & Tompa (2007) 提出的 MincutLazy 算法： 利用 CP-free 划分与连通子集的一一对应关系，将过滤工作转化为生成工作。 chain_4 上：生成 ~6-7 个连通子集，3 个通过补集检查，0 个被拒绝。

生成方式 位运算枚举所有子集 + 连通性过滤 BC-tree + BFS 扩张直接生成连通子集
CP-free 保证 后验 — 通过 IsValidJoinSplit 过滤 先天 — BFS 扩张保证 left 连通，只需检查 right

4. 关键设计决策

4.1 为什么 Memo Key 是 (RelSet, Property) 而不是 (RelSet, Property, Budget)？

DeHaan & Tompa 论文的一个重要发现是：accumulated-cost bounding 与 memoization 有根本冲突。 这是理解 TopDown 搜索效率的关键设计决策。

核心矛盾：如果把"当前已使用的预算"放进 memo key，同一个逻辑子图 (S, property) 会因为不同预算而被多次重新优化。 memoization 的复用能力被破坏——这正是论文实验中 TBNmcA（accumulated-cost）比 TBNmcP（predicted-cost）慢 10–100 倍的原因。

有两种方式在 TopDown 搜索中做 branch-and-bound 剪枝：

Accumulated-cost bounding (TBNmcA)：剪枝条件为 accumulated_parent_cost + best_child_cost + LowerBound(remaining) >= global_best。 因为 accumulated_parent_cost 来自调用路径（即从根到当前节点已经付出的 cost）， 同一个子图 (S, property) 从不同的父划分调用时，accumulated_parent_cost 不同， 剪枝结果也不同。为了正确 memoize，key 必须包含预算：(RelSet, Property, Budget)。 这导致 缓存碎片化——同一个子图被多次重新优化（每次预算不同）， memoization 的核心优势（每个子问题只解一次）被完全破坏。

Predicted-cost bounding (TBNmcP)（当前实现）：剪枝条件为 LowerBound(L) + LowerBound(R) + 1 >= current_best->cost （top_down_partitioning.cpp:196-201）。 LowerBound(S) 只取决于 S 自身的统计属性（scan_cost 之和）， 与调用路径无关。因此同一个 (S, property) 无论从哪个父划分调用， 其最优计划都相同——memo key 只需要 (RelSet, Property)。 缓存可以安全共享，每个子问题恰好求解一次。

具体例子：在 chain_4 搜索中，BestPlan({a,b}=3, Any) 被两个不同的划分路径调用： Partition (3,12) 需要 {a,b} 作为左子，Partition (7,8) 的 SortEnforcer 回退也间接需要。 两次调用的 accumulated_parent_cost 完全不同（前者在根层，后者在第三层递归内）， 但最优计划相同（HashJoin(a,b) cost=3200）。Predicted-cost bounding 让两次调用共享同一个缓存条目 ——第二次直接命中（cache_hits++）。如果用 accumulated-cost bounding， 第二次调用会因为预算不同而重新计算整个 {a,b} 子树。

论文的实验数据显示：在 JOB benchmark 上，TBNmcA 比 TBNmcP 慢 10-100 倍，且随着查询表数增加差距扩大。 这种设计权衡是 TopDown 搜索能够在 join reorder 领域有效工作的核心原因之一。

TopDown 策略通过 PartitionStrategy 枚举支持两种分区方法， 两种策略都已经完整实现并可随时切换（CLI: --strategy topdown vs --strategy topdown-mincut）。

MincutLazy 算法要点

Mincut 策略实现了 DeHaan & Tompa (SIGMOD 2007) 论文的核心算法，分为两个阶段：

  1. Phase 1 — BC-tree 构建 (BuildBCTree()): Tarjan DFS (mincut.cpp:121-172) 识别关节点和双连通分量， 构建 block-cut tree。关节点是连接两个双连通分量的顶点——删除关节点后图变得不连通。
  2. Phase 2 — 连通子集枚举 (EnumerateConnectedSubsets()): BFS 从每个种子顶点出发，通过添加 frontier 邻居逐层扩展。用 unordered_set<RelSet> 去重—— 保证每个连通子集恰好被生成一次（而不是被多条路径重复访问）。
  3. Phase 3 — CP-free 分区判定 (EnumeratePartitions()): 对每个连通子集 C，用 IsConnectedFast() 检查补集 relset\C 是否也连通。 仅当两边都连通且 C < complement（避免对称重复）才输出。
实测效果 (chain_4)：Naive 探索 26 个划分、拒绝 8 个 → Mincut 探索 18 个划分、拒绝 0 个。 在 star_4 上效果更显著：Naive 34→24、拒绝 10→0。详见 §7 实验数据。

Mincut 策略内部使用 lazy 初始化 (top_down_partitioning.cpp:162-165)： MincutPartitioner 对象在第一次调用 EnumeratePartitions() 时才构建， 之后同一个 JoinGraph 的后续查询直接复用，避免重复构建 BC-tree。

4.3 LowerBound 设计：predicted-cost vs accumulated-cost bounding

上一节讨论的 Memo Key 设计选择直接影响了 LowerBound 的实现方式。 当前 LowerBound 的实现是这条设计链上的自然结果——简单、安全，但在 toy 数据上太弱而无法触发剪枝。 下面深入分析这一点，并讨论改进方向。

4.3.1 当前 predicted-cost bounding 的缺陷

当前 LowerBound = sum(scan_costs) (top_down_partitioning.cpp:263-275)： 

double LowerBound(const JoinGraph &graph, RelSet relset,
                  const RequiredProperty &property) const {
  double lb = 0.0;
  for (const auto &relation : graph.Relations()) {
    if ((relset & (RelSet{1} << relation.id)) != 0) {
      lb += relation.scan_cost;
    }
  }
  (void)property; // property doesn't affect our simple lower bound
  return lb;
}

优点：安全 (admissible)

任何计划至少要扫描所有涉及的表，所以 LowerBound(S) ≤ true_optimal_cost(S) 恒成立。剪枝永远不会漏掉更好的计划。

缺陷：三方面

① 与真实 cost 差距过大：chain_4 上 LowerBound(15) = 1000+100+500+200 = 1800， 而真实最优 cost = 41500.24。差距 23×。 ② 忽略 property：(void)property 显式丢弃了属性参数。 Sorted 查询的 LowerBound 与 Any 相同——但实际上排序需要 O(n log n) 的额外 cost。 ③ 忽略 join cost：n 表查询至少需要 n-1 次 join， 每次 join 至少产生 hash 构建 + 探测的 CPU cost。当前下界完全不考虑。

后果：branches_pruned 在所有 6 个测试用例中都为 0——predicted-cost B&B 从未触发。 这不是 bug，而是 toy 数据下的预期行为——scan_cost 太小（100-1000）， 下界永远低于第一个候选 plans_costed。 在真实数据库（表大小在 10^6-10^9 行量级）上，scan_cost 差异足以拉开下界与 best-so-far 的差距。

4.3.2 Accumulated-cost bounding 对比

如果用 accumulated-cost bounding 代替 predicted-cost bounding， 剪枝条件变为：accumulated_cost + LowerBound(remaining) >= best->cost。

链式举例：在 BestPlan(15, Any) 的 Partition 1 ({a}, {b,c,d}) 中：
  • Predicted-cost：B&B 检查 = LowerBound({a}) + LowerBound({b,c,d}) + 1 = 1000 + (100+500+200) + 1 = 1801。 而 best 初始为 null → 永远不剪枝。
  • Accumulated-cost（假设 best 已有值 50000）：B&B 检查 = parent_accumulated(0) + cached_cost({a}) + cached_cost({b,c,d}) + 1。 如果 {b,c,d}-Any 已经算出实际 cost=~12000，{a}-Any 的 cached cost=1000， 则检查 = 0+1000+12000+1 = 13001 < 50000 → 不剪枝（仍然太弱）。 但如果 best 更紧（如 41500），在后续划分中可能触发。

然而 accumulated-cost bounding 的根本问题是 §4.1 分析的缓存碎片化： cached_cost({b,c,d}) 本身是从特定路径（某 accumulated_cost）计算出来的， 在另一条路径（不同 accumulated_cost）下可能不是最优——因为剪枝阈值不同。 这迫使 memo key 包含 budget → 缓存复用率暴跌 → 总搜索时间反而增加 10-100 倍。 这就是论文的 TBNmcA vs TBNmcP 实验结论。

4.3.3 LowerBound 强化方向

如果保持 predicted-cost bounding（推荐），有三种方法可以在不破坏 memoization 的前提下收紧 LowerBound：

  1. 加入 join 数量下限： LowerBound(S) = sum(scan_costs) + min_join_cost × (|S|-1)。 min_join_cost 可以是所有 join 谓词 selectivity 加权的最低可能 join cost。 这仍然是安全的——任何 n 表计划至少需要 (n-1) 次 join。 对于 chain_4 (n=4)：LowerBound = 1800 + min_join_cost × 3，如果 min_join_cost 取 ~2000， 则 LowerBound ≈ 7800，比原来的 1800 紧 4×。
  2. 复用已缓存的子计划 cost： 当 cache_ 中已有 (S, Any) 的最优计划时，直接用它的真实 cost 作为 LowerBound。 这仍然安全——因为缓存中的 cost 是 S 的真实最优值。 实现：LowerBound 在返回前检查 cache_.find(MakeKey(relset, RequiredProperty::Any()))， 若存在且缓存 cost > 当前 sum，则返回缓存 cost。 这是最直接且最安全的强化——不需要任何统计假设。
  3. Sorted 属性加入最小 sort cost： 当 !property.IsAny() 时，LowerBound 应至少包含一次 sort 的 cost： LowerBound(S, Sorted) = LowerBound(S, Any) + min_sort_cost。 min_sort_cost 可取 min(rows) × log2(min(rows))。 这不会导致 cache key 膨胀（LowerBound 只影响 B&B 决策，不影响 memo 键）， 但能让 Sorted 子问题的 B&B 剪枝更有效。
教学价值：当前 LowerBound 的"太弱"正是教学意图——它让读者直观感受到 "下界质量决定了剪枝效果"。上述三个强化方向是理解 advanced top-down optimization 的自然下一步， 每个方向都保持了 predicted-cost bounding 的独立性（不依赖调用路径）， 因此 memo key 始终是 (RelSet, Property)。

6. 与 Transformational 的本质对比

TopDown 和 Transformational 代表了 join enumeration 问题的两种根本范式。 它们的差异不是"一个快一个慢"，而是对"如何生成 join tree"这个问题的两种完全不同的回答。

6.1 核心范式差异

Transformational (Volcano/Cascades)

先生成逻辑空间，再做物理优化。 这是经典的优化器架构：用 transformation rules 在 LogicalExpr 上工作， 从已有表达式推导新的等价表达式——无需理解 join graph 的结构。

  • Phase 1 — BuildMemo：从单表 Get(r) 出发，枚举所有 valid partition 的 Join(L,R)， 然后进入 work-queue 循环：Commutativity: J(L,R) → J(R,L)、 Associativity: (A⋈B)⋈C → A⋈(B⋈C)。 每个新表达式插入 MemoStore（RelSet → vector<LogicalExpr>）， expression_keys_ 负责去重。
  • Phase 2 — Physical Optimization：在 MemoStore 上做 top-down 物理优化。 OptimizeGroup(full, prop) → 对每个 LogicalExpr 尝试 HashJoin/NLJoin/MergeJoin， 递归优化子 group。

关键问题：同一个 LogicalExpr 可通过多条规则序列到达。 例如 A⋈B⋈C 可通过 associativity 的两种方向产生重复， 导致 duplicates_generated = O(4n)。 在 clique_4 上: duplicates=290, rules=290 — 是缓存条目数 (43) 的 6.7 倍。

源码: transformational.cpp

TopDown Partitioning (DeHaan & Tompa)

边划分边做物理优化，不分两层。 直接在关系集上递归划分——没有 LogicalExpr，没有规则引擎， (L, R) 划分天然就是 plan 的生成器。

  • 单一 Phase — BestPlan：从 FullSet 出发，递归划分。 对每个 CP-free 划分 (L, R)： ① Memoization lookup by (RelSet, Property)； ② 若 |S|=1，构造 Scan ± Sort； ③ 递归求解 L 和 R 的最优计划，尝试 HashJoin/NLJoin/MergeJoin； ④ Predicted-cost B&B：若 LowerBound(L)+LowerBound(R)+1 ≥ best.cost，剪枝跳过。 结果直接存入 cache_。

关键优势：每个 (RelSet, Property) 组合只有一个最优计划—— 没有"多种表达方式"的问题。duplicates_generated ≡ 0。 B&B 剪枝在有意义的 cost 差异下可以减少物理 join 尝试（当前 toy 数据下 pruned=0，见 §4.3）。

源码: top_down_partitioning.cpp、 mincut.cpp

这两种范式不是"谁更快"的问题，而是对"join tree 的生成方式"的两种根本不同的回答： Transformational 用表达式和规则——join tree 是代数表达式经过等价变换的结果； TopDown 用关系集和划分——join tree 是关系集的递归二分。 在 join reorder 的特定领域内，关系集划分更直接，因为一棵 join tree 天然就是一个递归二分过程。 这也是为什么 TopDown 能在 join reorder 的 benchmark 中通常优于 Cascades 风格优化器的根本原因。

  • BottomUpDP 预先计算了 42 个 DP 格子，TopDown 只缓存了 22 个（52%）——按需计算 vs 全量预计算
  • Transformational 产生 60 个重复表达式（2.7× Cached=22）——4 表时 duplicate 问题比 3 表更明显
  • BestCost 一致（41500.24）——正确性验证

7.7 数据总结

  1. Mincut 在所有测试中 partitions_rejected = 0 — 不像 Naive 那样"生成后过滤"，Mincut 从源头保证每个候选都是 CP-free 的。
  2. Mincut 的优化效果与图稀疏度正相关： star_4 (29% reduction) > cycle_4 (21%) > clique_4 (0%)。图越稀疏，关节点越多，BC-tree 的优化空间越大。
  3. branches_pruned 仍为 0 — 当前 LowerBound = sum(scan_costs) 太弱，B&B 剪枝从未触发。改进下界（如利用缓存的子计划代价）是教学下一步。
  4. TopDown 的 plans_costed 始终 ≤ BottomUpDP — memoization + 按需计算避免了预填充所有 DP 单元格。
